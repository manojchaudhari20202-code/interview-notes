
### React Fundamentals
* **What is React**
  * React is a declarative, component-based JavaScript library for building user interfaces, primarily for single-page applications (SPAs). It focuses on the view layer and enables developers to create reusable UI components.
  * It was developed by Facebook (now Meta) and is maintained as an open-source project with a large ecosystem.
  * React uses a virtual DOM to efficiently update and render components, improving performance by minimizing direct manipulation of the real DOM.
  * It promotes unidirectional data flow (from parent to child via props), making state management more predictable and debugging easier.
  * React can be used for web (ReactDOM), mobile (React Native), and even desktop applications (via frameworks like Electron).

* **SPA architecture**
  * A Single-Page Application loads a single HTML page and dynamically updates content as the user interacts, without requiring full page reloads. This provides a fluid, app-like experience.
  * Routing is handled client-side using libraries like React Router, which updates the URL and renders different views without server round-trips.
  * SPAs rely heavily on JavaScript to manage state, fetch data from APIs (often REST or GraphQL), and render UI.
  * Benefits include faster interactions after initial load, reduced server load, and easier development with front-end frameworks.
  * Challenges include SEO (mitigated by SSR/SSG), initial bundle size, and client-side performance on low-end devices.

* **Virtual DOM**
  * The Virtual DOM is a lightweight JavaScript representation of the actual DOM. It is an in-memory object that mirrors the structure of the real DOM.
  * When a component's state changes, React creates a new virtual DOM tree and compares it with the previous one using a diffing algorithm.
  * The differences (minimal set of changes) are then batched and applied to the real DOM in a process called reconciliation.
  * This approach avoids costly direct DOM manipulations and improves performance by reducing browser reflows and repaints.
  * The virtual DOM is not a React-only concept; other libraries like Vue and Preact also use similar techniques.

* **Real DOM vs Virtual DOM**
  * Real DOM is the actual browser DOM API (e.g., `document.getElementById`). It is slow to manipulate because changes can trigger layout, painting, and compositing.
  * Virtual DOM is an abstraction; updates are cheap because they only modify JavaScript objects.
  * Direct real DOM updates are imperative and require manual handling, while virtual DOM updates are declarative—developers describe the desired state, and React handles the updates.
  * Virtual DOM enables cross-platform rendering (e.g., React Native renders to native views, not the browser DOM).
  * However, the virtual DOM is not always faster for trivial updates; its main advantage is maintaining a predictable development model while optimizing for complex UIs.

* **React rendering model**
  * React’s rendering model is based on the concept of a component tree. Rendering starts from the root and propagates downwards.
  * When a component’s state or props change, React re-renders that component and all its children by default (unless optimizations like `React.memo` are applied).
  * The render phase is pure and side-effect-free, meaning it should not directly interact with the browser or cause mutations.
  * After the render phase, React commits the changes to the DOM in the commit phase, which may cause side effects like layout calculations.
  * React 18 introduced concurrent rendering, allowing rendering to be interruptible and prioritizing updates based on urgency.

* **Declarative UI**
  * Declarative UI means describing what the UI should look like for a given state, rather than imperatively manipulating the DOM step by step.
  * In React, you define components that return markup based on props and state, and React takes care of updating the DOM when data changes.
  * This leads to more predictable and easier-to-understand code, as the UI is a function of state: `UI = f(state)`.
  * Debugging is simpler because you can reason about the state at any point, rather than tracking a sequence of DOM commands.
  * Declarative programming also improves developer productivity and enables features like time-travel debugging.

* **Component-based architecture**
  * React applications are built by composing independent, reusable components. Each component encapsulates its own structure, behavior, and styling.
  * Components can be nested, managed, and reused across the application, promoting DRY (Don't Repeat Yourself) principles.
  * The architecture encourages separation of concerns: components can be presentational (concerned with how things look) or container/stateful (concerned with how things work).
  * This modularity makes testing easier, as components can be tested in isolation.
  * Component-based design also facilitates collaboration in large teams, as developers can work on different components independently.

* **JSX syntax**
  * JSX is a syntax extension for JavaScript that allows writing HTML-like markup inside JavaScript files. It is not required but is widely used in React.
  * JSX is transformed by tools like Babel into `React.createElement` calls, producing React elements.
  * It provides a more readable and expressive way to define UI structures, combining markup with logic (e.g., embedding expressions with `{}`).
  * JSX prevents injection attacks by automatically escaping values (except when using `dangerouslySetInnerHTML`).
  * Key rules: JSX must have a single root element (or use fragments), and attribute names follow camelCase (e.g., `className` instead of `class`).

* **React elements vs components**
  * A React element is a plain object describing a DOM node or component instance. It is immutable and cheap to create (e.g., `React.createElement('div', null, 'Hello')`).
  * A component is a function or class that optionally accepts input (props) and returns a React element (or tree of elements).
  * Components can be reused and composed, while elements are the building blocks that React renders to the screen.
  * Elements are what components return; React internally maintains a tree of elements and updates the DOM accordingly.
  * Understanding the distinction helps in debugging: components are factories, elements are the output instances.

* **Functional vs class components**
  * Functional components are plain JavaScript functions that accept props and return JSX. They are simpler, easier to test, and encouraged in modern React.
  * Class components extend `React.Component` and have a `render()` method. They can hold local state and lifecycle methods (e.g., `componentDidMount`).
  * Before hooks, functional components were stateless; now hooks enable state and side effects in functional components, making class components less necessary.
  * Functional components have a smaller bundle size (no class overhead) and avoid issues with `this` binding.
  * Class components are still supported, but the React team recommends using functional components with hooks for new code.

* **Props**
  * Props (short for properties) are read-only inputs passed to components. They allow parent components to pass data and callbacks down to children.
  * Props are immutable within the child component; a component should never modify its own props.
  * Prop types can be documented using TypeScript, PropTypes, or Flow for runtime validation.
  * Special prop: `children` allows composition by passing elements or components as children.
  * Props drilling is a common issue where props are passed through many intermediate components; solutions include context or state management.

* **State**
  * State represents data that changes over time within a component. Unlike props, state is managed internally and can be updated by the component itself.
  * In functional components, state is managed with the `useState` hook or `useReducer` for complex logic.
  * State updates are asynchronous and batched for performance. The setter function from `useState` can accept a value or a function that receives the previous state.
  * State should be treated as immutable; never mutate state directly (e.g., `state.count++`), always use the setter.
  * Lifting state up is a pattern where state is moved to a common ancestor to share it among sibling components.

* **Event handling**
  * React events are named using camelCase (`onClick` instead of `onclick`) and are passed as functions, not strings.
  * Event handlers are wrapped in synthetic events, which are cross-browser wrappers around native events, ensuring consistent behavior.
  * To prevent default behavior, use `e.preventDefault()` (not `return false`).
  * In class components, event handlers need to be bound to the instance (or use arrow functions) to access `this`.
  * Passing arguments to event handlers can be done via arrow functions or the `bind` method, but care must be taken to avoid creating new functions on every render (performance impact).

* **Conditional rendering**
  * Conditional rendering in React works like regular JavaScript conditions, using `if`, `&&`, or ternary operators inside JSX.
  * Common patterns: `{isLoggedIn && <LogoutButton />}` renders the button only if `isLoggedIn` is true.
  * If-else blocks can be used outside JSX to return different elements, but inside JSX, ternary is often used.
  * Element variables can hold JSX and be conditionally rendered.
  * Avoid complex conditional logic inside JSX; extract to a function or component for readability.

* **Lists and keys**
  * Rendering lists is done using the `map()` function to transform an array of data into an array of JSX elements.
  * Keys are special string attributes that help React identify which items have changed, been added, or removed. They should be stable, unique, and preferably based on item IDs.
  * Using array indexes as keys is discouraged if the list can change (reorder, add/remove) because it can cause performance issues or bugs with component state.
  * Keys only need to be unique among siblings, not globally.
  * Missing keys can lead to warnings and inefficient updates (React may re-render entire lists).

* **Fragments**
  * Fragments allow grouping multiple children without adding extra nodes to the DOM. They are declared with `<React.Fragment>` or the shorthand `<>...</>`.
  * Fragments are useful when returning multiple elements from a component (e.g., a list of `<td>` elements) to avoid unnecessary wrapper divs.
  * They can accept keys only when using the explicit `<React.Fragment key={...}>` syntax (in maps).
  * Using fragments helps maintain cleaner DOM structure and avoids breaking CSS layouts like flexbox or grid that rely on parent-child relationships.
  * Fragments do not support props or styles (since they aren’t rendered).

* **Strict mode**
  * `<React.StrictMode>` is a tool for highlighting potential problems in an application. It does not render any visible UI but activates additional checks and warnings for its descendants.
  * It helps detect unsafe lifecycles, legacy string ref API usage, and unexpected side effects by double-invoking certain functions (e.g., `render`, `useEffect`).
  * It also warns about deprecated APIs and encourages best practices like using `key` props correctly.
  * Strict mode runs only in development mode; it has no impact on production builds.
  * It is recommended to wrap the entire app in strict mode during development to catch issues early.

### Components & Lifecycle
* **Component lifecycle overview**
  * Every React component goes through three main phases: mounting (when inserted into the DOM), updating (when props or state change), and unmounting (when removed).
  * In class components, lifecycle methods are available at each phase (e.g., `componentDidMount`, `componentDidUpdate`, `componentWillUnmount`).
  * Functional components use hooks like `useEffect` to mimic lifecycle behavior, combining mounting, updating, and unmounting into a single API.
  * Understanding the lifecycle helps in managing side effects (data fetching, subscriptions), optimizing performance, and avoiding memory leaks.
  * The lifecycle is also influenced by React’s rendering phases (render, pre-commit, commit) and concurrent features.

* **Mounting phase**
  * The mounting phase occurs when a component is first created and inserted into the DOM.
  * In class components, the order: constructor, `static getDerivedStateFromProps`, `render`, then `componentDidMount`.
  * `componentDidMount` is where side effects like API calls, subscriptions, or DOM manipulations should be performed, as it runs after the component is rendered.
  * In functional components, the `useEffect` hook with an empty dependency array (`[]`) runs after the initial render, analogous to `componentDidMount`.
  * Mounting is a good place to set up timers, event listeners, or external integrations, but remember to clean them up on unmount.

* **Updating phase**
  * The updating phase is triggered when props change, state is updated, or `forceUpdate` is called.
  * Class component lifecycle order: `static getDerivedStateFromProps`, `shouldComponentUpdate`, `render`, `getSnapshotBeforeUpdate`, `componentDidUpdate`.
  * `componentDidUpdate` receives previous props and state as arguments, useful for reacting to changes (e.g., refetching data when an ID prop changes).
  * In functional components, `useEffect` with dependencies runs after each update where dependencies have changed. `useLayoutEffect` can be used for synchronous DOM measurements.
  * Optimizing updates can be done with `React.memo`, `PureComponent`, or `shouldComponentUpdate` to prevent unnecessary re-renders.

* **Unmounting phase**
  * The unmounting phase occurs when a component is removed from the DOM.
  * In class components, `componentWillUnmount` is the only method called during unmount. It is used for cleanup: cancelling network requests, removing event listeners, clearing timers.
  * In functional components, the cleanup function returned from `useEffect` runs when the component unmounts (or before the next effect run if dependencies change).
  * Proper cleanup prevents memory leaks, such as dangling subscriptions or invalid state updates after unmount.
  * React guarantees that unmounting will happen when the component is no longer needed, e.g., conditional rendering or route changes.

* **Lifecycle methods (class components)**
  * `constructor`: initializes state and binds methods; called before mounting.
  * `static getDerivedStateFromProps`: used when state depends on props; returns an object to update state or null. Rarely needed.
  * `render`: required method that returns JSX; should be pure.
  * `componentDidMount`: invoked immediately after mounting; good for side effects.
  * `shouldComponentUpdate`: determines if re-rendering should happen; can optimize performance (return false to skip).
  * `getSnapshotBeforeUpdate`: captures DOM information (e.g., scroll position) before updates; passed to `componentDidUpdate`.
  * `componentDidUpdate`: runs after updates; for side effects based on changes.
  * `componentWillUnmount`: cleanup before destruction.

* **Functional lifecycle with hooks**
  * `useEffect` combines `componentDidMount`, `componentDidUpdate`, and `componentWillUnmount` into one API. The dependency array controls when it runs.
  * `useLayoutEffect` runs synchronously after DOM mutations but before browser paint; use for measurements or DOM manipulations that need to happen immediately.
  * `useMemo` and `useCallback` help optimize performance by memoizing values/functions, indirectly affecting update cycles.
  * Hooks allow splitting related logic into custom hooks, improving reusability and separation of concerns.
  * The order of hook calls must be consistent across renders; React relies on this to preserve state.

* **Side effects in React**
  * Side effects are operations that affect something outside the component’s scope, like data fetching, subscriptions, timers, or manually changing the DOM.
  * In React, side effects should be handled inside lifecycle methods (class) or `useEffect`/`useLayoutEffect` (functional).
  * The cleanup of side effects is crucial to avoid memory leaks; for example, unsubscribing from a WebSocket or clearing intervals.
  * Side effects that are synchronous and need to block painting (like measuring DOM) should use `useLayoutEffect`.
  * React’s strict mode double-invokes effects in development to help detect missing cleanup.

* **Controlled vs uncontrolled components**
  * Controlled components have their form data handled by React state. The input’s value is bound to state, and changes trigger state updates via `onChange`.
  * They provide a single source of truth, making it easier to validate, transform, or conditionally enable inputs.
  * Uncontrolled components store their own state in the DOM itself. You can access values via refs (e.g., `ref.current.value`).
  * Uncontrolled components are simpler for quick forms but less integrated with React’s data flow. They are useful when integrating with non-React code.
  * React recommends controlled components for most cases, but uncontrolled can be used for performance with large forms or when you need to interact with native form APIs.

* **Forms handling**
  * In React, forms can be built using controlled components, where each input element’s value is controlled by state.
  * Multiple inputs can be handled by using a single `onChange` handler that uses the input’s `name` attribute to update the corresponding state.
  * Form submission is handled by `onSubmit` on the `<form>` element, preventing default behavior and processing data.
  * Validation can be done on each change, on blur, or on submit. Libraries like Formik or React Hook Form simplify complex form logic.
  * Uncontrolled forms can still be used, but you lose real-time validation and reactivity.

* **Component composition**
  * Composition is a core pattern in React for reusing code and building complex UIs from simple pieces.
  * It involves combining smaller components to create larger ones, often using the `children` prop or explicit props that accept components.
  * Examples: a `Card` component that takes `header`, `body`, and `footer` as children; a `Modal` that renders its content via `children`.
  * Composition is preferred over inheritance for code reuse in React.
  * Higher-order components (HOCs) and render props are also composition patterns but are often replaced by hooks.

* **Container vs presentational components**
  * This pattern separates concerns: container components are concerned with how things work (data fetching, state management), while presentational components are concerned with how things look (UI, styling).
  * Presentational components are often functional, receive data via props, and emit events via callbacks.
  * Container components may be class-based or use hooks to manage logic and pass down data.
  * With hooks, the distinction blurs, but it’s still useful to separate logic from presentation for testability and reusability.
  * The pattern encourages a clean architecture where UI components are dumb and easy to reuse across different data sources.

### React Hooks
* **Rules of hooks**
  * Only call hooks at the top level of a React function component or custom hook, not inside loops, conditions, or nested functions. This ensures hooks are called in the same order on every render.
  * Only call hooks from React function components or custom hooks, not regular JavaScript functions.
  * Hooks rely on a fixed call order to preserve state between renders; breaking the order leads to bugs.
  * ESLint plugin `eslint-plugin-react-hooks` enforces these rules automatically.
  * Custom hooks should be named with `use` prefix to indicate they follow the rules and can contain hooks.

* **useState**
  * `useState` returns an array with the current state value and a setter function to update it. It accepts an initial state (or a function for lazy initialization).
  * The setter function can take a new value or a function that receives the previous state, useful for updates based on previous state.
  * State updates are asynchronous and may be batched; relying on the previous state via functional update avoids stale closures.
  * When state is an object, updates are not merged automatically (unlike class component `setState`); you need to manually merge using spread or `useReducer`.
  * Multiple `useState` calls can be used in a single component for independent pieces of state.

* **useEffect**
  * `useEffect` lets you perform side effects in functional components. It takes a function (the effect) and an optional dependency array.
  * The effect runs after every render by default; with an empty dependency array (`[]`), it runs only once after the initial render.
  * If dependencies are specified, the effect runs only when those dependencies change.
  * The effect can return a cleanup function that runs before the next effect or on unmount.
  * Common use cases: data fetching, subscriptions, manually changing DOM, timers.

* **useContext**
  * `useContext` accepts a context object (created by `React.createContext`) and returns the current context value.
  * It eliminates the need for a `Context.Consumer` component, making it easier to consume context in functional components.
  * When the context value changes, the component using `useContext` will re-render.
  * `useContext` can be combined with `useReducer` to create a simple state management solution.
  * It’s important to memoize context values to avoid unnecessary re-renders of consumers.

* **useReducer**
  * `useReducer` is an alternative to `useState` for managing complex state logic that involves multiple sub-values or transitions that depend on previous state.
  * It accepts a reducer function `(state, action) => newState` and an initial state, returning the current state and a `dispatch` function.
  * Actions are typically objects with a `type` property, making state transitions explicit and traceable.
  * `useReducer` is often used with context to manage global state without external libraries.
  * It can be optimized with `useMemo` and `useCallback` to prevent unnecessary re-renders when passing dispatch down.

* **useRef**
  * `useRef` returns a mutable ref object whose `.current` property persists across renders. Changing it does not trigger re-renders.
  * Common uses: accessing DOM elements directly (e.g., focusing an input), storing previous values, keeping mutable instance-like variables.
  * Unlike state, updating a ref does not cause a re-render; it’s useful for values that need to persist but don’t affect the UI.
  * Refs can also be used to store timer IDs, subscription handles, or any mutable data.
  * `useRef` can be used to create references to child components (via `forwardRef`) for imperative actions.

* **useMemo**
  * `useMemo` memoizes the result of a computation, recalculating only when dependencies change.
  * It is used to optimize expensive calculations and prevent them from running on every render.
  * `useMemo` returns the memoized value; if dependencies haven’t changed, it returns the cached value.
  * Overusing `useMemo` can be counterproductive; it should only be applied to computations that are genuinely expensive.
  * `useMemo` can also be used to stabilize objects passed as props to memoized children to prevent unnecessary re-renders.

* **useCallback**
  * `useCallback` returns a memoized version of a callback function that only changes if dependencies change.
  * It is useful when passing callbacks to optimized child components that rely on reference equality to prevent unnecessary re-renders (e.g., with `React.memo`).
  * `useCallback(fn, deps)` is equivalent to `useMemo(() => fn, deps)`.
  * Like `useMemo`, it should be used judiciously; not all functions need to be memoized.
  * It helps in maintaining stable references for dependency arrays in other hooks like `useEffect`.

* **Custom hooks**
  * Custom hooks are JavaScript functions that start with `use` and may call other hooks. They allow extracting component logic into reusable functions.
  * They enable sharing stateful logic between components without changing the component hierarchy (unlike HOCs or render props).
  * Examples: `useLocalStorage` to sync state with localStorage, `useFetch` for data fetching, `useWindowSize` for responsive design.
  * Custom hooks can return any value: state, functions, or JSX.
  * They follow the same rules as hooks and can be composed together.

* **Hook dependencies**
  * Dependency arrays in hooks (`useEffect`, `useMemo`, `useCallback`) list values that the hook depends on. If any dependency changes, the hook re-runs.
  * Omitting dependencies can lead to stale closures, where the callback references outdated variables.
  * The ESLint rule `exhaustive-deps` helps ensure dependencies are correctly specified.
  * For `useEffect`, if you intentionally want to run an effect only once, pass an empty array, but ensure you don’t reference any props/state that change over time without including them.
  * For `useCallback` and `useMemo`, missing dependencies can cause bugs where the memoized value/function doesn’t update when needed.

* **Stale closures**
  * A stale closure occurs when a callback captures variables from an older render (closure over outdated props/state) due to missing dependencies.
  * This commonly happens in `useEffect`, `useCallback`, or `setTimeout`/`setInterval` where the function references a value that has changed.
  * Using the functional update form for state setters can mitigate some stale closure issues.
  * Including all necessary dependencies in the dependency array ensures callbacks always use the latest values.
  * In `useEffect`, returning a cleanup function can help reset intervals or subscriptions to avoid stale closures.

* **Hook performance patterns**
  * Use `useMemo` and `useCallback` to avoid expensive recalculations and stabilize references, but measure performance to avoid premature optimization.
  * Split state into multiple `useState` or `useReducer` calls to avoid unnecessary updates (e.g., when unrelated pieces of state change together).
  * Lazy initialization of state with `useState(() => expensiveComputation())` to avoid running expensive logic on every render.
  * Avoid creating new objects/arrays in render that are passed as props to memoized components; use `useMemo` to stabilize them.
  * Use `useRef` for mutable values that don’t need to trigger re-renders, like tracking previous values.

### State Management
* **Local component state**
  * Local state is managed within a single component using `useState` or `useReducer`. It is not shared with other components unless lifted.
  * It is ideal for UI state like toggles, form inputs, or temporary data that doesn’t need to be global.
  * Local state should be as minimal as possible; derived data can be computed on the fly.
  * Encapsulating local state within a component promotes reusability and testability.
  * When local state grows complex, consider using `useReducer` or extracting logic into custom hooks.

* **Lifting state up**
  * Lifting state up is a pattern where state is moved to the closest common ancestor of components that need to share it.
  * The ancestor holds the state and passes it down via props, along with callbacks to modify it.
  * This promotes a single source of truth and makes data flow explicit.
  * It avoids duplication and inconsistencies that can arise when multiple components manage the same data separately.
  * However, excessive lifting can lead to prop drilling, which may be mitigated with composition or context.

* **Prop drilling**
  * Prop drilling occurs when props are passed through multiple intermediate components that don’t need the data, just to reach deeper components.
  * It makes components harder to refactor and can clutter the codebase with unnecessary props.
  * Solutions include using context, composition (e.g., passing components as children), or a global state management library.
  * Prop drilling is not inherently bad for shallow component trees; it’s a trade-off between explicitness and convenience.
  * React’s composition model (e.g., `children` prop) can reduce drilling by allowing components to receive rendered content directly.

* **Context API**
  * The Context API provides a way to share values (like theme, user) across the component tree without explicit prop passing.
  * It consists of `React.createContext`, a `Provider` component that supplies the value, and a `Consumer` or `useContext` hook to consume it.
  * Context is ideal for global but relatively static data; frequent updates can cause performance issues (all consumers re-render).
  * To optimize, split contexts by domain (e.g., `ThemeContext`, `UserContext`) and memoize provider values.
  * Context does not replace state management for complex state logic; it’s often combined with `useReducer` or external libraries.

* **Global state patterns**
  * Global state is data that many components across the app need access to. Patterns include:
    - Using React Context with `useReducer` for a lightweight Redux-like pattern.
    - Using external libraries like Redux, Zustand, or Recoil for more features (devtools, middleware, persistence).
    - Lifting state to the topmost component and passing down, but that’s not scalable for large apps.
  * Global state should be used sparingly; not everything needs to be global. Local state is often sufficient.
  * State normalization (like in Redux) helps avoid duplication and simplifies updates.
  * Consider using atomic state management (e.g., Recoil, Jotai) for fine-grained reactivity.

* **Redux fundamentals**
  * Redux is a predictable state container based on the Flux pattern. It uses a single store, actions, and reducers.
  * State is read-only and can only be changed by dispatching actions, which are plain objects describing what happened.
  * Reducers are pure functions that take the current state and an action, returning the new state immutably.
  * Middleware (like redux-thunk, redux-saga) allows side effects and async logic.
  * Redux DevTools provides time-travel debugging and action logging.

* **Redux Toolkit**
  * Redux Toolkit (RTK) is the official, opinionated toolset for efficient Redux development. It simplifies store setup, reducers, and actions.
  * It includes `createSlice`, which automatically generates action creators and action types based on reducer functions.
  * RTK Query is a powerful data fetching and caching solution built into RTK, eliminating the need for manual `useEffect` and caching logic.
  * It uses Immer internally to allow mutable-like updates in reducers (while producing immutable state).
  * RTK encourages best practices like using `configureStore` which sets up the store with good defaults (thunk middleware, DevTools).

* **React Query**
  * React Query (now TanStack Query) is a library for managing server state (asynchronous, remote data) in React.
  * It handles caching, background updates, stale data, pagination, and refetching out of the box.
  * Key concepts: queries (using `useQuery` for fetching), mutations (`useMutation` for updates), and query invalidation.
  * It reduces boilerplate compared to manual `useEffect` + `useState` patterns and optimizes performance by deduping requests.
  * React Query is not a state management library for client state; it’s often used alongside local state or context.

* **Zustand**
  * Zustand is a small, fast, and scalable state management library that uses a hook-based API.
  * It creates a store with a combination of state and actions, and components can subscribe to specific parts of the state.
  * Zustand doesn’t require providers; it works outside React’s context, making it lightweight and easy to use.
  * It supports middleware (like persist, devtools) and allows updates via immer or plain objects.
  * Zustand is often preferred for its simplicity and performance compared to Redux.

* **Recoil**
  * Recoil is a state management library by Facebook that introduces atoms (units of state) and selectors (derived state).
  * Atoms can be read from and written to from any component; components that use an atom subscribe to its changes.
  * Selectors compute derived state synchronously or asynchronously, allowing efficient recomputation only when dependencies change.
  * Recoil handles concurrent rendering well and avoids unnecessary re-renders by tracking dependencies at a granular level.
  * It’s still evolving but offers a compelling alternative to Redux for complex state needs.

* **State normalization**
  * State normalization is the practice of storing data in a structured, relational way to avoid duplication and simplify updates.
  * In Redux, normalized state often uses objects keyed by IDs, with separate arrays for IDs to maintain order.
  * For example, instead of storing an array of post objects with nested author objects, store posts and authors in separate lookup tables.
  * Normalization makes updates easier (you only update the author in one place) and improves performance for lookups.
  * Libraries like Normalizr can help normalize nested API responses.

* **Immutable updates**
  * In React, state should be treated as immutable to ensure predictable updates and enable features like time-travel debugging.
  * Direct mutation can lead to bugs where components don’t re-render because references don’t change.
  * Techniques for immutable updates: spread operator (`...`), `Array.map`, `Array.filter`, `Object.assign`, or libraries like Immer.
  * Immer allows writing mutable-like code that produces immutable states; it’s used in Redux Toolkit and Zustand.
  * Understanding immutable patterns is crucial for debugging and performance optimizations (e.g., `React.memo` relies on reference equality).

### React Rendering & Reconciliation
* **React reconciliation**
  * Reconciliation is the process by which React updates the DOM by comparing the new virtual DOM tree with the previous one.
  * It uses a diffing algorithm that makes assumptions to achieve O(n) complexity: elements of different types produce different trees, and keys help identify stable elements.
  * When comparing two trees, React first compares the root elements. If they are of different types, it tears down the old tree and builds a new one.
  * If the root elements are of the same type, React updates the attributes and then recursively processes children.
  * Reconciliation is what enables React to efficiently update the UI without re-rendering everything.

* **Diffing algorithm**
  * React’s diffing algorithm compares elements by their type and key. It prioritizes performance by avoiding unnecessary tree traversals.
  * When comparing lists, keys are crucial. React matches children by key, ensuring that items are moved, added, or removed correctly.
  * If keys are not used or are unstable, React may reorder or recreate elements unnecessarily, leading to performance issues and potential state loss.
  * The algorithm assumes that if the parent element type changes, the entire subtree is re-rendered.
  * Understanding the diffing algorithm helps in optimizing component hierarchies and choosing appropriate keys.

* **Fiber architecture**
  * React Fiber is the reimplementation of React’s core reconciliation algorithm, introduced in React 16.
  * It enables incremental rendering, allowing work to be split into chunks and prioritized, which is essential for concurrent rendering.
  * Fiber represents a unit of work with a linked list structure, making it possible to pause, resume, or abort rendering.
  * It also enables features like error boundaries, fragments, and returning multiple elements from render.
  * The Fiber architecture is the foundation for concurrent mode and Suspense.

* **Render vs commit phase**
  * The render phase is where React calls component functions and determines what changes need to be made. It is pure and can be paused or aborted.
  * The commit phase is where React applies those changes to the DOM. It is synchronous and cannot be interrupted.
  * During the render phase, React builds a tree of Fiber nodes and computes the effect list.
  * In the commit phase, React updates the DOM, runs lifecycle methods (e.g., `componentDidMount`), and executes effects (useLayoutEffect).
  * Concurrent rendering allows React to work on multiple tasks during the render phase without blocking the main thread.

* **Batching updates**
  * Batching is when React groups multiple state updates into a single re-render to avoid unnecessary work.
  * In React 17 and earlier, updates inside React event handlers were batched, but updates in promises, setTimeout, or async code were not.
  * React 18 introduced automatic batching for all updates, regardless of where they occur, using `createRoot`.
  * Batching improves performance by reducing the number of renders and ensures consistency (no partial renders).
  * You can opt out of batching using `flushSync` if needed.

* **Concurrent rendering**
  * Concurrent rendering is a set of features in React 18 that allow rendering to be interruptible, enabling a more responsive UI.
  * It lets React work on multiple tasks at once, prioritize urgent updates (like user input) over less urgent ones (like background data fetching).
  * Key APIs: `useTransition` for marking non-urgent state updates, `useDeferredValue` for deferring a value, and `Suspense` for data fetching.
  * Concurrent rendering is opt-in and requires using `createRoot` instead of `ReactDOM.render`.
  * It enables features like streaming server rendering and selective hydration.

* **Re-render triggers**
  * A re-render is triggered when a component’s state changes, when props change, or when a parent component re-renders (by default, children re-render even if props haven’t changed).
  * Context consumers re-render when the context value changes, regardless of memoization.
  * Calling `forceUpdate` in class components or using a hook that triggers update (like `useReducer` dispatch) also cause re-renders.
  * React’s default behavior is to re-render all child components when a parent renders, which can be optimized with `React.memo` or `useMemo`.
  * Understanding re-render triggers is key to performance optimization.

* **Key stability**
  * Keys help React identify which items in a list have changed, been added, or removed. They should be stable, predictable, and unique among siblings.
  * Using array indexes as keys can lead to issues when the list changes (e.g., reordering, adding/removing) because React may reuse component state incorrectly.
  * Keys should be based on the data, such as a unique ID from the database.
  * For static lists, index keys might be acceptable, but it’s still recommended to use stable IDs.
  * Key stability also matters for component instances: changing a key will cause React to unmount and remount the component.

* **Memoization strategies**
  * Memoization caches the result of a computation to avoid recalculating it when inputs haven’t changed.
  * In React, `React.memo` memoizes functional components, preventing re-renders if props are shallowly equal.
  * `useMemo` memoizes values, and `useCallback` memoizes functions.
  * Memoization should be applied where there’s a measurable performance gain, as it adds overhead for comparisons.
  * Strategies include: using `React.memo` on pure presentational components, memoizing expensive computations, and stabilizing props to leverage memoization.

* **Preventing unnecessary renders**
  * Techniques to avoid unnecessary re-renders include:
    - Using `React.memo` on components that render often with the same props.
    - Lifting state down or colocating state to reduce the scope of re-renders.
    - Using `useMemo` and `useCallback` to stabilize references.
    - Splitting context into multiple providers to avoid broad re-renders.
    - Using keys to help React reuse existing DOM elements.
  * Avoid creating new objects/arrays in render if they are passed as props to memoized components.
  * Profile with React DevTools to identify components that re-render unnecessarily.

### Performance Optimization
* **React.memo**
  * `React.memo` is a higher-order component that memoizes a functional component, causing it to re-render only when its props change (shallow comparison).
  * It’s useful for components that render frequently with the same props, reducing unnecessary renders.
  * You can provide a custom comparison function as the second argument to control equality checking.
  * `React.memo` only checks prop changes; if the component uses context or internal state, it will still re-render when those change.
  * Overusing `React.memo` on components that always receive different props can add overhead without benefit.

* **useMemo vs useCallback**
  * `useMemo` memoizes a computed value, while `useCallback` memoizes a function reference.
  * Both accept a dependency array and return a memoized version that only changes when dependencies change.
  * `useCallback(fn, deps)` is equivalent to `useMemo(() => fn, deps)`.
  * Use `useMemo` for expensive calculations; use `useCallback` for passing stable callbacks to child components that are memoized with `React.memo`.
  * Overusing them can harm performance because of the cost of dependency comparisons and memory overhead.

* **Lazy loading components**
  * Lazy loading defers loading of components until they are needed, reducing the initial bundle size and improving time-to-interactive.
  * React provides `React.lazy` for dynamic imports: `const MyComponent = React.lazy(() => import('./MyComponent'))`.
  * Lazy components must be wrapped in `<Suspense>` with a fallback UI while loading.
  * Code splitting can also be done at the route level using libraries like React Router.
  * Lazy loading is most effective for large components that are not immediately visible (e.g., modals, heavy charts).

* **Code splitting**
  * Code splitting is the practice of splitting the application bundle into smaller chunks that can be loaded on demand.
  * It can be achieved via dynamic `import()` syntax, which returns a promise and works with Webpack (or other bundlers).
  * React.lazy leverages code splitting to load components only when rendered.
  * Route-based code splitting is a common pattern where each route loads its own chunk.
  * Tools like Webpack’s `SplitChunksPlugin` can also split vendor code and shared modules.

* **Suspense**
  * Suspense is a React feature that allows components to “wait” for something (like data or code) before rendering, displaying a fallback UI in the meantime.
  * It works with `React.lazy` for code splitting and is also used for data fetching with frameworks like Relay or Next.js.
  * Suspense enables better user experience by coordinating loading states and avoiding spinners.
  * In React 18, Suspense supports streaming SSR and selective hydration.
  * Suspense boundaries can be nested to provide granular loading states.

* **Virtualization**
  * Virtualization (or windowing) is a technique to render only the visible portion of a large list or grid, improving performance.
  * Libraries like `react-window` and `react-virtualized` provide components that dynamically render items as the user scrolls.
  * Virtualization reduces the number of DOM nodes, memory usage, and render time.
  * It’s essential for displaying long lists (e.g., chat messages, infinite scroll feeds).
  * When implementing virtualization, maintain proper accessibility and keyboard navigation.

* **Debouncing & throttling**
  * Debouncing and throttling are techniques to limit the rate at which a function is executed, often used for performance optimization in event handlers (e.g., search input, scroll).
  * Debouncing delays execution until a certain amount of time has passed since the last call; useful for API calls as the user types.
  * Throttling ensures a function is called at most once in a specified time interval; useful for scroll or resize events.
  * In React, these can be implemented with libraries like lodash or custom hooks.
  * Be mindful of cleanup to avoid memory leaks and stale closures.

* **Bundle optimization**
  * Optimizing the JavaScript bundle improves load time and performance.
  * Techniques include:
    - Tree shaking to eliminate dead code (relies on ES modules).
    - Minification and compression (e.g., using Terser, gzip/Brotli).
    - Analyzing the bundle with tools like Webpack Bundle Analyzer.
    - Splitting vendor chunks and using dynamic imports.
    - Removing unnecessary dependencies and using lighter alternatives.
  * Also consider using modern JavaScript syntax (ES6+) and targeting modern browsers for smaller polyfills.

* **Tree shaking**
  * Tree shaking is a dead-code elimination technique that relies on static structure of ES modules (`import`/`export`).
  * It removes unused exports from the final bundle, reducing size.
  * For tree shaking to work, bundlers need to know which code is side-effect-free; configure `sideEffects` in package.json.
  * Libraries like lodash have modular versions (`lodash-es`) that support tree shaking.
  * In React, ensure you import only what you need (e.g., `import { useState } from 'react'` instead of `import React from 'react'` if possible, though React requires the default export for JSX).

* **Profiling with React DevTools**
  * React DevTools Profiler records performance information about component renders.
  * It helps identify slow components, unnecessary re-renders, and render phase durations.
  * The flamegraph view shows which components took the longest to render.
  * The ranked chart lists components by render time, helping prioritize optimization efforts.
  * Profiling should be done in production mode to get accurate timings (development mode is slower).

### React Router
* **Client-side routing**
  * Client-side routing allows navigation between views without full page reloads, using the History API to manipulate the URL.
  * In React, React Router is the most popular library for client-side routing.
  * It synchronizes the UI with the URL, enabling deep linking and bookmarkable states.
  * Client-side routing improves perceived performance because only the necessary components are re-rendered.
  * It requires a server fallback to serve the main HTML file for any route (in SPAs).

* **React Router architecture**
  * React Router v6 is the current version, which uses a declarative, component-based approach.
  * The core components: `<BrowserRouter>` (or `<HashRouter>`), `<Routes>`, `<Route>`, `<Link>`, and `<Navigate>`.
  * Routes are defined using `<Route>` elements inside `<Routes>`, which matches the URL and renders the appropriate component.
  * Nested routes are supported with the `element` prop and relative links.
  * The router uses a context provider to pass routing state to child components.

* **Route matching**
  * Route matching in React Router v6 is done by the `<Routes>` component, which compares the current URL with the `path` prop of its child `<Route>` elements.
  * It uses a scoring system to find the best match, and only the first matching route is rendered.
  * Paths can be exact (default behavior in v6, but you can use `*` for catch-all).
  * Dynamic segments are denoted by `:` (e.g., `/users/:id`), and the value is available via the `useParams` hook.
  * Optional segments and wildcards are also supported.

* **Nested routes**
  * Nested routes allow components to be rendered inside parent components based on the URL structure.
  * In v6, nesting is achieved by placing `<Route>` elements inside a parent `<Route>` or using the `element` prop to render an outlet.
  * The parent component renders an `<Outlet />` component that acts as a placeholder for the child route.
  * Relative links and paths automatically work with nested routes.
  * Nested routes simplify layout management and code organization.

* **Dynamic routes**
  * Dynamic routes contain parameters that vary, such as `/users/:userId`. The parameter is extracted via `useParams`.
  * Multiple parameters can be defined (e.g., `/category/:catId/product/:prodId`).
  * Dynamic routes can also be used with optional parameters using `?` or using `*` for catch-all (e.g., `/files/*` to match any path after `/files`).
  * These parameters can be used to fetch data or conditionally render content.

* **Navigation patterns**
  * Navigation in React Router is done via `<Link>` or `<NavLink>` (which adds active styling).
  * Programmatic navigation can be achieved with the `useNavigate` hook (or `Navigate` component for redirects).
  * `useNavigate` returns a function that can be called with a to path or delta (go back/forward).
  * Navigation can be replaced (not adding to history) using `navigate(to, { replace: true })`.
  * For redirects in loaders or actions (data APIs), use `redirect` helper.

* **Protected routes**
  * Protected routes restrict access based on authentication status. Typically implemented using a wrapper component that checks auth state and either renders the child route or redirects to login.
  * In v6, you can create a component that uses `useNavigate` or `<Navigate>` conditionally.
  * You can also use a layout route that handles authentication and renders `<Outlet>` only if authenticated.
  * For more complex logic, consider using route loaders (data APIs) to perform checks before rendering.
  * State management for auth (like context) is often combined with protected routes.

* **Route guards**
  * Route guards are mechanisms to prevent navigation based on conditions (e.g., unsaved forms, permissions).
  * React Router v6 does not have built-in guards, but you can use the `useBlocker` hook (unstable) or create custom logic with `useEffect` and `useNavigate`.
  * Another approach is to use a custom component that checks conditions in the route's `element` and redirects if needed.
  * For data loading, you can use route loaders to enforce permissions before rendering.
  * Libraries like `react-router-guards` exist but are not officially maintained.

* **Lazy routes**
  * Lazy loading routes involves using `React.lazy` to dynamically import route components.
  * Wrap the lazy component in `<Suspense>` with a fallback.
  * Example: `<Route path="/dashboard" element={<React.Suspense fallback={<Spinner />}><Dashboard /></React.Suspense>} />`.
  * Lazy routes reduce initial bundle size by splitting code per route.
  * React Router v6 supports this seamlessly; just ensure the lazy component is imported correctly.

### Forms & Validation
* **Controlled forms**
  * In controlled forms, form inputs are controlled by React state. The input’s `value` is set from state, and `onChange` updates state.
  * This gives React full control over the form data, enabling instant validation, conditional fields, and dynamic inputs.
  * Each change triggers a re-render, which can be performance-intensive for large forms.
  * Controlled components are the recommended approach for most forms due to their predictability.
  * They also make it easier to integrate with form libraries.

* **Uncontrolled forms**
  * Uncontrolled forms store input values in the DOM itself. You access them via refs (e.g., `ref.current.value`) when needed (e.g., on submit).
  * They are simpler to implement and can be more performant for very large forms because they don’t re-render on every keystroke.
  * However, they don’t provide real-time validation or reactivity.
  * Uncontrolled forms are useful when integrating with non-React code or when you need to access form data imperatively.
  * Default values can be set using the `defaultValue` attribute instead of `value`.

* **Form performance**
  * Performance issues in forms can arise from excessive re-renders (in controlled forms) or complex validation.
  * Techniques to improve form performance:
    - Use uncontrolled components for fields that don’t need live validation.
    - Debounce validation or input handlers.
    - Memoize expensive components and avoid re-rendering the entire form on each keystroke.
    - Use libraries like React Hook Form that minimize re-renders by isolating field updates.
    - Virtualize long lists of form fields if necessary.

* **Form libraries**
  * Form libraries simplify form state management, validation, and submission handling.
  * They abstract away boilerplate and provide utilities for complex forms.
  * Popular libraries: Formik, React Hook Form, and Final Form.
  * Benefits include easier validation schema integration (Yup, Zod), better performance (isolated re-renders), and built-in error handling.
  * They also support field arrays, dynamic fields, and form-level/submit validation.

* **Formik**
  * Formik is a popular library for building forms in React. It manages form state, validation, and submission.
  * It provides hooks like `useFormik` or a `<Formik>` component with render props.
  * Formik supports synchronous and asynchronous validation, and integrates with Yup for schema validation.
  * It can be verbose but offers a lot of flexibility. However, it may cause re-renders of the entire form on each change.
  * Formik is suitable for medium-complexity forms and teams familiar with its patterns.

* **React Hook Form**
  * React Hook Form is a performant library that uses uncontrolled components and refs to reduce re-renders.
  * It provides the `useForm` hook, which registers inputs and tracks their values without causing re-renders on every change.
  * Validation can be done with built-in rules, HTML5 validation, or schema libraries like Yup and Zod.
  * It supports field arrays, error handling, and integration with UI libraries.
  * React Hook Form is often preferred for its performance and smaller bundle size.

* **Validation strategies**
  * Validation can be performed on change, on blur, or on submit. Each has trade-offs.
  * On-change validation provides immediate feedback but can be distracting and performance-heavy.
  * On-blur validation is less intrusive but may delay feedback until the user leaves the field.
  * On-submit validation ensures all fields are validated at once, but users may not see errors until after submission.
  * Schema-based validation (with Yup, Zod) centralizes validation rules and makes them reusable.
  * Async validation (e.g., checking username availability) should be debounced to avoid excessive requests.

* **Async validation**
  * Async validation involves checking with a server (e.g., username uniqueness) asynchronously.
  * It should be debounced to avoid API spam.
  * Most form libraries support async validation by returning a promise from the validation function.
  * Proper error handling and loading states are important for good UX.
  * Async validation can be combined with on-blur or on-change strategies.

### API Integration
* **Fetching data in React**
  * Data fetching in React typically occurs inside `useEffect` (or lifecycle methods) for client-side apps.
  * State variables hold the data, loading status, and error.
  * Fetching logic should handle cleanup to avoid state updates on unmounted components (use an abort controller or flag).
  * For better performance and caching, use libraries like React Query, SWR, or RTK Query.
  * Data fetching can also happen in route loaders (Next.js, Remix) or via server components.

* **useEffect data fetching**
  * A common pattern: `useEffect(() => { let isMounted = true; fetchData().then(data => { if (isMounted) setData(data); }); return () => { isMounted = false; }; }, [deps])`.
  * Dependencies should include any variables used in the fetch (e.g., pagination, filters).
  * Race conditions can occur if the dependency changes before the previous request completes; use abort controller or ignore stale responses.
  * Error handling should set error state and possibly retry logic.
  * For complex data fetching, consider using a dedicated library to avoid reinventing the wheel.

* **REST integration**
  * REST APIs are commonly consumed using `fetch` or libraries like Axios.
  * Typically, you define service functions for each endpoint (GET, POST, PUT, DELETE) that return promises.
  * In React, you call these functions inside effects or event handlers, managing loading and error states.
  * Authentication headers (e.g., JWT) need to be attached to requests, often via interceptors.
  * Caching and deduplication can be handled manually or with a library.

* **GraphQL integration**
  * GraphQL clients like Apollo Client or Urql provide hooks for queries and mutations.
  * They handle caching, normalized data, and optimistic updates automatically.
  * In Apollo, `useQuery` hook manages loading, data, and error states; `useMutation` for updates.
  * GraphQL allows fetching exactly the data needed, reducing over-fetching and under-fetching.
  * Subscriptions for real-time data are also supported.

* **Axios usage**
  * Axios is a promise-based HTTP client with features like request/response interceptors, automatic JSON transformation, and cancellation.
  * It provides a cleaner API than native fetch for some use cases (e.g., timeout, progress).
  * In React, you can create an Axios instance with base URL and default headers.
  * Interceptors can be used to add auth tokens, handle global errors, or refresh tokens.
  * Axios supports cancellation using cancel tokens (or AbortController in newer versions).

* **Error handling**
  * API errors should be caught and displayed to users appropriately (toast messages, inline errors).
  * Distinguish between network errors, server errors (5xx), and client errors (4xx) for better feedback.
  * Use try/catch blocks or `.catch` to handle promise rejections.
  * Retry logic can be implemented for transient errors (e.g., with exponential backoff).
  * Global error boundaries can catch rendering errors but not async errors; you need to handle those in the component.

* **Loading states**
  * While data is being fetched, show a loading indicator (spinner, skeleton) to improve UX.
  * Use a boolean loading state that is set to true before the fetch and false after completion.
  * For multiple independent fetches, consider having separate loading states or a more sophisticated state machine.
  * Libraries like React Query provide built-in loading states and auto-retry.
  * Skeleton screens are often better than spinners because they give a sense of the content structure.

* **Caching strategies**
  * Caching prevents unnecessary network requests and improves perceived performance.
  * Simple caching can be done with a JavaScript object keyed by request URL.
  * More advanced caching includes stale-while-revalidate (used by React Query/SWR), where stale data is shown while re-fetching in background.
  * Cache invalidation is tricky; strategies include time-based expiry, mutation triggers, or manual refetch.
  * Normalized caching (Apollo) stores entities by ID and automatically updates all references.

* **Optimistic updates**
  * Optimistic updates immediately update the UI before the server confirms, assuming the operation will succeed.
  * If the request fails, you roll back to the previous state (with a notification).
  * This provides a snappy user experience, especially for actions like toggling likes or adding comments.
  * Libraries like React Query and Apollo support optimistic updates via updater functions.
  * Care must be taken to handle concurrent updates and ensure consistency.

### Styling in React
* **CSS modules**
  * CSS Modules locally scope CSS by automatically generating unique class names, avoiding global conflicts.
  * In React, you import a `.module.css` file as an object, and use styles.className.
  * They are supported by default in Create React App and Vite.
  * CSS Modules allow writing regular CSS with all its features (media queries, animations) while keeping styles modular.
  * They are a good middle ground between global CSS and CSS-in-JS.

* **Styled components**
  * Styled Components is a CSS-in-JS library that allows writing actual CSS in JavaScript using tagged template literals.
  * It creates React components with styles attached, e.g., `const Button = styled.button`background: blue;``.
  * Styles are scoped to the component, and dynamic props can be used via interpolation.
  * It supports theming, automatic vendor prefixing, and critical CSS extraction.
  * Performance considerations: styles are injected at runtime; for server-side rendering, you need to collect and inject styles.

* **Emotion**
  * Emotion is another CSS-in-JS library with similar features to Styled Components but with a smaller footprint and flexible APIs.
  * It offers both styled components and the `css` prop for inline styles.
  * Emotion is used by many frameworks and can be used with or without React.
  * It also supports source maps, theming, and server-side rendering.
  * The `css` prop allows writing styles directly on elements, which can be convenient.

* **Tailwind with React**
  * Tailwind CSS is a utility-first CSS framework that provides low-level utility classes.
  * In React, you apply these classes via `className` (e.g., `className="flex items-center p-4"`).
  * Tailwind promotes rapid UI development and consistency through a design system.
  * It can be customized via a config file and purges unused CSS in production.
  * With React, you can combine Tailwind with component composition to create reusable UI elements.

* **Inline styles**
  * Inline styles in React are specified as objects with camelCased properties (e.g., `style={{ color: 'red' }}`).
  * They are useful for dynamic styles that depend on props or state.
  * However, they lack support for media queries, pseudo-classes, and animations.
  * Inline styles can be less performant than CSS classes because they are applied to each element individually.
  * They are often used in combination with other styling methods for specific cases.

* **Theming**
  * Theming involves providing a set of design tokens (colors, fonts, spacing) that can be accessed throughout the app.
  * CSS-in-JS libraries (Styled Components, Emotion) offer a `ThemeProvider` context to pass theme down.
  * With CSS Modules or Tailwind, theming can be achieved via CSS custom properties (variables) or by generating different class sets.
  * React Context can also be used to provide theme values, but may cause re-renders if not optimized.
  * Theming enables light/dark mode switching and brand consistency.

* **Design systems**
  * A design system is a collection of reusable components, patterns, and guidelines for building applications.
  * In React, design systems are often implemented as a component library (e.g., Material-UI, Chakra UI) or an in-house library.
  * They enforce consistency, speed up development, and bridge design and development.
  * Building a design system involves creating a set of primitive components (Button, Input, Modal) with consistent APIs and styling.
  * Tools like Storybook help document and develop design system components in isolation.

### Testing React
* **Unit testing components**
  * Unit tests focus on testing individual components in isolation, verifying that they render correctly and handle props/state.
  * Tools: Jest as the test runner, React Testing Library (RTL) for rendering and interacting with components.
  * Tests should resemble how users interact with the component (queries by role/text, not implementation details).
  * Mock child components or external dependencies to isolate the unit.
  * Test edge cases: empty props, loading states, error states.

* **Integration testing**
  * Integration tests verify that multiple components work together, often involving data fetching, context, or routing.
  * They may render a whole page or feature and test user flows.
  * Mock API calls using libraries like MSW (Mock Service Worker) to intercept network requests.
  * Integration tests are slower but give more confidence than unit tests.
  * Focus on critical user journeys (e.g., login, add to cart).

* **Jest**
  * Jest is a JavaScript testing framework developed by Facebook, commonly used with React.
  * It provides a test runner, assertion library, mocking utilities, and code coverage.
  * Jest’s snapshot testing can capture the rendered output of a component to detect unexpected changes.
  * It integrates with Babel, TypeScript, and various testing libraries.
  * Jest’s watch mode runs only tests related to changed files, improving development speed.

* **React Testing Library**
  * React Testing Library (RTL) is a lightweight library for testing React components by simulating user interactions.
  * Its guiding principle: test components as users would, not implementation details.
  * Provides queries (getBy, findBy, queryBy) to find elements based on accessibility roles, text, etc.
  * Utilities like `fireEvent` and `userEvent` simulate clicks, typing, etc.
  * Encourages writing maintainable tests that don’t break when refactoring.

* **Mocking APIs**
  * When testing components that fetch data, you should mock the API layer to avoid network calls.
  * Options:
    - Jest’s manual mocks: mock modules like `axios` or `fetch`.
    - Mock Service Worker (MSW): intercepts network requests at the service worker level, works in both Node and browser.
  - MSW allows writing tests with realistic server responses without changing component code.
  * Mocked responses should cover success, error, and loading states.

* **Snapshot testing**
  * Snapshot tests capture the rendered output of a component and compare it to a stored snapshot file.
  * They are useful for detecting unintended changes in UI, but can be brittle (failing on trivial changes).
  * Snapshots should be committed to version control and reviewed when they change.
  * Overuse of snapshots can lead to large, meaningless snapshots; prefer targeted assertions.
  * They are often used in combination with other tests.

* **Test coverage**
  * Code coverage measures how much of your code is executed during tests (lines, branches, functions).
  * Jest can generate coverage reports with `--coverage` flag.
  * While 100% coverage is not always feasible or necessary, it helps identify untested critical paths.
  * Coverage should be used as a guideline, not a goal; focus on testing behavior rather than covering lines.
  * Tools like Coveralls or Codecov can track coverage over time.

* **Testing hooks**
  * Custom hooks can be tested in isolation using `@testing-library/react-hooks` (now part of RTL as `renderHook`).
  * `renderHook` renders a test component that uses the hook and returns the result, which can be accessed and asserted.
  * You can also test hooks by using them in a component and testing the component’s behavior.
  * For hooks that use context, you need to wrap them in a provider.
  * Ensure to test updates and cleanup.

### Advanced React Concepts
* **Higher-order components**
  * A higher-order component (HOC) is a function that takes a component and returns a new component with enhanced functionality.
  * HOCs are used for cross-cutting concerns like authentication, logging, or data fetching (e.g., `withRouter`, `connect` in Redux).
  * They promote code reuse but can lead to wrapper hell and naming collisions.
  * HOCs should be pure and not mutate the original component.
  * With hooks, many HOC use cases can be replaced by custom hooks, which are often simpler.

* **Render props**
  * Render props is a pattern where a component accepts a function as a prop that returns JSX, allowing the parent to control rendering.
  * Example: `<DataFetcher render={(data) => <div>{data}</div>} />`.
  * It enables sharing logic without HOCs, but can lead to deeply nested render functions.
  * Render props are flexible and can be used for state sharing, mouse tracking, etc.
  * Hooks have largely superseded render props for logic reuse, but the pattern is still valid.

* **Compound components**
  * Compound components are a set of components that work together to form a cohesive unit, sharing implicit state via context or cloneElement.
  * Example: `<Select>` with `<Select.Option>` children, where the parent manages the selected value.
  * They provide a flexible API while keeping internal state management hidden.
  * Common in UI libraries (e.g., Tabs, Accordion).
  * Implementation often uses React Context to share state between parent and children.

* **Portals**
  * Portals allow rendering children into a DOM node that exists outside the parent component’s hierarchy (e.g., modals, tooltips).
  * Created with `ReactDOM.createPortal(child, container)`.
  * Portal children still participate in React’s event bubbling (events bubble up through React tree, not DOM tree).
  * Useful for overlays that need to break out of overflow:hidden or z-index contexts.
  * Ensure the portal container exists in the DOM (often appended to document.body).

* **Error boundaries**
  * Error boundaries are React components that catch JavaScript errors anywhere in their child component tree and display a fallback UI.
  * They are implemented using class components with `static getDerivedStateFromError()` and/or `componentDidCatch()`.
  * Error boundaries do not catch errors in event handlers, async code, or server-side rendering.
  * In functional components, there is no direct hook equivalent; you must use a class component or a library like `react-error-boundary`.
  * They help prevent the entire app from crashing due to a component error.

* **Context performance pitfalls**
  * When a context value changes, all consumers (components using `useContext` or `Context.Consumer`) re-render, even if they only need part of the value.
  * To mitigate, split context into multiple smaller contexts (e.g., `ThemeContext`, `UserContext`) so changes only affect relevant consumers.
  * Memoize the context value object to avoid unnecessary re-renders (use `useMemo`).
  * For frequent updates, consider using state management libraries that allow fine-grained subscriptions (e.g., Zustand, Recoil).
  * Alternatively, pass separate contexts for state and dispatch (like Redux’s store).

* **Controlled re-rendering**
  * Controlled re-rendering refers to techniques that give you fine-grained control over when and why components re-render.
  * Examples: using `React.memo`, `useMemo`, `useCallback`, and key changes.
  * It also includes patterns like lifting state down so that only the relevant part re-renders.
  * In some cases, you may want to force a re-render (e.g., with a key change) to reset state.
  * Avoid over-optimizing; let React handle most cases unless you identify performance bottlenecks.

* **Ref forwarding**
  * Ref forwarding is a technique to pass a ref from a parent component to a child component’s DOM element.
  * It is done using `React.forwardRef`, which returns a component that can receive a `ref` prop.
  * Useful for accessing DOM nodes in reusable components (e.g., input focus, scroll position).
  * In functional components, you can forward refs to a DOM element or another component.
  * Ref forwarding can also be used with `useImperativeHandle` to customize the instance value exposed to parent refs.

* **React internals overview**
  * Understanding React internals helps in debugging and optimizing performance.
  * React elements are plain objects describing what to render.
  * The reconciler (Fiber) manages the work loop, diffing, and committing.
  * Hooks are implemented as linked lists on the Fiber node, maintaining order.
  * The scheduler prioritizes updates and cooperatively schedules work.
  * Knowing these internals can help when working with complex scenarios like concurrent rendering or custom renderers.

### React 18+ Features
* **Concurrent rendering**
  * Concurrent rendering is an opt-in feature that allows React to work on multiple tasks simultaneously and interrupt rendering to handle high-priority updates.
  * It is enabled by using `createRoot` instead of `ReactDOM.render`.
  * Concurrent features include `startTransition`, `useDeferredValue`, and `<Suspense>` for data fetching.
  * The goal is to make apps more responsive by avoiding blocking the main thread.
  * It also enables new capabilities like streaming server rendering.

* **Automatic batching**
  * React 18 introduces automatic batching for all updates, even in promises, setTimeout, and event handlers.
  * Previously, only React event handlers were batched; now updates are batched wherever they occur.
  * This reduces the number of renders and improves performance.
  * You can opt out using `flushSync` from `react-dom` if you need immediate DOM updates.
  * Automatic batching works in both concurrent and legacy modes.

* **Suspense improvements**
  * Suspense in React 18 is enhanced to support data fetching on the server and streaming HTML.
  * It allows components to suspend while loading data, showing fallback UIs.
  * On the server, Suspense enables streaming HTML, sending the fallback first and then replacing it with the actual content once ready.
  * Suspense also works with lazy loading and can be used to coordinate loading states.
  * The `SuspenseList` component helps control the order of loading states.

* **Transitions**
  * Transitions are a new concept in React 18 for marking non-urgent state updates that can be interrupted.
  * The `startTransition` API allows you to wrap state updates that may cause slow renders, telling React to keep the UI responsive.
  * During a transition, React can show a pending indicator (e.g., via `useTransition`’s `isPending`).
  * Transitions help avoid jank by deferring less important updates.
  * They are a key part of concurrent rendering.

* **useTransition**
  * `useTransition` is a hook that returns a `isPending` flag and a `startTransition` function.
  * `startTransition` wraps state updates that should be treated as transitions.
  * While the transition is pending, you can show a loading state (e.g., a spinner) to indicate background work.
  * Transitions are interruptible: if a higher-priority update occurs (like typing), React will pause the transition.
  * Use `useTransition` for expensive re-renders like filtering a large list or switching tabs.

* **useDeferredValue**
  * `useDeferredValue` accepts a value and returns a deferred version that may lag behind the original during urgent updates.
  * It’s useful for deferring rendering of a part of the UI based on a value that changes frequently (e.g., search input).
  * The deferred value is updated in the background, allowing urgent updates (like typing) to stay responsive.
  * It’s similar to debouncing but built into React and integrated with concurrent rendering.
  * Use it to keep the UI responsive while rendering expensive derived data.

* **Streaming SSR**
  * React 18 supports streaming server-side rendering with `renderToPipeableStream` (Node) or `renderToReadableStream` (web streams).
  * The server can send HTML in chunks as they are ready, including Suspense boundaries.
  * The initial HTML contains fallback content (e.g., spinners) which is later replaced by the actual content on the client (hydration).
  * This improves perceived performance because users see content faster.
  * Streaming SSR works with Suspense and selective hydration.

### Server-Side Rendering
* **SSR concepts**
  * Server-side rendering (SSR) generates the HTML for a page on the server and sends it to the client, where React then hydrates it to make it interactive.
  * Benefits: better SEO (search engines can crawl fully rendered content) and faster initial page load (users see content sooner).
  * Challenges: increased server load, complexity, and potential for mismatches between server and client.
  * SSR can be implemented with frameworks like Next.js or manually with Node.js and ReactDOMServer.
  * Hydration is the process of attaching event listeners to the server-rendered HTML.

* **Hydration**
  * Hydration is when React takes over the server-rendered HTML and attaches event handlers, making the page interactive.
  * The initial HTML must match the React component tree exactly; mismatches can cause hydration errors and force React to re-render on the client.
  * React 18 improves hydration with selective hydration, where parts of the page can hydrate independently based on user interaction.
  * Hydration can be resource-intensive; techniques like progressive hydration can improve performance.
  * In concurrent rendering, hydration can be interrupted and prioritized.

* **Next.js architecture**
  * Next.js is a React framework that provides SSR, static site generation (SSG), and other features out of the box.
  * It uses file-based routing; pages are React components in the `pages` directory.
  * Data fetching methods: `getServerSideProps` (SSR), `getStaticProps` (SSG), `getStaticPaths` (dynamic SSG).
  * It also supports API routes, middleware, and incremental static regeneration (ISR).
  * Next.js optimizes performance with automatic code splitting, image optimization, and prefetching.

* **Static generation**
  * Static generation (SSG) pre-renders pages at build time, generating HTML files that can be cached and served via CDN.
  * It’s ideal for content that doesn’t change often (blogs, marketing pages).
  * In Next.js, `getStaticProps` fetches data at build time and passes it as props to the page component.
  * For dynamic routes, `getStaticPaths` defines which paths to pre-render.
  * SSG offers the best performance because there’s no server-side computation per request.

* **Incremental static regeneration**
  * Incremental Static Regeneration (ISR) allows you to update static content after the site is built, without rebuilding the entire site.
  * In Next.js, you can set a `revalidate` property in `getStaticProps` to specify how often the page should be regenerated (in seconds).
  * When a request comes in, the stale page is served while the new version is generated in the background.
  * ISR combines the benefits of static generation with dynamic updates, making it suitable for e-commerce, blogs, etc.
  * It reduces server load while keeping content fresh.

* **SEO considerations**
  * SSR and SSG improve SEO because search engines can crawl fully rendered content.
  * For SPAs, search engines may execute JavaScript, but it’s not guaranteed; SSR provides a reliable solution.
  * Meta tags (title, description) can be set dynamically on the server using frameworks like Next.js (`next/head`).
  * Proper use of semantic HTML, heading structure, and alt text also boosts SEO.
  * Performance (Core Web Vitals) is an SEO factor; SSR can help with Largest Contentful Paint (LCP).

### React Ecosystem
* **Next.js**
  * Next.js is the most popular React framework for production, offering SSR, SSG, API routes, and more.
  * It provides a zero-config setup with built-in optimizations like image optimization, font optimization, and script loading.
  * The App Router (Next.js 13+) introduces React Server Components, streaming, and nested layouts.
  * Next.js is ideal for SEO-critical applications, e-commerce, and content-driven sites.
  * It has a large ecosystem of plugins and is supported by Vercel.

* **Vite**
  * Vite is a build tool that provides a fast development server with hot module replacement (HMR) and optimized production builds.
  * It supports React out of the box via plugins (`@vitejs/plugin-react`).
  * Vite uses native ES modules in development, avoiding bundling, leading to faster startup and updates.
  * It’s an alternative to Create React App (CRA) and offers more flexibility and speed.
  * Many new React projects now use Vite due to its performance.

* **CRA alternatives**
  * Create React App (CRA) has been the standard for bootstrapping React apps, but it’s slower and less flexible.
  * Alternatives include Vite, Next.js (for SSR/SSG), Remix (another full-stack framework), and Parcel.
  * These alternatives offer faster builds, better customization, and additional features.
  * For simpler SPAs, Vite is a popular choice; for more advanced needs, Next.js or Remix.

* **Redux ecosystem**
  * The Redux ecosystem includes Redux Toolkit (official toolset), Redux DevTools, and middleware like redux-thunk, redux-saga, and redux-observable.
  * Redux Toolkit simplifies store setup, reducers, and immutable updates with Immer.
  * RTK Query is a powerful data fetching and caching solution built into Redux Toolkit.
  * The ecosystem also includes bindings for React (`react-redux`) and utilities like reselect for memoized selectors.

* **React Query ecosystem**
  * React Query (TanStack Query) is the leading library for server-state management.
  * It provides hooks for queries and mutations, with built-in caching, background refetching, and pagination.
  * The ecosystem includes TanStack Table for data tables, TanStack Router for routing, and TanStack Form.
  * React Query is often used alongside Zustand or Redux for client state.
  * It supports React Native and other frameworks.

* **UI libraries**
  * UI libraries provide pre-built, customizable components to speed up development.
  * Popular ones: Material-UI (MUI), Chakra UI, Ant Design, and React Bootstrap.
  * They offer components like buttons, modals, tables, and form controls with consistent design.
  * Most support theming, accessibility, and are tree-shakable.
  * Choosing a UI library depends on design preferences, customization needs, and bundle size.

* **Material UI**
  * Material-UI (now MUI) is a popular React component library implementing Google’s Material Design.
  * It provides a vast set of components with extensive customization via themes and styles (emotion/styled).
  * MUI includes advanced components like data grid, date pickers, and autocomplete.
  * It supports server-side rendering and is highly accessible.
  * The library has a large community and thorough documentation.

* **Chakra UI**
  * Chakra UI is a modular, accessible component library with a focus on simplicity and developer experience.
  * It uses styled-system for styling props, allowing quick styling via props like `mt={4}`.
  * Components are built with accessibility in mind (WAI-ARIA compliant).
  * Chakra supports dark mode out of the box with a theme provider.
  * It’s lightweight and customizable, making it a favorite for startups and personal projects.

* **Ant Design**
  * Ant Design is a comprehensive UI library originating from China, with a set of high-quality components.
  * It follows the Ant Design language, which is more opinionated and enterprise-oriented.
  * Ant Design includes i18n support, advanced tables, and form validation.
  * It has a large component set but can be heavy; tree-shaking helps.
  * Suitable for admin dashboards and enterprise applications.

### Accessibility
* **ARIA roles**
  * ARIA (Accessible Rich Internet Applications) roles provide additional semantics to elements for assistive technologies.
  * In React, you can add ARIA attributes directly (e.g., `role="button"`, `aria-label="Close"`).
  * Common roles: `navigation`, `main`, `dialog`, `alert`, `tablist`.
  * ARIA should only be used when native HTML semantics are insufficient.
  * Ensure roles are used correctly to avoid confusing screen readers.

* **Keyboard navigation**
  * Ensure all interactive elements are focusable and operable via keyboard.
  * Use semantic HTML (buttons, links) that inherently support keyboard events.
  * Manage focus appropriately: when opening a modal, move focus inside; when closing, return focus to the trigger.
  * Implement custom keyboard shortcuts for complex widgets (e.g., arrow keys in tabs).
  * Test with only keyboard to verify tab order and focus visibility.

* **Screen reader support**
  * Screen readers rely on semantic HTML and ARIA to convey information.
  * Use proper heading hierarchy (`h1`-`h6`), landmarks, and alt text for images.
  * Ensure dynamic content updates are announced (e.g., using `aria-live` regions).
  * Test with screen readers like NVDA, VoiceOver, or JAWS.
  * Provide text alternatives for non-text content (icons, charts).

* **Semantic HTML in React**
  * Use semantic elements like `<header>`, `<nav>`, `<main>`, `<article>`, `<section>`, `<button>` instead of generic `<div>`s.
  * Semantic HTML improves accessibility, SEO, and code readability.
  * In React, you can directly use these elements in JSX.
  * For custom components that act like buttons, use `<button>` or add `role="button"` and keyboard handlers.
  * Avoid using `<div>` for click handlers; use `<button>` when appropriate.

* **Accessibility testing**
  * Automated tools: `eslint-plugin-jsx-a11y` catches common accessibility issues in code.
  * Browser extensions like axe DevTools, WAVE, or Lighthouse can audit pages.
  * Manual testing with keyboard and screen readers is essential.
  * Consider including accessibility tests in CI pipelines (e.g., jest-axe).
  * Regular accessibility audits help maintain standards.

### Security
* **XSS in React**
  * React automatically escapes values embedded in JSX, preventing XSS attacks by default.
  * However, `dangerouslySetInnerHTML` bypasses this and should be used with caution (sanitize HTML first).
  * User-provided URLs in `href` or `src` can lead to javascript: links; validate and sanitize.
  * Avoid using `eval()` or `new Function()` with user input.
  * Keep React and dependencies updated to patch known vulnerabilities.

* **Sanitizing inputs**
  * Sanitize any user input that will be rendered as HTML (e.g., with `dompurify`).
  * For URLs, validate protocol (http/https) to prevent javascript: URLs.
  * When using `dangerouslySetInnerHTML`, always pass through a sanitization library.
  * In forms, validate and sanitize on both client and server sides.
  * Be cautious with user-generated content in rich text editors.

* **Secure API handling**
  * Store API keys and secrets on the server, never in client-side code.
  * Use environment variables for build-time configuration, but remember they are exposed in the client if prefixed with `REACT_APP_` (CRA) or `VITE_` (Vite). Sensitive data should not be placed there.
  * Implement proper authentication (JWT, OAuth) and include tokens in HTTP headers (Authorization).
  * Use HTTPS to encrypt data in transit.
  * Validate and sanitize all data coming from APIs before rendering.

* **Auth flows**
  * Common auth flows: JWT-based authentication, OAuth2, or session cookies.
  * In React, store tokens securely (httpOnly cookies are safer than localStorage, but require backend support).
  * Protect routes by checking auth state and redirecting to login.
  * Handle token expiration and refresh tokens gracefully.
  * Implement CSRF protection if using cookies.

* **Token storage**
  * Storing tokens in `localStorage` or `sessionStorage` makes them accessible to any JavaScript on the same origin, vulnerable to XSS.
  * For better security, store tokens in httpOnly cookies (set by server) which are not accessible to JavaScript, mitigating XSS risks.
  * However, httpOnly cookies are vulnerable to CSRF; use SameSite=strict and CSRF tokens.
  * If using localStorage, ensure strong Content Security Policy (CSP) to mitigate XSS.
  * Consider using libraries like `js-cookie` for cookie management.

### Architecture & Design Patterns
* **Component architecture patterns**
  * Common patterns: container/presentational, higher-order components, render props, hooks, and compound components.
  * Choosing a pattern depends on the problem: hooks for stateful logic, compound components for cohesive UI, etc.
  * A well-architected component library separates concerns, promotes reusability, and is easy to maintain.
  * Patterns like atomic design (atoms, molecules, organisms) help organize components.
  * Consider scalability and team conventions when selecting patterns.

* **Folder structure strategies**
  * Popular strategies: feature-based, file-type-based, or domain-driven.
  * Feature-based: group by feature (e.g., `features/auth`, `features/dashboard`) containing components, hooks, and tests.
  * File-type-based: separate folders for components, hooks, utils, etc. This can become messy in large apps.
  * Domain-driven: align with business domains (e.g., `products`, `orders`).
  * Also consider colocating tests, styles, and stories with components.
  * A good structure balances scalability and developer navigation.

* **Feature-based architecture**
  * Organize code by features/modules rather than technical layers.
  * Each feature contains its own components, hooks, API calls, and state management (e.g., Redux slice).
  * This promotes encapsulation and makes it easier to add/remove features.
  * Features can be lazy-loaded independently.
  * Common/shared code goes into a `shared` or `common` folder.

* **Micro-frontend architecture**
  * Micro-frontends extend microservices to the frontend, allowing independent teams to develop and deploy parts of a web app.
  * Integration approaches: iframes, Web Components, module federation (Webpack 5), or single-spa.
  * React can be used within micro-frontends, with each micro-frontend being a separate React app.
  * Challenges: shared dependencies, styling conflicts, communication between apps.
  * Module federation enables dynamic loading of remote modules at runtime, making it a popular choice.

* **Scalable React architecture**
  * A scalable architecture should handle growing codebase, team size, and performance needs.
  * Principles: modularity, clear separation of concerns, consistent patterns, and testing.
  * Use state management wisely; not everything needs to be global.
  * Implement code splitting and lazy loading to keep bundles manageable.
  * Enforce coding standards via linting and code reviews.

* **Separation of concerns**
  * Separate UI from business logic (e.g., using custom hooks).
  * Separate data fetching from components (e.g., using React Query or services).
  * Keep components pure and focused on presentation.
  * Use container components for state and side effects.
  * This makes code more testable, maintainable, and reusable.

### Real-World Scenarios
* **Large app state design**
  * Design state based on domain and usage patterns. Normalize data to avoid duplication.
  * Use a combination of local state, context, and global stores. Consider using atomic state for fine-grained reactivity.
  * Implement selectors to compute derived data and memoize where needed.
  * Plan for scalability by structuring state slices and using tools like Redux Toolkit.
  * Ensure state updates are predictable and debuggable.

* **Performance bottleneck debugging**
  * Identify bottlenecks using React DevTools Profiler, Chrome DevTools Performance tab.
  * Look for components that render too often, long render times, or large commits.
  * Check for unnecessary re-renders due to prop changes or context updates.
  * Use `why-did-you-render` library to track unnecessary renders.
  * Optimize by memoizing, splitting components, or using virtualization.

* **Handling slow APIs**
  * Show loading states immediately; use skeletons to improve perceived performance.
  * Implement caching with libraries like React Query to avoid repeated requests.
  * Use optimistic updates for a snappy feel.
  * Consider using pagination or infinite scrolling to limit data.
  * For slow endpoints, use background refetching and stale-while-revalidate.

* **Optimizing re-renders**
  * Use `React.memo` on components that receive the same props frequently.
  * Memoize objects and functions with `useMemo` and `useCallback` to stabilize references.
  * Lift state down so that only the minimal part of the tree re-renders.
  * Use keys to help React identify changes and avoid unnecessary remounts.
  * Avoid inline functions in render unless they are stable via `useCallback`.

* **Scaling React applications**
  * Adopt a consistent folder structure and coding standards.
  * Use feature-based architecture with lazy loading for better performance.
  * Implement a robust state management strategy that grows with the app.
  * Set up automated testing and CI/CD to ensure stability.
  * Monitor performance in production using tools like Sentry, LogRocket, or New Relic.

* **Migration strategies**
  * Migrating from older React versions: update dependencies incrementally, use codemods, and address deprecations.
  * Migrating from class components to functional hooks: start with simple components, use hooks gradually, and test thoroughly.
  * Migrating from legacy state management (e.g., old Redux) to Redux Toolkit: rewrite slices incrementally.
  * When migrating to Next.js from CRA, start with a single page and expand.
  * Always have a rollback plan and ensure feature parity.

* **Legacy React refactoring**
  * Refactor class components to functional components with hooks for better readability and performance.
  * Replace HOCs with custom hooks where possible.
  * Update deprecated lifecycle methods (e.g., `componentWillReceiveProps`).
  * Gradually adopt modern patterns like context for theming, Suspense for lazy loading.
  * Ensure thorough testing before and after refactoring.

* **SSR vs CSR decisions**
  * Choose SSR for SEO-critical, content-heavy sites (blogs, e-commerce).
  * Choose CSR for highly interactive apps behind authentication (dashboards, admin panels).
  * Consider hybrid approaches: static generation for marketing pages, CSR for authenticated areas.
  * Performance trade-offs: SSR improves initial load but increases server load; CSR may have slower initial load but cheaper hosting.
  * Use frameworks like Next.js that support both and allow per-page decisions.

* **Frontend system design**
  * When designing a frontend system, consider requirements: user flows, scale, performance, team structure.
  * Choose tech stack based on project needs (e.g., Next.js for SSR, Vite for SPA).
  * Design component hierarchy and state management early.
  * Plan for error handling, logging, monitoring, and analytics.
  * Consider internationalization, theming, and accessibility from the start.
