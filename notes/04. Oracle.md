## Oracle Fundamentals

*   **Oracle database architecture overview:** Understand the core components: background processes, memory structures (SGA, PGA), and physical storage (datafiles, control files, redo logs). The architecture ensures data integrity, performance, and recoverability, separating the physical (what's on disk) from the logical (tablespaces, segments).
*   **Oracle instance vs database:** An **instance** is the combination of memory (SGA) and background processes. A **database** is the physical set of files on disk (datafiles, control files, redo logs). An instance can mount and open only one database. You can have multiple instances (RAC) accessing one database, or an instance without a database (nomount state).
*   **Oracle background processes:** Key processes include **SMON** (System Monitor - instance recovery, coalesces free space), **PMON** (Process Monitor - cleans up failed processes), **DBWn** (Database Writer - writes dirty buffers from SGA to datafiles), **LGWR** (Log Writer - writes redo log buffers to online redo log files), and **CKPT** (Checkpoint - updates control/files with checkpoint information).
*   **Oracle memory structures:** The **System Global Area (SGA)** is a shared memory area allocated at instance startup, containing the database buffer cache, shared pool, redo log buffer, etc. The **Program Global Area (PGA)** is private memory for each server/background process, containing session-specific information like sort areas and private SQL areas.
*   **SGA and PGA:** **SGA** is shared by all server processes and is sized by parameters like `SGA_TARGET`. **PGA** is dedicated to a single process and managed via `PGA_AGGREGATE_TARGET`. The PGA is crucial for memory-intensive operations like sorting and hash joins, with work areas sized automatically.
*   **Oracle data files:** Physical files on disk that store actual table and index data. They are logically grouped into tablespaces. Key features: can auto-extend, can be in hot backup mode, and support features like Oracle Managed Files (OMF) for simplified management.
*   **Control files:** Small binary files tracking the physical structure of the database (database name, datafile/redo log locations, timestamps, and checkpoint/SCN information). Essential for database startup and recovery. Multiplexing (multiple copies) is critical for protection.
*   **Redo log files:** Record all changes made to the database data. Consist of **online redo logs** (written to cyclically by LGWR) and **archived redo logs** (filled online logs copied to a destination). They are the cornerstone of instance and media recovery.
*   **Oracle server architecture:** The user process (client) communicates with a server process. The server process receives SQL, parses/executes it, and returns results. This can be a **dedicated server** (one-to-one with user process) or a **shared server** (dispatchers route requests to a pool of shared server processes).
*   **Dedicated vs shared server:** **Dedicated** is simpler, recommended for long-running or batch jobs, but can consume significant memory with many concurrent connections. **Shared server** reduces memory overhead by sharing server processes, but can introduce performance overhead for certain operations and is less common now than connection pooling in the middle tier.
*   **Oracle editions:** Different feature sets for licensing. **Enterprise Edition (EE)** includes advanced features like partitioning, RAC, Data Guard, and advanced compression. **Standard Edition (SE/SE2)** has core features but lacks many enterprise add-ons. **Express Edition (XE)** is a free, entry-level, small-footprint database.
*   **Oracle installation basics:** Key steps include pre-install tasks (OS kernel parameters, users/groups, environment variables), running the Oracle Universal Installer (OUI), and executing configuration scripts as the root user (for setting permissions and starting listeners).
*   **Database startup and shutdown modes:** Startup progresses through **NOMOUNT** (instance started), **MOUNT** (control file opened), and **OPEN** (database accessible). Shutdown modes include **NORMAL** (waits for users to disconnect), **TRANSACTIONAL** (waits for transactions to complete), **IMMEDIATE** (rolls back active transactions), and **ABORT** (instant crash, requires recovery on next startup).
*   **Oracle client connectivity:** Uses Oracle Net Services. Clients use **TNS names** (aliases in `tnsnames.ora`) or Easy Connect strings (`host:port/service_name`) to specify connection details. The connection request is directed to a listener on the server.
*   **Listener architecture:** A separate process (`tnslsnr`) that listens on a specific port (default 1521) for incoming client connections. It receives the client request and, based on the service name, either hands it off to a dedicated server process or redirects the client to a dispatcher (shared server). It registers database services dynamically via **PMON** or statically via `listener.ora`.

## SQL Fundamentals

*   **SQL basics:** Structured Query Language for managing and manipulating relational databases. Core operations (CRUD) are performed via `SELECT`, `INSERT`, `UPDATE`, `DELETE`. The `SELECT` statement is the primary means of querying data and has a specific logical order of operations (`FROM`, `WHERE`, `GROUP BY`, `HAVING`, `SELECT`, `ORDER BY`).
*   **SELECT statements:** Used to query data. Key clauses include `DISTINCT` (unique rows), `WHERE` (row filtering), `GROUP BY` (row grouping), `HAVING` (group filtering), and `ORDER BY` (sorting). Expressions and functions can be used in the select list.
*   **Filtering with WHERE:** Filters rows before grouping. Uses operators like `=`, `>`, `<`, `IN`, `BETWEEN`, `LIKE` (with wildcards `%` and `_`), `IS NULL`, and logical operators `AND`/`OR`. The order of evaluation can be controlled with parentheses.
*   **Sorting data:** `ORDER BY` clause sorts the final result set. Default is ascending (`ASC`). Can sort by column name, alias, or position (`ORDER BY 3`). Can specify `NULLS FIRST` or `NULLS LAST` to control placement of null values.
*   **Grouping and aggregation:** `GROUP BY` collapses rows sharing common column values into a single row. Used with aggregate functions like `COUNT`, `SUM`, `AVG`, `MIN`, `MAX`. All columns in the `SELECT` list that aren't aggregates must be in the `GROUP BY` clause.
*   **HAVING clause:** Filters groups created by the `GROUP BY` clause, based on the result of aggregate functions. The `WHERE` clause filters rows *before* grouping, `HAVING` filters *after* grouping.
*   **Joins (inner, outer, self, cross):** Combine rows from two or more tables. **Inner join** returns rows where a match exists in both tables. **Outer join** (`LEFT`, `RIGHT`, `FULL`) returns all rows from one table and matched rows from the other, with nulls where no match exists. **Self join** joins a table to itself (using aliases). **Cross join** produces a Cartesian product.
*   **Subqueries:** A query nested inside another query. Can be used in `SELECT` list (scalar), `FROM` clause (inline view), or `WHERE`/`HAVING` clause. Can be single-row, multiple-row, or correlated.
*   **Correlated subqueries:** A subquery that references columns from the outer query. It is evaluated row-by-row for each row processed by the outer query. Can be powerful but potentially performance-intensive.
*   **EXISTS vs IN:** `EXISTS` checks for the existence of rows in a subquery and is often more efficient for correlated subqueries as it can stop processing upon finding the first match. `IN` compares a value to a list returned by a subquery. `NOT EXISTS` and `NOT IN` behave differently with nulls; `NOT IN` can return unexpected results if the subquery result contains nulls.
*   **Set operators:** Combine results of multiple queries. `UNION` (all distinct rows), `UNION ALL` (all rows, including duplicates), `INTERSECT` (common rows), `MINUS` (rows in first set but not second). All queries must have the same number and data types of columns.
*   **Views:** Stored queries that act like virtual tables. Can simplify complex queries, provide security by restricting access to underlying tables, and mask data complexity. **Updatable views** have restrictions (must not contain joins, set ops, etc.).
*   **Index basics:** Schema objects that provide fast access paths to data, avoiding full table scans. They are built on table columns and maintained automatically by Oracle. They improve `SELECT` performance but add overhead to `DML` operations.
*   **Constraints:** Rules enforced on table columns to ensure data integrity. Types: `NOT NULL`, `UNIQUE`, `PRIMARY KEY`, `FOREIGN KEY`, `CHECK`. Can be defined at column or table level. They can be deferred (`INITIALLY DEFERRED`) to be checked at transaction commit.
*   **NULL handling:** `NULL` represents the absence of a value. Arithmetic with `NULL` results in `NULL`. Comparisons with `NULL` using `=` or `<>` are unknown. Use `IS NULL` / `IS NOT NULL`. Functions like `NVL`, `COALESCE`, and `NULLIF` help handle nulls.
*   **DML operations:** `INSERT` (add rows), `UPDATE` (modify existing rows), `DELETE` (remove rows). `MERGE` (UPSERT) performs an insert or update based on a condition. DML changes are not permanent until a `COMMIT`.
*   **Transaction control:** `COMMIT` makes all DML changes in the current transaction permanent and visible to others. `ROLLBACK` undoes all changes since the last commit. `SAVEPOINT` allows partial rollback. Transactions ensure atomicity.
*   **ACID properties:** The foundation of reliable transactions. **Atomicity** (all or nothing), **Consistency** (transaction brings database from one valid state to another), **Isolation** (transactions do not interfere), **Durability** (committed changes persist after a failure).

## Advanced SQL

*   **Analytic functions:** Also known as window functions. Compute an aggregate value over a group of rows (window) but return a result for each row, unlike aggregates which collapse rows. Key functions: `SUM() OVER()`, `AVG() OVER()`, `LAG`, `LEAD`.
*   **Window functions:** The `OVER()` clause defines the window for the function. It can include `PARTITION BY` (divides result set into groups) and `ORDER BY` (defines order within the partition, crucial for running totals and ranking).
*   **Ranking functions:** A subset of analytic functions. `ROW_NUMBER()` (sequential number), `RANK()` (gaps in ranking for ties), `DENSE_RANK()` (no gaps for ties). Used for top-N per group queries and row deduplication.
*   **Partitioning queries:** Refers to using the `PARTITION BY` clause in analytic functions to reset the calculation for each group. For example, `SUM(salary) OVER (PARTITION BY department_id)` gives the total salary per department on every row.
*   **CTE (WITH clause):** Common Table Expression, also known as subquery factoring. Defines a temporary result set that can be referenced multiple times within the main query. Improves readability and can be used for recursion.
*   **Recursive queries:** A CTE that references itself, using an anchor member (base result) and a recursive member (joins back to the CTE). Used for traversing hierarchical or graph-based data (e.g., bill of materials, org charts).
*   **Hierarchical queries:** Oracle's proprietary `CONNECT BY` syntax. `START WITH` defines the root rows, `CONNECT BY PRIOR` defines the relationship (e.g., `CONNECT BY PRIOR employee_id = manager_id`). Uses pseudo-columns like `LEVEL` and functions like `SYS_CONNECT_BY_PATH`.
*   **CONNECT BY:** Part of hierarchical queries. `CONNECT BY NOCYCLE` prevents infinite loops caused by cycles in the data. The `PRIOR` operator signifies the direction of the hierarchy.
*   **MODEL clause:** A highly advanced feature for inter-row calculations, allowing you to treat rows like cells in a multidimensional array. Can perform complex spreadsheet-like calculations, forecasting, and what-if analysis within SQL.
*   **Pivot and unpivot:** `PIVOT` transforms rows into columns (e.g., turning distinct product categories into column headers showing sales). `UNPIVOT` transforms columns into rows. These are part of Oracle's SQL for data reshaping.
*   **Advanced joins:** Beyond simple joins, this includes **partitioned outer joins** (which densify data by filling in missing rows for specified partitions), and using the `LATERAL` clause (or `CROSS/OUTER APPLY`) to allow a right-hand side subquery to reference columns from left-hand side tables.
*   **Query optimization:** The process of making a SQL statement execute efficiently. This involves examining execution plans, understanding optimizer decisions, ensuring proper statistics, and rewriting inefficient SQL.
*   **Execution plans:** A step-by-step set of operations the database performs to execute a SQL statement. Generated by the optimizer. Analyzed using `EXPLAIN PLAN FOR` or `DBMS_XPLAN`. Key operations to look for: full table scans, index access paths, join methods (Nested Loops, Hash Join, Sort-Merge Join).
*   **Query rewriting:** Manually restructuring a SQL statement to be more efficient without changing its result. Examples: using `EXISTS` instead of `IN`, rewriting `OR` conditions as `UNION ALL`, or removing unnecessary outer joins.

## PL/SQL Basics

*   **PL/SQL architecture:** PL/SQL is Oracle's procedural language extension to SQL. The **PL/SQL engine** processes procedural statements and passes SQL statements to the **SQL engine**. This tight integration allows for efficient data manipulation and business logic execution within the database.
*   **Blocks and structure:** The fundamental unit of PL/SQL is a block. It consists of: **Declarative** section (optional, for variables, cursors), **Execution** section (mandatory, contains the main logic), and **Exception** section (optional, for error handling).
*   **Variables and datatypes:** PL/SQL supports SQL datatypes (`NUMBER`, `VARCHAR2`, `DATE`) plus PL/SQL-specific types (`BOOLEAN`, `BINARY_INTEGER`, `PLS_INTEGER`). Variables are declared in the declarative section and can be assigned values using `:=`.
*   **Control statements:** `IF-THEN-ELSE` and `CASE` statements for conditional branching. These allow the program to make decisions based on data values or program state.
*   **Loops:** Three looping constructs: **Basic `LOOP`** (must contain an `EXIT`), **`WHILE` loop** (condition checked at start), and **`FOR` loop** (iterates over a range of integers or over the results of a cursor). `EXIT` and `CONTINUE` control loop execution.
*   **Exception handling:** The `EXCEPTION` section allows you to trap and handle errors gracefully. When an error occurs, control transfers to the exception section, preventing the program from failing abruptly.
*   **Cursors:** A handle or pointer to a private SQL area that stores information about processing a specific SQL statement. Used to retrieve and manipulate multiple rows returned by a query, one row at a time.
*   **Explicit vs implicit cursors:** **Implicit cursors** are automatically created by Oracle for single-row `SELECT INTO` or DML statements; their attributes (`SQL%FOUND`, `SQL%ROWCOUNT`) are accessible. **Explicit cursors** are declared and controlled by the programmer for queries returning multiple rows, giving fine-grained control over the fetch process.
*   **Parameterized cursors:** Explicit cursors defined to accept input parameters. This allows the same cursor to be used to fetch different sets of data based on the parameter values passed when the cursor is opened.
*   **Stored procedures:** Named PL/SQL blocks stored in the database. They perform a specific action, may accept parameters, and do not return a value directly. Called via `EXEC` or from other PL/SQL blocks.
*   **Functions:** Named PL/SQL blocks stored in the database. They must return a single value (of any datatype) using the `RETURN` statement. Can be used in SQL statements if they obey purity rules.
*   **Packages:** A schema object that groups logically related types, variables, cursors, subprograms (procedures and functions) into a single, named unit. Consists of a **specification** (public interface) and a **body** (implementation).
*   **Triggers:** Stored PL/SQL blocks that are automatically executed (`fired`) in response to a specific event on a particular table or view, such as an `INSERT`, `UPDATE`, or `DELETE`.
*   **Anonymous blocks:** Unnamed PL/SQL blocks that are not stored in the database. They are compiled and executed on the fly, commonly used in scripts, applications, or ad-hoc tools like SQL*Plus.

## Advanced PL/SQL

*   **Bulk processing:** A technique to improve performance by transferring an entire collection of data between the PL/SQL and SQL engines in a single operation, drastically reducing context switches.
*   **Collections:** PL/SQL provides three collection types: **Associative arrays** (formerly index-by tables), **Nested tables**, and **Variable-size arrays (VARRAYs)**. They are key to implementing array-like processing in PL/SQL.
*   **Associative arrays:** A set of key-value pairs. They are sparse (can have non-consecutive indexes). They are temporary and only exist in PL/SQL for the duration of a block or session unless used as a package variable. Index can be integer or string.
*   **Nested tables:** A table that is a column in another table. As a PL/SQL collection type, they are dense initially but can become sparse after deletions. Can be stored in database columns. Are unbounded.
*   **VARRAYs:** Variable-size arrays with a fixed maximum size (bounded). They are dense (cannot have gaps). Always stored as a single object when used as a column type. Best for small, ordered, static lists.
*   **Bulk collect:** The `BULK COLLECT INTO` clause fetches multiple rows from a query or cursor and loads them into a collection in a single operation. Massively reduces context switches compared to row-by-row `FETCH`.
*   **FORALL:** A `FORALL` statement issues a series of DML statements (all using the same SQL template) from the contents of one or more collections. It is not a loop; it is a bulk bind operation that sends all DMLs to the SQL engine at once.
*   **Dynamic SQL:** Building and executing SQL statements at runtime. Necessary for DDL, where the full text of the SQL is not known at compile time, or for queries with a variable number of `WHERE` clause predicates.
*   **Native dynamic SQL:** Using the `EXECUTE IMMEDIATE` statement. Simpler and faster than `DBMS_SQL` for most operations. Supports both DML and DDL. Must handle binding variables carefully to avoid SQL injection.
*   **Ref cursors:** A pointer to a cursor (result set). A **REF CURSOR** is a PL/SQL data type whose value is the address of a cursor. Allows result sets to be passed between programs (e.g., from a stored procedure to a client application like Java).
*   **Strong vs weak REF cursors:** A **strong** `REF CURSOR` is defined with a `RETURN` type (`RETURN employees%ROWTYPE`), making it less error-prone but less flexible. A **weak** `REF CURSOR` is defined without a return type (`SYS_REFCURSOR`) and can point to any query result set.
*   **Autonomous transactions:** An independent transaction started from within another transaction. `PRAGMA AUTONOMOUS_TRANSACTION` is used. Changes in the autonomous transaction are committed or rolled back independently of the main transaction. Useful for logging errors regardless of the main transaction's outcome.
*   **PL/SQL performance tuning:** Focuses on reducing context switches (using bulk operations), optimizing logic (avoiding unnecessary calculations), ensuring efficient SQL within PL/SQL, and using appropriate collection types.
*   **Error handling strategies:** Robust strategies include using named exceptions, logging errors to a table using autonomous transactions, handling specific exceptions before `WHEN OTHERS`, and re-raising exceptions after logging.
*   **Dependency management:** Oracle tracks dependencies between schema objects. If a table a view depends on is altered, the view becomes invalid and needs recompilation. Package bodies can be altered without invalidating the specification, protecting dependent code.

## Oracle Data Structures

*   **Tablespaces:** Logical storage units within a database that group related physical files (datafiles). They hold schema objects like tables and indexes. Types include `PERMANENT` (user data), `TEMPORARY` (sort/hash areas), and `UNDO` (for read consistency and rollback).
*   **Data blocks:** The smallest unit of I/O and storage in an Oracle database. A block contains a header (block type, transaction info), table directory, row directory, and the row data itself. Standard block size is 8KB but can be configured.
*   **Segments:** A set of extents allocated for a specific logical object within a tablespace. For example, a table has a data segment, an index has an index segment, and a LOB column has its own LOB segment.
*   **Extents:** A contiguous set of data blocks allocated for a segment. When a segment needs more space, it allocates another extent. Extents are managed locally in the tablespace (LMT - Locally Managed Tablespace).
*   **Schema objects:** Objects that directly reference or contain data within a schema (a collection of objects owned by a user). Examples: tables, indexes, views, synonyms, sequences, stored procedures, packages.
*   **Partitioned tables:** Large tables broken into smaller, more manageable pieces called partitions, based on a key column (range, list, hash). Enables partition pruning (querying only relevant partitions), improves parallel execution, and eases data lifecycle management.
*   **Index types:** Various structures to speed up data access. The choice depends on data distribution and query patterns.
*   **B-tree indexes:** The default and most common index type. Balanced tree structure, ideal for high-cardinality data (many distinct values). Good for equality and range queries. Leaf blocks contain the indexed values and ROWIDs.
*   **Bitmap indexes:** Stores a bitmap for each distinct key value, where each bit represents a row. Ideal for low-cardinality data (e.g., gender, status) and data warehousing queries with complex `AND`/`OR` conditions. Not suitable for heavy DML.
*   **Function-based indexes:** An index created on an expression or function (e.g., `UPPER(last_name)`). Allows the optimizer to use the index for queries that use that same expression, avoiding a full table scan.
*   **Global vs local indexes:** Relevant for partitioned tables. A **local index** is partitioned identically to the underlying table; each table partition has its own index partition. A **global index** is a single index spanning all table partitions, which can be partitioned differently or not at all.
*   **Materialized views:** A database object that stores the results of a query physically, like a table. Used for summary management and query rewrite in data warehouses to pre-join and pre-aggregate data for faster query performance.
*   **Clusters:** A method of storing tables that are frequently joined together physically stored together. The cluster key is the common column. If one block contains data from both tables for a given key, join I/O is dramatically reduced.

## Oracle Transactions & Concurrency

*   **Transaction management:** A transaction is a logical unit of work comprising one or more SQL statements. Oracle manages transactions implicitly; the first DML starts a transaction, and it ends with `COMMIT` or `ROLLBACK`.
*   **Read consistency:** Oracle guarantees a consistent view of the data as of the point a query (or a statement within a transaction) begins. It achieves this using **undo data**, which provides a before-image of changed data.
*   **Undo segments:** Special segments in an undo tablespace that store old versions of data blocks. They are used for read consistency, rolling back transactions, and enabling flashback features. Managed automatically in **Automatic Undo Management (AUM)** mode.
*   **Redo logs:** The redo log records every change made to the database. It's used for recovery. When a `COMMIT` occurs, LGWR flushes the redo log buffer to the online redo log file before the DBWn writes the changed blocks to the datafile. This is **write-ahead logging**.
*   **Locking mechanisms:** Oracle uses locks to manage concurrent access to data, ensuring data integrity. The two main types are **TX locks** (row-level transaction locks, automatically acquired for DML) and **TM locks** (table locks, to prevent DDL that would conflict with a transaction).
*   **Isolation levels:** Defined by the ANSI SQL standard. Oracle supports `READ COMMITTED` (default) and `SERIALIZABLE`. `READ COMMITTED` provides statement-level read consistency. `SERIALIZABLE` provides transaction-level read consistency (as of the transaction's start).
*   **Deadlacks:** Occur when two or more sessions are waiting for a resource held by the other, resulting in a circular wait. Oracle automatically detects deadlocks, resolves them by rolling back the statement of one of the sessions (not the entire transaction), and returns an error.
*   **Savepoints:** A marker within a transaction that allows for partial rollback. `ROLLBACK TO SAVEPOINT` undoes all work performed after that savepoint but keeps the rest of the transaction intact, allowing other statements to proceed.
*   **Flashback features:** A set of features leveraging undo data to view or recover past states of data. Includes `FLASHBACK QUERY`, `FLASHBACK TABLE`, `FLASHBACK DROP` (recycle bin), and `FLASHBACK DATABASE` (requires flashback logs).

## Oracle Performance Tuning

*   **SQL tuning methodology:** A systematic approach: 1) Identify high-load SQL (from AWR, ASH). 2) Gather its execution plan and explain. 3) Gather object statistics. 4) Restructure the SQL or indexes. 5) Verify the improvement.
*   **Explain plan analysis:** Use `DBMS_XPLAN.DISPLAY`, `DISPLAY_CURSOR`, or `DISPLAY_AWR` to view plans. Look for costly operations (TABLE ACCESS FULL on large tables), inefficient join orders, and access predicate vs filter predicate. Understand plan operation costs.
*   **Cost-based optimizer (CBO):** The CBO's goal is to generate the most efficient execution plan for a SQL statement based on object statistics, system statistics, and initialization parameters. It calculates a "cost" for each potential plan and chooses the lowest.
*   **Statistics gathering:** The CBO relies on accurate object and system statistics. Use `DBMS_STATS` package (e.g., `GATHER_TABLE_STATS`). For partitioned tables, incremental statistics (`INCREMENTAL`) can be used to avoid re-scanning entire table. Missing or stale stats are the #1 cause of bad plans.
*   **Index tuning:** Selecting the right index type, ensuring leading index columns match `WHERE` clause predicates, avoiding functions on indexed columns in predicates (unless using function-based index), and monitoring index usage (`ALTER INDEX MONITORING USAGE`).
*   **Execution plan optimization:** The optimizer can choose various join methods (Nested Loop for small driven set, Hash Join for large unindexed sets, Sort-Merge for large sorted sets), access paths (Index Scan vs Full Scan), and join orders.
*   **Wait events:** Metrics that indicate a session is waiting for a resource. Key categories: I/O (db file sequential/scattered read), Concurrency (buffer busy waits), and Idle events. Analyzed via `V$SESSION`, `V$SESSION_WAIT`, and ASH reports.
*   **AWR reports:** Automatic Workload Repository reports provide a snapshot of database performance over a time period. Includes top SQL by various metrics (elapsed time, CPU, I/O), instance efficiency percentages, and segment statistics.
*   **ASH reports:** Active Session History reports provide a real-time, sampled view of active session activity, great for drilling into specific short-term performance spikes. Shows what sessions were waiting for and the SQL they were running.
*   **ADDM:** Automatic Database Diagnostic Monitor uses AWR data to automatically identify performance bottlenecks and provide recommendations (e.g., "increase buffer cache", "run SQL tuning advisor on SQL ID x").
*   **Memory tuning:** Tuning SGA components (buffer cache, shared pool, large pool) and PGA. Using Automatic Memory Management (`MEMORY_TARGET`) simplifies this but may not be optimal for all workloads. Monitor buffer cache hit ratio and PGA memory advisory.
*   **I/O tuning:** Identifying and mitigating I/O bottlenecks. Strategies: distribute datafiles across different physical disks (stripe), separate redo logs and datafiles, use faster storage, tune SQL to reduce I/O, use table compression.
*   **Parallel execution:** Decomposing a SQL operation into multiple parallel processes to leverage multiple CPUs/I/O paths. Useful for full table scans of large tables, large sorts, and DML operations on partitioned tables. Tuned via `PARALLEL` hint or degree of parallelism at object level.
*   **Query hints:** Instructions embedded in SQL comments (`/*+ */`) to force the optimizer's behavior. Used cautiously to override the CBO when it makes a suboptimal choice (e.g., `/*+ INDEX(table_name index_name) */` or `/*+ LEADING(e d) USE_HASH(d) */`). Hints can be ignored if they lead to an invalid plan.
*   **Caching strategies:** Using the buffer cache effectively. For frequently accessed small tables, consider caching them (`ALTER TABLE ... CACHE`). For result sets of frequently executed functions/queries, consider the `RESULT_CACHE` hint or clause.

## Oracle Backup & Recovery

*   **RMAN architecture:** Recovery Manager is Oracle's client for backup, restore, and recovery. It connects to a target database and stores backup metadata in the control file of the target or in a separate **recovery catalog database**.
*   **Backup strategies:** Key decisions: **Full vs Incremental**, **Backup to disk vs tape**, **Frequency of backups**. A common strategy is a weekly Level 0 (full) backup plus daily Level 1 (incremental) backups, using a recovery catalog for tracking.
*   **Full vs incremental backup:** A **full backup** in RMAN is not a full database backup per se, but a backup of used blocks. An **incremental backup** backs up blocks changed since a previous backup. Level 1 can be **differential** (all changes since last level 0) or **cumulative** (all changes since last level 0, preferred for simpler recovery).
*   **Recovery scenarios:** Types: **Instance recovery** (automatic after instance failure, using online redo logs). **Media recovery** (restoring lost/corrupted datafiles from backup and applying archived/online redo logs to roll forward). Managed by RMAN.
*   **Media recovery:** The process of restoring a datafile backup and then applying redo logs to bring it to the current state. Can be complete (up to current time) or incomplete/PITR (up to a specific SCN or time).
*   **Flashback database:** An alternative to media recovery for fast database rewinding. It uses **flashback logs** (stored in the Fast Recovery Area) to revert the entire database to a past point in time, much faster than a traditional restore/recovery.
*   **Point-in-time recovery (PITR):** Recovering the database (or a tablespace/table) to a specific target time, SCN, or log sequence number, not the current time. Useful for undoing user errors like an accidentally dropped table.
*   **Archive logs:** Filled online redo log files that have been copied to one or more destinations (the Fast Recovery Area or a custom archive location) by the ARCH process. They are critical for media recovery.
*   **Data guard basics:** Oracle's solution for disaster recovery and high availability. Maintains one or more synchronized standby databases (physical, logical, or snapshot). Supports failover (for disasters), switchover (for planned maintenance), and active data guard for read-only access.

## Oracle Security

*   **User management:** Creating database users (schemas). Authenticating users via password (local), OS, network, or external services. Passwords are managed via profiles (expiry, complexity, failed login attempts).
*   **Roles and privileges:** **Privileges** are rights to perform actions: system-level (`CREATE SESSION`, `CREATE TABLE`) and object-level (`SELECT ON hr.employees`). **Roles** are named groups of privileges that can be granted to users, simplifying administration.
*   **Profiles:** A set of resource limits and password parameters applied to a user. They control things like CPU/session time, idle time, password lifetime, and failed login attempts, helping enforce security and resource usage policies.
*   **Auditing:** Monitoring and recording specific database actions. Can audit successful/failed logins, access to specific objects, or execution of system privileges. **Unified Auditing** (introduced in 12c) consolidates all audit trails into a single format.
*   **Encryption:** Protecting data at rest and in transit. **Transparent Data Encryption (TDE)** encrypts data written to disk (datafiles, redo logs, backups) automatically without application changes. **Network encryption** secures data as it travels over the network via Oracle Net Services.
*   **Data masking:** Replacing sensitive data (like credit card numbers or SSNs) with realistic but fictitious data. Used for non-production environments (development, testing) to protect privacy. Oracle provides Data Masking and Subsetting capabilities.
*   **Secure configuration:** Following best practices: least privilege (grant only necessary privileges), securing the listener (setting a password, restricting admin commands), patching regularly, and removing default accounts.
*   **Access control:** Controlling which users can access which objects. `GRANT` and `REVOKE` statements. Row-level security can be enforced via **Virtual Private Database (VPD)** , which automatically appends a dynamic `WHERE` clause to all queries based on a user's attributes.

## Oracle Administration

*   **Database creation:** Can be done via DBCA (Database Configuration Assistant - GUI) or manually with `CREATE DATABASE` script. Manual process includes setting environment variables, creating init.ora/spfile, starting instance nomount, and executing the create database command.
*   **User management:** `CREATE USER`, `ALTER USER`, `DROP USER`. Granting quotas on tablespaces, assigning default and temporary tablespaces, and managing profiles and roles.
*   **Storage management:** Understanding and managing ASM (Automatic Storage Management), file systems, and raw devices for Oracle datafiles. Monitoring free space, resizing datafiles, and managing the Fast Recovery Area.
*   **Tablespace management:** Creating permanent, temporary, and undo tablespaces. Adding, resizing, renaming, and relocating datafiles within a tablespace. Setting tablespace to `READ ONLY` or `OFFLINE`.
*   **Datafile management:** `ALTER DATABASE` commands to add, rename, resize, and bring online/take offline datafiles. Monitoring space usage within datafiles via `DBA_DATA_FILES` and `DBA_FREE_SPACE`.
*   **Listener configuration:** Managing the `listener.ora` and `sqlnet.ora` files. Using Listener Control utility (`lsnrctl`) to start, stop, and check the status of listeners. Dynamic service registration with the listener via PMON.
*   **Patch management:** Applying Release Updates (RUs) and Release Update Revisions (RURs) using `opatch` (OPatch utility). Planning patch windows, ensuring rollback strategies are in place, and following the Oracle Recommended Patch (Database Proactive Bundle Patches).
*   **Upgrades and migrations:** Methods include manual upgrade (running scripts), using the Database Upgrade Assistant (DBUA), and "cutover" methods like export/import, Data Pump, or transportable tablespaces for migrations across platforms/versions.
*   **Monitoring database health:** Proactive monitoring using Enterprise Manager Cloud Control, `V$` views (`V$SESSION`, `V$SYSMETRIC`), and custom scripts. Key metrics: alert log monitoring, tablespace usage, wait events, and long-running operations.
*   **Resource management:** Using Database Resource Manager (DBRM) to control the allocation of CPU, parallel execution, and I/O among different user sessions and consumer groups, ensuring predictable performance.

## Oracle High Availability

*   **Data Guard:** A comprehensive HA/DR solution. A primary database can have up to 30 synchronized standby databases (Physical, Logical, or Snapshot). Supports **Maximum Protection**, **Maximum Availability**, and **Maximum Performance** data protection modes.
*   **RAC architecture:** Real Application Clusters allows multiple instances running on different servers (in a cluster) to access a single shared database. Provides high availability (node failure) and scalability (adding nodes). Uses Cache Fusion to transfer blocks directly between instances via high-speed interconnect.
*   **Clusterware basics:** Oracle Clusterware is the required cluster management software for RAC. It manages the cluster membership, node fencing, and resources (voting disk, OCR). It provides the infrastructure for RAC to function.
*   **Failover mechanisms:** In RAC, if a node fails, the clusterware ensures other nodes remain operational and connections can be failed over (TAF - Transparent Application Failover) to surviving instances. In Data Guard, a role transition (Switchover/Failover) moves the primary role to a standby.
*   **Load balancing:** Distributing workload. In RAC, connections can be load-balanced across instances by the listener. In Data Guard, read-only workloads can be offloaded to physical standby databases (Active Data Guard).
*   **Replication strategies:** Beyond HA, for data distribution or offloading reporting, strategies include GoldenGate (log-based, real-time, heterogeneous), Advanced Replication (deprecated), and Data Guard (for database-level HA/DR).

## Advanced Oracle Architecture

*   **Multitenant architecture:** Introduced in 12c. A single **container database (CDB)** can host multiple **pluggable databases (PDBs)** . This makes a PDB look and feel like a non-CDB to applications, while allowing centralized management of the CDB.
*   **Container databases (CDB):** The root container that holds system metadata and common users. It has control over all PDBs. Contains the `SYSTEM`, `SYSAUX`, `TEMP`, and `UNDO` (if local undo is enabled) tablespaces at the CDB level.
*   **Pluggable databases (PDB):** A portable collection of schemas, objects, and data that appears as a separate database to an application. PDBs can be quickly plugged/unplugged/cloned, enabling rapid provisioning and consolidation.
*   **Oracle internals:** Deep dive into how the database works, including algorithms for cache management (LRU), concurrency control (latching and enqueues), and transaction management (SCN propagation).
*   **Latch and enqueue mechanisms:** **Latches** are low-level, lightweight serialization mechanisms to protect shared data structures in the SGA (e.g., cache buffers chains). They are held for a very short time. **Enqueues** (locks) are more sophisticated locks for database objects (tables, rows) and coordinate concurrent access with queuing and deadlock detection.
*   **Buffer cache management:** Managed as a set of chains of buffers. Accessed via hash buckets (protected by latches). Managed with an LRU (Least Recently Used) list to keep frequently accessed blocks in memory. Touch count algorithm determines block age.
*   **Shared pool tuning:** Contains the library cache (parsed SQL, execution plans) and data dictionary cache. Tuning involves sizing it correctly (`SHARED_POOL_SIZE`), avoiding SQL that is not shared (no literals, so use bind variables), and monitoring library cache hit ratio.
*   **Cursor management:** Private SQL areas (in PGA) hold run-time information (bind values, row counts). Shared SQL areas (in library cache) hold parse information and execution plan. High version counts (multiple child cursors for the same SQL) can indicate problems like environment mismatches.

## Oracle Networking

*   **Oracle Net architecture:** Oracle's proprietary networking stack enabling communication between client applications and the database server. It handles session establishment, data transfer, and security.
*   **Listener configuration:** The `listener.ora` file defines the listener's name, listening protocol (TCP/IP), ports, and static service registrations (SID_LIST). The listener can also be configured for external procedures and heterogeneous services.
*   **TNS names:** The `tnsnames.ora` file on the client defines connection aliases. Each alias maps to a connect descriptor (`PROTOCOL`, `HOST`, `PORT`, `SERVICE_NAME`/`SID`). Simplifies client connection strings.
*   **Network troubleshooting:** Tools and techniques: `tnsping <alias>` (checks connectivity and name resolution to the listener), `lsnrctl services` (shows services registered with the listener), checking `sqlnet.ora` for tracing/logging, and examining listener and client sqlnet logs.
*   **Secure network communication:** Configuring `sqlnet.ora` for encryption (`SQLNET.ENCRYPTION_SERVER/CLIENT`) and integrity checksums. Using **TCPS** (TCP with SSL) for certificate-based authentication and encryption. Firewalls and connection managers (CMAN) for access control.

## Oracle Integration

*   **Oracle with applications:** How applications (Java, .NET, Python, etc.) connect to Oracle. Using appropriate drivers (JDBC, ODP.NET, cx_Oracle) and connection pools (HikariCP, UCP) for performance.
*   **JDBC connectivity:** Oracle provides several JDBC drivers: **Thin driver** (pure Java, platform-independent, for client-server), **OCI driver** (uses Oracle client libraries, faster, platform-specific), and **Server-side internal driver** (for Java inside the database).
*   **OCI drivers:** Oracle Call Interface. A low-level, C-language API for building applications. The OCI driver for JDBC is thicker and leverages the full Oracle client installation, offering features like advanced connection pooling and access to all Oracle features.
*   **External tables:** A feature that allows you to query data residing in flat files outside the database as if it were a table. Useful for loading data without an ETL tool, using SQL directly. Defined with an `ORACLE_LOADER` or `ORACLE_DATAPUMP` access driver.
*   **Data import/export:** Traditional `exp`/`imp` utilities (original) and the modern, faster `expdp`/`impdp` (Data Pump) utilities. Data Pump operates on the server-side, provides parallel execution, restartable jobs, and fine-grained object selection.
*   **SQL*Loader:** A utility for loading data from external files into database tables. Supports various formats, multiple tables per load, and complex data transformation. Can be controlled via a control file (.ctl).
*   **Data pump:** The preferred high-speed utility for logical backups and data movement. `expdp` unloads data and metadata into a set of dump files. `impdp` loads them. Supports network imports (directly from a source database) and remapping tablespaces/schemas.

## Advanced Features

*   **Partitioning strategies:** Advanced techniques: **Interval partitioning** (automatic creation of range partitions), **Reference partitioning** (child table inherits parent's partition key), **Virtual column-based partitioning**, and **Composite partitioning** (e.g., Range-List, List-Hash).
*   **In-memory database:** Oracle Database In-Memory (DBIM) option stores tables and partitions in a highly optimized columnar format in the SGA (In-Memory Column Store). Accelerates analytic queries by orders of magnitude without affecting OLTP workloads. Queries are routed to the appropriate format automatically.
*   **Advanced compression:** Includes **Advanced Row Compression** (for OLTP, reduces storage), **Hybrid Columnar Compression (HCC)** (for Exadata and certain storage, high compression for warehousing), and **SecureFiles** deduplication and compression for LOBs.
*   **Sharding:** A scalability and availability feature that horizontally partitions a database across many independent Oracle databases (shards) that share no hardware/software. Each shard is an independent Oracle database. Provides near-linear scalability and fault isolation.
*   **Exadata concepts:** Oracle's engineered system. Key features: **Smart Scan** (offloads query processing to storage cells), **Hybrid Columnar Compression (HCC)** , **InfiniBand** networking, and **storage indexes** (quickly prunes I/O on storage servers).
*   **Autonomous database:** A cloud-based database that automates patching, tuning, backups, and scaling. Powered by machine learning. Offers two workload types: Autonomous Data Warehouse (ADW) and Autonomous Transaction Processing (ATP).
*   **Cloud integration:** Options include running Oracle Database on OCI (Oracle Cloud Infrastructure), using Oracle's cloud services (Exadata Cloud Service, Autonomous Database), and hybrid solutions like Data Guard between on-prem and cloud.

## PL/SQL Fundamentals

*   **PL/SQL architecture:** PL/SQL is a procedural language tightly integrated with SQL. The PL/SQL engine processes procedural blocks; it parses, compiles, and executes them, handing off embedded SQL statements to the SQL engine for execution. This integration offers performance benefits by reducing network traffic.
*   **PL/SQL block structure:** The basic unit of PL/SQL is a block. The **DECLARE** section (optional) is for variables, constants, cursors. The **BEGIN** section (mandatory) contains executable statements. The **EXCEPTION** section (optional) handles errors. The block ends with `END;`.
*   **Anonymous blocks:** Unnamed PL/SQL blocks. They are compiled each time they are run and are not stored in the database. Used for scripts, ad-hoc tasks, and wrapping DDL in PL/SQL.
*   **Named blocks:** Include procedures, functions, packages, and triggers. These are compiled and stored in the database, promoting reusability, modularity, and better performance as they are pre-validated.
*   **Declaration section:** The part of a block where variables, constants, cursors, and user-defined exceptions are declared. Declarations are local to that block.
*   **Execution section:** Contains the core procedural logic and SQL statements. It must have at least one statement; `NULL;` can be used as a placeholder.
*   **Exception section:** Handles runtime errors. When an error occurs in the execution section, control passes to the exception section. After handling, the block terminates; control does not return to the execution section.
*   **Variables and constants:** Variables hold temporary data. Constants (`constant` keyword) hold a fixed value that cannot be changed. Declared with a name, datatype, and optionally an initial value (`variable_name datatype [not null] [:= default_value];`).
*   **PL/SQL data types:** Includes all SQL datatypes (`NUMBER`, `VARCHAR2`, `DATE`) plus PL/SQL-specific types (`BOOLEAN`, `PLS_INTEGER`, `BINARY_INTEGER`, `REF CURSOR`, and composite types).
*   **%TYPE and %ROWTYPE:** `%TYPE` declares a variable of the same datatype as a specific table column (e.g., `v_emp_id employees.employee_id%TYPE;`). `%ROWTYPE` declares a record variable that can hold a whole row from a table or cursor (e.g., `v_emp_rec employees%ROWTYPE;`). Increases code maintainability.
*   **Scope and visibility:** A variable's scope is the block in which it's declared. It is visible to all nested sub-blocks unless a variable with the same name is declared in the nested block, which then hides the outer variable (scope within the nested block is the local variable).
*   **Bind variables:** Variables declared in the host environment (like SQL*Plus or Java) and referenced in PL/SQL using a colon prefix (`:bind_var`). They allow values to be passed between the host environment and PL/SQL. Crucial for security and performance in dynamic SQL.
*   **Comments and documentation:** Single-line comments start with `--`. Multi-line comments are enclosed in `/* ... */`. Essential for code readability and maintenance.

## Control Structures

*   **Conditional statements:** `IF-THEN-END IF` (simple condition), `IF-THEN-ELSE-END IF` (two-way branch), and `IF-THEN-ELSIF-ELSE-END IF` (multiple conditions). The conditions are evaluated sequentially.
*   **IF / ELSIF / ELSE:** `ELSIF` (note spelling) allows checking multiple mutually exclusive conditions. Once a condition is true, the rest are skipped. An optional `ELSE` clause handles cases where no condition is true.
*   **CASE statements:** A more readable alternative to complex `IF-ELSIF` constructs. Two forms: **simple CASE** (evaluates a single expression) and **searched CASE** (evaluates multiple Boolean conditions). `CASE` must have an `ELSE` clause (or a `WHEN` must match) to avoid a `CASE_NOT_FOUND` exception.
*   **LOOP:** A basic loop that executes repeatedly. It must contain an `EXIT` or `EXIT WHEN` statement to terminate; otherwise, it's an infinite loop.
*   **WHILE loop:** Evaluates a condition at the start of each iteration. The loop body is executed only if the condition is true. Useful when the number of iterations is not known beforehand.
*   **FOR loop:** Iterates over a fixed integer range (`FOR i IN 1..10 LOOP`) or implicitly over the results of a cursor (`FOR rec IN (SELECT * FROM employees) LOOP`). The loop index or record is automatically declared and is local to the loop.
*   **Nested loops:** Placing one loop inside another. Must use different loop index variable names. Can be used for processing two-dimensional data structures or performing complex lookups.
*   **EXIT and CONTINUE:** `EXIT` terminates the loop entirely. `EXIT WHEN condition` provides a conditional exit. `CONTINUE` (available in Oracle 11g+) skips the rest of the current loop iteration and proceeds to the next. `CONTINUE WHEN condition` makes it conditional.
*   **GOTO statement:** An unconditional branch to a labeled block or statement. Generally considered bad practice as it leads to spaghetti code and makes logic hard to follow and debug; its use is highly discouraged in structured programming.

## SQL in PL/SQL

*   **Static SQL:** SQL statements whose full text is known at compilation time and embedded directly in PL/SQL code. They are parsed and optimized once and reused, offering performance and safety benefits.
*   **DML in PL/SQL:** `INSERT`, `UPDATE`, `DELETE`, and `MERGE` statements can be used directly within PL/SQL blocks. They function just as they would in standalone SQL, affecting tables and firing triggers.
*   **SELECT INTO:** Used to fetch a single row from the database into PL/SQL variables or a record. It must return exactly one row; otherwise, it raises `NO_DATA_FOUND` (zero rows) or `TOO_MANY_ROWS` (more than one row).
*   **DML returning clause:** After an `INSERT`, `UPDATE`, or `DELETE`, the `RETURNING ... INTO ...` clause can retrieve values from the affected rows (e.g., values of columns populated by triggers or sequences). Useful for capturing generated keys or old values.
*   **Implicit cursor attributes:** Attributes (`SQL%FOUND`, `SQL%NOTFOUND`, `SQL%ROWCOUNT`, `SQL%ISOPEN`) that provide information about the most recently executed implicit cursor (any DML or single-row `SELECT`). `SQL%ISOPEN` is always `FALSE` for implicit cursors.
*   **SQL cursor interaction:** PL/SQL seamlessly passes SQL statements to the SQL engine and receives results back, populating variables or collections. This is the core of its data manipulation power.
*   **Transaction control in PL/SQL:** `COMMIT`, `ROLLBACK`, and `SAVEPOINT` statements can be issued within PL/SQL. They work exactly as they do in SQL, affecting the current transaction. Their placement is critical for maintaining data integrity.
*   **COMMIT and ROLLBACK:** `COMMIT` makes all changes permanent and visible. `ROLLBACK` undoes all changes since the last `COMMIT` or `SAVEPOINT`. The `WORK` keyword (e.g., `COMMIT WORK`) is optional.
*   **Savepoints:** Named points in a transaction. `ROLLBACK TO savepoint_name` undoes work after the savepoint but retains work before it. The transaction remains active, allowing further statements or a final `COMMIT`.

## Cursors

*   **Cursor fundamentals:** A cursor is a handle to a private SQL area that stores information about processing a specific SQL statement. For queries, it points to the result set and tracks the current row.
*   **Implicit cursors:** Automatically created by Oracle for all DML and single-row `SELECT` statements. Managed internally. Their attributes (`SQL%...`) are available after the statement executes.
*   **Explicit cursors:** Defined and controlled by the programmer for queries that return multiple rows. The programmer declares the cursor, opens it, fetches rows one by one or in bulk, and closes it.
*   **Cursor lifecycle:** 1. **Declare:** Define the cursor with its associated query. 2. **Open:** Memory is allocated, the query is parsed and executed, and the result set is identified. 3. **Fetch:** Retrieve the current row into variables. 4. **Close:** Release the memory and resources.
*   **Cursor attributes:** For explicit cursors, attributes are `%ISOPEN`, `%FOUND`, `%NOTFOUND`, and `%ROWCOUNT`. They provide status information after opening, fetching, and closing.
*   **Parameterized cursors:** Explicit cursors that accept parameters. The query can use these parameters. Different sets of data can be fetched by opening the cursor with different parameter values.
*   **Cursor FOR loops:** A simplified way to process explicit cursors. The loop automatically declares a record variable, opens the cursor, fetches each row into the record, and closes the cursor when done. Eliminates manual open/fetch/close and loop termination logic.
*   **Nested cursors:** Involves declaring a cursor that fetches rows, and within the loop over that cursor, opening and fetching from another cursor that depends on the outer cursor's current value. A correlated subquery approach, but can be done manually. Can be inefficient.
*   **REF cursors:** A pointer or reference to a cursor. A `REF CURSOR` type is defined, and variables of that type can point to different result sets at runtime.
*   **Strong vs weak REF cursors:** Strongly typed `REF CURSOR` variables are defined with a `RETURN` type, providing compile-time safety. Weakly typed `REF CURSOR` variables (`SYS_REFCURSOR`) are flexible and can point to any query result set.
*   **Cursor variables:** Variables of type `REF CURSOR`. They can be passed as parameters to procedures and functions, allowing result sets to be passed between PL/SQL programs and to client applications (e.g., returning a result set to Java).

## Exception Handling

*   **Exception architecture:** PL/SQL provides a robust mechanism for dealing with errors. When an error occurs, normal execution stops, and control is transferred to the exception handler of the current block.
*   **Predefined exceptions:** Errors that are automatically raised by PL/SQL and have predefined names, such as `NO_DATA_FOUND`, `TOO_MANY_ROWS`, `DUP_VAL_ON_INDEX`, `ZERO_DIVIDE`. They are declared globally in the `STANDARD` package.
*   **User-defined exceptions:** Exceptions declared by the programmer for application-specific error conditions. They must be declared in the declarative section and explicitly raised using the `RAISE` statement.
*   **RAISE statement:** Used to explicitly raise an exception. Can raise a user-defined exception, a predefined exception, or re-raise the current exception (by using `RAISE;` alone in an exception handler).
*   **RAISE_APPLICATION_ERROR:** A built-in procedure used to raise a user-defined error message and associate it with an error number in the range -20000 to -20999. Often used in stored subprograms to return meaningful error messages to the calling application.
*   **Exception propagation:** If an exception is not handled in the current block, it is propagated to the enclosing block. This continues up the call stack until it is handled. If never handled, the unhandled exception is passed back to the host environment, causing the program to fail.
*   **Exception handling best practices:** Handle specific, expected exceptions first. Use `WHEN OTHERS` only as a catch-all at the outermost block. Avoid using `WHEN OTHERS` without re-raising or logging. Log errors before handling or re-raising.
*   **WHEN OTHERS usage:** Catches any exception not already handled. It's useful for logging unexpected errors and then re-raising (`RAISE;`) to ensure the error isn't silently ignored. Should not be used to suppress errors.

## Stored Programs

*   **Stored procedures:** Subprograms that perform an action. They are compiled and stored in the database. They can accept parameters but do not directly return a value. Called using an anonymous block or from another subprogram.
*   **Stored functions:** Subprograms that compute and return a single value. They can be used in SQL statements (subject to purity rules). They have a `RETURN` clause that specifies the datatype of the value being returned.
*   **IN / OUT / IN OUT parameters:** Parameter modes define data flow: **`IN`** (default) passes a value to the subprogram (read-only). **`OUT`** returns a value from the subprogram to the caller (write-only). **`IN OUT`** passes an initial value to the subprogram and returns a modified value (read-write).
*   **Parameter modes:** `IN` parameters can be literals, constants, or expressions. `OUT` and `IN OUT` parameters must be variables to hold the returned value.
*   **Default parameters:** Parameters can be assigned default values using the `DEFAULT` keyword or `:=`. This allows the caller to omit the argument for that parameter when calling the subprogram.
*   **Function purity:** A level of "side effects" a function has (e.g., reading/writing database state, reading package state). For a function to be callable from a SQL statement, it must be pure (asserted with `PRAGMA RESTRICT_REFERENCES` in older versions) and have no `OUT/IN OUT` parameters.
*   **Deterministic functions:** A function `DETERMINISTIC` returns the same result value whenever it is called with the same input parameters. This hint allows Oracle to cache results and use the function in function-based indexes.
*   **Overloading programs:** Within a package or a type, multiple subprograms can have the same name but different formal parameters (different numbers, types, or parameter modes). Oracle determines which one to call at compile time based on the arguments provided.
*   **Dependency management:** Stored subprograms depend on the schema objects they reference. If an underlying table is altered, the subprogram becomes invalid. It will be automatically recompiled (or manually recompiled) the next time it's called.

## Packages

*   **Package specification:** The public interface. Declares types, variables, constants, exceptions, cursors, and subprograms that are visible and callable from outside the package. It does not contain the code for subprograms.
*   **Package body:** Implements the subprograms declared in the specification. Can also contain private declarations (variables, types, subprograms) that are only accessible within the package body.
*   **Package variables:** Variables declared in the package specification are public and persist for the duration of the session (session state). Variables declared in the package body are private but also persistent for the session.
*   **Package initialization:** The first time a package is referenced in a session, it is instantiated. An optional initialization block at the end of the package body can be used to set initial values for package variables.
*   **Public vs private members:** Public members are declared in the specification and are accessible to any user/program with `EXECUTE` privilege on the package. Private members are declared only in the body and are hidden from external code, encapsulating implementation details.
*   **Package state:** The values of package-level variables and open cursors constitute the package state. This state persists for the user's entire session, allowing data to be shared across calls without database reads/writes.
*   **Stateless vs stateful packages:** A **stateless** package has no persistent variables, making it safe for web applications. A **stateful** package relies on session-persistent data, which can cause issues in connection-pooled environments if not handled carefully.
*   **Package overloading:** Within a package, multiple subprograms (procedures or functions) can share the same name but have different parameter lists. This is a form of polymorphism, making the package interface more intuitive.

## Triggers

*   **Trigger fundamentals:** A stored PL/SQL block (or a single SQL statement) that is automatically executed when a specific event occurs (e.g., DML on a table, DDL, database startup/shutdown). They are often used for auditing, data validation, and maintaining derived data.
*   **BEFORE triggers:** Fire before the triggering DML statement is executed. Used for setting column values (e.g., populating primary keys from a sequence), validating data before it's written, or implementing complex business rules.
*   **AFTER triggers:** Fire after the DML statement has completed. Used for auditing changes and performing actions that depend on the successful completion of the statement (e.g., updating summary tables).
*   **INSTEAD OF triggers:** Defined only on non-editable views. They provide a way to update the underlying base tables of a view through DML on the view itself. The trigger body contains the code to perform the appropriate DML on the base tables.
*   **Row-level triggers:** Fire once for each row affected by the triggering DML statement. They have access to the `:NEW` and `:OLD` pseudo-records, which hold the new and old column values.
*   **Statement-level triggers:** Fire once for the triggering DML statement, regardless of how many rows are affected. They cannot access `:NEW` and `:OLD` per-row values. Useful for logging the execution of the statement itself.
*   **Trigger execution order:** For a given DML statement on a table: 1) `BEFORE STATEMENT` triggers, 2) `BEFORE ROW` triggers (for each row), 3) Execute the DML, 4) `AFTER ROW` triggers, 5) `AFTER STATEMENT` triggers. `INSTEAD OF` triggers replace steps 2-4.
*   **Mutating table error:** A runtime error (ORA-04091) that occurs when a row-level trigger queries or modifies the table that owns the trigger. The table is in a state of change ("mutating"), leading to inconsistent results. Common solutions involve using compound triggers, `AFTER STATEMENT` triggers with collection, or autonomous transactions (use with caution).
*   **Compound triggers:** Introduced in 11g, combines the four timing points (BEFORE STATEMENT, BEFORE ROW, AFTER ROW, AFTER STATEMENT) into a single trigger body. Allows sharing of common data (like collections) across the timing points, elegantly solving the mutating table problem.
*   **Trigger restrictions:** Code within a trigger cannot issue transaction control statements (`COMMIT`, `ROLLBACK`, `SAVEPOINT`) unless it is an autonomous transaction. Long-running logic in triggers can negatively impact DML performance.

## Collections

*   **Collection fundamentals:** PL/SQL collections are like multi-dimensional arrays or lists, allowing you to manipulate sets of data in memory. They are the foundation for bulk processing.
*   **Associative arrays:** Sets of key-value pairs (indexed by `PLS_INTEGER` or `VARCHAR2`). They are dense or sparse. They are stored in the PL/SQL context and are ideal for small, transient lookup tables within a session.
*   **Nested tables:** A table type that can be a column datatype. In PL/SQL, they are initially dense but can become sparse after deletions. Unbounded (size can grow dynamically). Can be created at schema level or within PL/SQL.
*   **VARRAYs:** Variable-size arrays with a fixed maximum size (bounded). They are always dense (no gaps). When stored as a column, the entire array is stored as a single object. Best for ordered collections with a known maximum size.
*   **Collection methods:** Built-in procedures and functions to manage collections: `EXISTS`, `COUNT`, `LIMIT`, `FIRST`, `LAST`, `PRIOR`, `NEXT`, `EXTEND`, `TRIM`, `DELETE`. Essential for dynamic manipulation.
*   **Collection manipulation:** Assigning values, adding elements (`EXTEND`), removing elements (`DELETE`), and iterating through collections using loops.
*   **Bulk operations with collections:** Using collections with `BULK COLLECT` and `FORALL` to transfer data efficiently between SQL and PL/SQL.
*   **Memory considerations:** Collections, especially large nested tables, reside in the session's PGA. Large collections can consume significant memory and impact performance. They should be used judiciously and cleared (`collection.DELETE`) when no longer needed.

## Bulk Processing

*   **Bulk collect:** The `BULK COLLECT INTO` clause is used with `SELECT`, `FETCH`, or `RETURNING INTO`. It retrieves multiple rows from a SQL operation and loads them into a collection in a single pass, dramatically reducing context switches.
*   **FORALL statement:** Not a loop, but a bulk bind construct. It sends a collection of DML statements to the SQL engine in one operation. The DML statement is written once with collection elements used as bind variables via `FORALL i IN 1..collection.COUNT`.
*   **Bulk DML operations:** Using `FORALL` with `INSERT`, `UPDATE`, `DELETE`, or `MERGE` can yield enormous performance gains (sometimes orders of magnitude) over row-by-row processing, especially when dealing with thousands of rows.
*   **Performance benefits:** The primary benefit is the reduction of context switches between the PL/SQL and SQL engines. Each switch adds overhead; bulk operations minimize this by processing many rows in a single switch.
*   **Bulk exception handling:** The `SAVE EXCEPTIONS` clause in a `FORALL` statement allows the DML to continue even if some rows fail. Errors are saved into a special collection `SQL%BULK_EXCEPTIONS`, which can be iterated over after the `FORALL` completes.
*   **SAVE EXCEPTIONS:** When used with `FORALL`, processing continues past DML errors. After the `FORALL`, the `SQL%BULK_EXCEPTIONS` pseudo-collection contains one entry for each error, including the error code and the array index of the failed row.

## Dynamic SQL

*   **Dynamic SQL concepts:** Building and executing SQL statements whose structure is not known until runtime. Essential for DDL, queries with varying numbers of predicates, and schema-generic operations.
*   **Native dynamic SQL:** Using the `EXECUTE IMMEDIATE` statement. It is the preferred method for most dynamic SQL due to its simplicity and performance.
*   **EXECUTE IMMEDIATE:** Parses and immediately executes a dynamic SQL statement. Can be used with `INTO` clause (for single-row results) and `USING` clause (for bind variables). `RETURNING INTO` clause can capture values from DML.
*   **DBMS_SQL package:** An older, more complex API for dynamic SQL. It provides more control, such as parsing, binding by name, defining columns, and executing. Still useful for handling SQL with an unknown number of select list items or for processing dynamic SQL with `OPEN-FOR` variations.
*   **Dynamic queries:** Building `SELECT` statements on the fly. Using `EXECUTE IMMEDIATE ... INTO ...` for single-row results. For multi-row results, use a `REF CURSOR` with `OPEN ... FOR` using a dynamic string.
*   **Bind variables in dynamic SQL:** Using the `USING` clause in `EXECUTE IMMEDIATE` is critical for performance (cursor reuse) and security (preventing SQL injection). Always use bind variables for input values, never concatenate them into the SQL string.
*   **Security considerations:** SQL injection is the primary risk with dynamic SQL. Always validate and sanitize inputs. Use bind variables for all data values. Use `DBMS_ASSERT` to validate object names if they are built from user input.

## Advanced Data Types

*   **Records:** A composite data structure that can hold a row of data from a table or cursor (`%ROWTYPE`) or a custom set of fields defined by the programmer. Useful for passing data as a single unit.
*   **User-defined types (UDTs):** Creating object types (`CREATE OR REPLACE TYPE ... AS OBJECT`) that encapsulate data attributes and methods. Used in object-relational features and for creating complex collections.
*   **Object types:** The foundation of Oracle's object-relational model. They define a structure (attributes) and behavior (methods) for a real-world entity. Can be used as datatypes for table columns, object tables, and collections.
*   **LOB handling:** Large Objects (CLOB, BLOB, BFILE). PL/SQL provides the `DBMS_LOB` package for piecewise access and manipulation of LOB data. Use bind variables of LOB type. For large operations, use the `DBMS_LOB` routines for efficiency.
*   **XML data:** Oracle provides the `XMLType` datatype and a rich set of functions and packages (`XMLQUERY`, `XMLTABLE`, `DBMS_XMLGEN`) to store, query, and generate XML from relational data.
*   **JSON in PL/SQL:** Starting with 12c, Oracle provides native JSON support. The `JSON` datatype, `JSON_OBJECT`, `JSON_ARRAY` functions, and `JSON_TABLE` for querying JSON. PL/SQL can generate and process JSON using these tools.

## Performance Optimization

*   **PL/SQL performance tuning:** Focuses on reducing the number of context switches (bulk processing), optimizing algorithm efficiency within PL/SQL, ensuring SQL statements within PL/SQL are tuned, and minimizing parsing overhead.
*   **Context switching:** The single biggest performance drain in PL/SQL. Every SQL statement executed from PL/SQL involves a switch between the PL/SQL and SQL engines. Bulk operations are the primary tool to mitigate this.
*   **Bulk processing optimization:** Use `BULK COLLECT` with a `LIMIT` clause to avoid consuming too much PGA memory when fetching very large result sets. This processes rows in batches, balancing memory and performance.
*   **Native compilation:** PL/SQL code can be compiled into native C code and linked into Oracle, rather than interpreted bytecode. This can significantly improve performance for compute-intensive procedural logic. Managed via `ALTER COMPILATION`.
*   **PL/SQL optimizer:** The PL/SQL compiler has an optimization level (`PLSQL_OPTIMIZE_LEVEL`). Higher levels (e.g., 2 or 3) perform more aggressive code transformations to improve runtime performance, sometimes at the cost of compilation time.
*   **Profiling PL/SQL code:** Using tools like `DBMS_PROFILER` to identify performance bottlenecks at the line-of-code level. Provides execution counts and time spent in each line, guiding optimization efforts.
*   **Memory usage optimization:** Be mindful of large collections, which reside in PGA. Use `LIMIT` with `BULK COLLECT`. Ensure large collections are freed (`collection.DELETE`) when no longer needed. Consider using `PLS_INTEGER` for integer arithmetic for better performance.

## Transaction Management

*   **Transaction control:** PL/SQL inherits the transaction model from SQL. The first DML statement in a session starts a transaction. It ends with an explicit `COMMIT` or `ROLLBACK`.
*   **Autonomous transactions:** `PRAGMA AUTONOMOUS_TRANSACTION` allows a subprogram to start a new, independent transaction. Its `COMMIT` or `ROLLBACK` does not affect the state of the main transaction. Ideal for logging errors, auditing independently of the main transaction outcome.
*   **Savepoints:** Can be set within PL/SQL. `ROLLBACK TO savepoint` undoes work after the savepoint, while preserving prior work. The transaction remains active.
*   **Locking behavior:** DML in PL/SQL acquires the necessary locks (TX and TM). The duration of locks is until the transaction ends (commit or rollback). Autonomous transactions can interleave locking in complex ways.
*   **Concurrency control:** PL/SQL code runs under the database's isolation level (usually `READ COMMITTED`). The programmer must be aware of potential concurrency issues like lost updates, especially in long-running transactions.
*   **Read consistency:** PL/SQL queries benefit from Oracle's statement-level read consistency. The result set is consistent as of the time the query began.

## Security

*   **Definer vs invoker rights:** **Definer's rights (`AUTHID DEFINER`)** are the default. The subprogram executes with the privileges of the schema that owns it. **Invoker's rights (`AUTHID CURRENT_USER`)** make the subprogram execute with the privileges of the user who calls it, accessing objects in the invoker's schema. Crucial for multi-tenant applications and centralizing data access logic.
*   **Privilege management:** `GRANT EXECUTE ON package_name TO user/role` is required to allow others to run your stored code. With definer's rights, this grants access to the underlying objects indirectly, without granting direct object privileges.
*   **Secure coding practices:** Validate all inputs. Never concatenate user input into dynamic SQL strings; always use bind variables. Avoid `WHEN OTHERS THEN NULL;` which silently swallows errors. Use `AUTHID CURRENT_USER` with caution.
*   **SQL injection prevention:** The primary defense is **bind variables** for all data values. For identifiers (table/column names) built dynamically, use `DBMS_ASSERT` to validate them. Sanitize any input used in dynamic construction.
*   **Role usage in PL/SQL:** In definer's rights subprograms, roles are disabled. The subprogram has only the privileges granted directly to the definer. In invoker's rights subprograms, roles are active, so the effective privilege set is that of the invoker.

## PL/SQL Architecture & Internals

*   **PL/SQL engine:** The component within the Oracle instance that processes PL/SQL units. It parses the code, checks for syntax errors, compiles it (if needed), and executes the procedural logic.
*   **SQL engine interaction:** When the PL/SQL engine encounters a SQL statement, it passes it to the SQL engine for execution. The SQL engine returns results or status information back to the PL/SQL engine. This back-and-forth is a context switch.
*   **Call stack:** The hierarchy of nested subprogram calls. Functions like `DBMS_UTILITY.FORMAT_CALL_STACK` and `DBMS_UTILITY.FORMAT_ERROR_BACKTRACE` allow you to inspect the call stack, which is invaluable for debugging and logging.
*   **Dependency tracking:** Oracle automatically tracks dependencies between PL/SQL units and the schema objects they reference. If a table is altered, dependent procedures, functions, and packages are marked `INVALID` and recompiled automatically upon next use.
*   **Compilation model:** PL/SQL code is compiled into p-code (pseudo-code). This p-code is stored in the database and interpreted at runtime. With native compilation, it's compiled into native machine code for faster execution.
*   **Stored code management:** Code is stored in the data dictionary (`USER_SOURCE`, `ALL_SOURCE`, `DBA_SOURCE`). Management includes granting `EXECUTE`, handling versions (edition-based redefinition), and using source control.

## Debugging & Testing

*   **Debugging PL/SQL:** Using `DBMS_OUTPUT.PUT_LINE` for simple tracing. More advanced: using Oracle SQL Developer's debugger (setting breakpoints, watching variables) or `DBMS_DEBUG`.
*   **Logging techniques:** Implementing robust logging, preferably using an autonomous transaction to log errors to a table without affecting the main transaction. Logging should include timestamps, user, module, error code, error message, and backtrace.
*   **Exception tracing:** Using `SQLCODE` and `SQLERRM` in exception handlers to get error details. Using `DBMS_UTILITY.FORMAT_ERROR_BACKTRACE` to get the exact line number where the error originated, even across nested blocks.
*   **Unit testing PL/SQL:** Writing test cases for individual subprograms to validate behavior. Frameworks like `utPLSQL` can be used. Testing should cover expected paths, boundary conditions, and exception scenarios.
*   **Code instrumentation:** Adding code to log the execution path and timing information. This is not just for errors but for performance monitoring. `DBMS_APPLICATION_INFO.SET_MODULE` and `SET_ACTION` are good for tracing user sessions.

## Integration Features

*   **Calling PL/SQL from applications:** Using JDBC, OCI, ODP.NET, or other language-specific drivers. Typically involves preparing a callable statement, registering output parameters, and executing.
*   **PL/SQL with JDBC:** Using `CallableStatement` to invoke stored procedures/functions. For functions, use `registerOutParameter` to retrieve the return value. For procedures with `OUT` parameters, use the same technique. REF CURSORs are retrieved as `ResultSet`.
*   **External procedures:** Allowing PL/SQL to call programs written in other languages (C, Java) that are stored outside the database. Configured via the `EXTPROC` listener. Use `AS LANGUAGE C` or `AS LANGUAGE JAVA` in the subprogram declaration.
*   **Web services integration:** Using `UTL_HTTP` and `UTL_DBWS` to call external web services from PL/SQL. This enables database code to integrate with external systems.
*   **Scheduler jobs:** Using `DBMS_SCHEDULER` to schedule and manage PL/SQL programs to run at specific times or based on events. Provides robust scheduling, job chaining, and resource management capabilities.

## Advanced Features

*   **Pipelined table functions:** Table functions that return a collection of rows incrementally, acting like a virtual table. Data is "piped" out as it is generated, allowing the result set to be consumed before the function completes. Useful for ETL and transforming data.
*   **Parallel execution:** Table functions can be parallelized. By adding `PARALLEL_ENABLE` and partitioning input data, Oracle can execute multiple instances of the function in parallel on different data slices.
*   **Result cache:** The `RESULT_CACHE` hint or clause caches the results of a function or query in a shared memory area. Subsequent calls with the same parameters retrieve the result from the cache, bypassing computation/database I/O.
*   **Deterministic caching:** Related to function-based indexes and `DETERMINISTIC` functions. Oracle can cache the results of deterministic functions during query execution to avoid recomputation.
*   **Edition-based redefinition (EBR):** A feature for online application upgrades. It allows you to have multiple versions of database objects (including PL/SQL packages) in the same schema, enabling zero-downtime patching and upgrades.

## Real-World Scenarios

*   **Performance bottleneck tuning:** A typical scenario involves identifying a critical process. Steps: find the SQL/PL/SQL involved, check execution plans, identify high I/O or CPU, look for row-by-row processing, and refactor to use bulk operations and set-based SQL.
*   **Bulk data processing design:** Designing a process to handle millions of records. Use `BULK COLLECT` with a `LIMIT`, loop over the batches, and for each batch, perform a `FORALL` to update/insert. Implement logging for each batch to track progress and resume from failures.
*   **Error handling strategy:** A robust strategy: create a log table. In your main process, have exception handlers that call a logging procedure (autonomous transaction) to record the error, the SQL statement, the backtrace, and the key data values. Decide whether to `RAISE` or continue based on business rules.
*   **High-volume transaction design:** For OLTP systems, focus on short, simple transactions. Use bind variables. Keep PL/SQL logic minimal in triggers. Ensure indexes are efficient. Use statement-level triggers instead of row-level where possible. Monitor for contention.
*   **Refactoring legacy PL/SQL:** A task to improve a giant, monolithic procedure. Break it down into smaller, modular procedures and functions within packages. Introduce bulk operations where row-by-row processing exists. Add proper exception handling and logging. Create unit tests.
*   **Debugging production failures:** When a failure occurs, first check the alert log and trace files. Then, check application logs. If an Oracle error was raised, use the error number and message. If you have logging, check the log table. Use the call stack from logs to pinpoint the failing code.
*   **Scalability planning:** As user/data load grows, evaluate PL/SQL code. Remove session-state dependencies that prevent scaling in a stateless mid-tier. Ensure code can run in parallel (e.g., pipelined table functions). Optimize for set-based operations.
*   **Performance bottleneck analysis:** Use ASH to see what sessions were waiting for during a slowdown. If "PL/SQL lock timer" is high, it's due to `DBMS_LOCK.SLEEP` or similar. If "cursor: pin S wait on X" is high, it could be due to frequent invalidations/recompilations. Use AWR to find top SQL by elapsed time and drill down.
*   **Database corruption recovery:** If a block is reported corrupt (`ORA-01578`), first identify the object using the file/block number. Restore the affected datafile from a backup and perform recovery. If it's a logical corruption within a PL/SQL operation, identify the transaction and potentially use Flashback technologies.
*   **Handling deadlocks:** When an application gets a deadlock error, analyze the trace file generated. It shows the sessions, resources, and SQL statements involved. Redesign the application to access tables in a consistent order (e.g., always update table A then table B).
*   **Scaling Oracle systems:** Options include: moving to RAC for scale-out and HA, using partitioning for manageability and parallel query, implementing read-only replicas (Active Data Guard) for reporting, and using sharding for extreme scale.
*   **Disaster recovery planning:** Design a strategy based on Recovery Point Objective (RPO) and Recovery Time Objective (RTO). This could involve regular RMAN backups to a remote site, Data Guard in Maximum Performance or Protection mode, or GoldenGate for real-time replication.
*   **High-availability architecture:** For a critical system, architecture might involve RAC for instance-level HA, Data Guard with a physical standby for site-level DR, and regular RMAN backups for media recovery and point-in-time needs. All components need to be tested regularly with failover drills.
*   **Migration strategies:** Upgrading an application with PL/SQL changes. Use Edition-Based Redefinition to apply changes online with zero downtime. For data migrations, use Data Pump, GoldenGate, or custom PL/SQL with logging and restart capability. Plan for rollback.
*   **Production troubleshooting:** Systematic approach: 1) Assess the impact (is the app down? is it slow?). 2) Check the alert log and listener log. 3) Connect to the database and check `V$SESSION` for blocking locks or long-running operations. 4) Use `V$SQL` to find the SQL being run by problem sessions. 5) Look at recent AWR/ASH reports. 6) Formulate a hypothesis and test it.
