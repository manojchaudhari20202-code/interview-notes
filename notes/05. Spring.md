## Spring Core

### IoC (Inversion of Control)
- Inversion of Control is a principle where the control of object creation and lifecycle is transferred from the application to the container (Spring IoC container). This decouples the execution of a task from its implementation.
- IoC is achieved through Dependency Injection (DI) or Service Locator pattern; Spring primarily uses DI. The container "injects" dependencies rather than the object creating them.
- Benefits include loose coupling, enhanced testability, and easier management of object lifecycles. Components become more modular and interchangeable.
- In Spring, the `ApplicationContext` interface represents the IoC container, responsible for instantiating, configuring, and assembling beans based on configuration metadata.
- The container reads configuration (XML, annotations, Java config) to understand dependencies and wire them appropriately.

### Dependency Injection (DI)
- DI is a design pattern where an object receives its dependencies from an external source rather than creating them internally. In Spring, the container injects dependencies via constructor, setter, or field injection.
- **Constructor Injection**: Dependencies are provided through a class constructor. It is recommended for mandatory dependencies and ensures immutability and thread-safety.
- **Setter Injection**: Dependencies are set via setter methods after object instantiation. Useful for optional dependencies or when you need to reconfigure at runtime.
- **Field Injection**: Using `@Autowired` directly on fields. It's concise but less testable and hides dependencies. Generally discouraged in favor of constructor injection.
- DI promotes loose coupling, making it easier to swap implementations (e.g., using mocks in tests) and adhere to the Dependency Inversion Principle.

### Bean Lifecycle
- Beans in Spring go through a well-defined lifecycle: instantiation, population of properties, setting bean name and bean factory, pre-initialization (BeanPostProcessors), initialization callbacks, and finally destruction.
- Key callback interfaces: `InitializingBean` (afterPropertiesSet()), `DisposableBean` (destroy()), and custom init/destroy methods specified via `@PostConstruct`/`@PreDestroy` or XML attributes.
- `BeanPostProcessor` intercepts bean creation, allowing custom logic before/after initialization (e.g., proxying, wrapping). `postProcessBeforeInitialization` and `postProcessAfterInitialization` methods.
- `BeanFactoryPostProcessor` operates on bean definitions before any beans are created, allowing modification of bean metadata.
- The lifecycle is managed by the container, and understanding it is crucial for tasks like resource initialization/cleanup and AOP proxy creation.

### Bean Scopes
- Bean scope defines the lifecycle and visibility of a bean within the container. Spring provides several built-in scopes: singleton, prototype, request, session, application, and websocket.
- **Singleton** (default): One instance per Spring IoC container. Shared for all requests. Used for stateless beans.
- **Prototype**: A new instance is created each time the bean is requested. Useful for stateful beans.
- **Request**: A single instance per HTTP request. Valid only in web-aware Spring applications.
- **Session**: One instance per HTTP session. Also web-scoped.
- **Application**: Scoped to the ServletContext lifecycle (one instance per web application).
- Custom scopes can be implemented by implementing the `Scope` interface.

### ApplicationContext vs BeanFactory
- `BeanFactory` is the root interface for accessing a Spring bean container. It provides basic IoC functionality (e.g., `getBean()`). Lazy instantiation by default.
- `ApplicationContext` extends `BeanFactory` and adds enterprise features: internationalization (MessageSource), event publication (ApplicationEventPublisher), resource loading, and easier integration with AOP.
- `ApplicationContext` eagerly instantiates singleton beans at startup (unless lazy), whereas `BeanFactory` creates beans on demand.
- Most Spring applications use `ApplicationContext` (e.g., `ClassPathXmlApplicationContext`, `AnnotationConfigApplicationContext`) because of its richer feature set.
- `BeanFactory` is lightweight and can be used in resource-constrained environments like applets or mobile devices.

### Spring Container Internals
- The Spring IoC container is built around the `BeanFactory` and `ApplicationContext` interfaces. It reads configuration metadata to create and manage beans.
- Internally, the container maintains a registry of bean definitions (`BeanDefinition` objects) containing class name, scope, constructor arguments, property values, etc.
- During startup, the container parses configuration, registers bean definitions, processes `BeanFactoryPostProcessor`s (e.g., property placeholders), and then instantiates and initializes beans.
- `BeanPostProcessor`s are applied during bean creation, allowing proxies (AOP) and custom wrapping. The container also handles dependency resolution and injection.
- The container also manages events (e.g., `ContextRefreshedEvent`) and provides hooks for customizing the context (e.g., `ApplicationContextAware`).

### Autowiring Modes
- Autowiring allows Spring to automatically resolve and inject collaborating beans. Modes: `no`, `byName`, `byType`, `constructor`, and `autodetect` (deprecated).
- **byName**: Spring looks for a bean with the same name as the property. Matches based on property name.
- **byType**: Spring looks for a bean of the same type as the property. If multiple beans of that type exist, it fails (unless a primary is specified).
- **constructor**: Similar to byType but applies to constructor arguments. If multiple constructors, ambiguity can occur.
- **autodetect**: Deprecated; attempted constructor then byType.
- Modern approach uses `@Autowired` annotation with fine-grained control (`@Qualifier`, `@Primary`). Autowiring can be combined with explicit wiring for clarity.

### @Component, @Service, @Repository, @Controller
- These are stereotype annotations that mark classes as Spring-managed beans, enabling auto-detection via classpath scanning.
- `@Component` is the generic stereotype for any Spring-managed component.
- `@Service` indicates a service layer bean (business logic). It's a specialization of `@Component`; adds no extra behavior but improves clarity and may be targeted by AOP pointcuts.
- `@Repository` indicates a DAO/repository bean. It adds translation of persistence exceptions to Spring's `DataAccessException` hierarchy.
- `@Controller` marks a web controller (MVC). Often used with `@RequestMapping` for handling web requests. In Spring MVC, it's detected for handler mapping.

### @Configuration and @Bean
- `@Configuration` classes are used to define beans via `@Bean` methods. They are processed by Spring to generate bean definitions and proxy inter-bean dependencies.
- A `@Bean` method indicates that the method returns an object that should be registered as a bean in the Spring context. The method name becomes the bean name by default.
- Inter-bean dependencies: In a `@Configuration` class, calling another `@Bean` method will cause Spring to intercept and return the singleton instance from the container, not create a new instance.
- `@Configuration` classes are themselves CGLIB-proxied to ensure singleton behavior for `@Bean` method calls. Lite mode (plain `@Component` with `@Bean`) does not proxy, so calls create new instances.
- `@Bean` supports initMethod, destroyMethod, autowireCandidate, and other attributes for fine-grained control.

### XML vs Java-based configuration
- **XML configuration**: Traditional approach, using `<beans>`, `<bean>`, `<property>`, etc. Provides clear separation of configuration from code but verbose.
- **Java-based configuration**: Uses `@Configuration` and `@Bean` to define beans in Java code. Type-safe, refactor-friendly, and easier to read for developers.
- **Component scanning**: An alternative where classes are annotated with stereotypes and scanned automatically, reducing explicit configuration.
- Modern Spring Boot favors Java configuration and auto-configuration, but XML may still be used in legacy projects or for specific scenarios.
- Hybrid approaches are possible: import XML config with `@ImportResource` or include Java config classes in XML.

### @Value and property injection
- `@Value` is used to inject externalized properties (from properties files, environment variables, etc.) into Spring beans.
- It supports SpEL (Spring Expression Language) for dynamic expressions, e.g., `@Value("#{systemProperties['user.region']}")`.
- Properties can be injected into fields, constructor parameters, or setter methods. Placeholders like `${property.name}` are resolved by a `PropertySourcesPlaceholderConfigurer`.
- Default values can be provided: `${property.name:defaultValue}`. Useful for optional configuration.
- `@Value` works with any property source: application.properties, YAML, environment variables, system properties, etc.

### Profiles (@Profile)
- `@Profile` allows beans to be registered conditionally based on which profiles are active. It can be applied at class level or method level.
- Profiles enable environment-specific configuration (e.g., dev, test, prod). Beans are only loaded if the specified profile(s) are active.
- Profiles can be activated via `spring.profiles.active` property (command line, environment variable, or programmatically).
- Default profile (`default`) is used if none is active. You can also set `spring.profiles.include` to include other profiles.
- In tests, `@ActiveProfiles` can specify profiles for that test context.

### Environment abstraction
- The `Environment` interface represents the environment in which the current application is running. It provides access to property sources and profiles.
- It abstracts both properties (key-value pairs from various sources) and profiles (named bean definition groups).
- `Environment` can be injected into beans via `@Autowired` or by implementing `EnvironmentAware`.
- It resolves properties with placeholder syntax (`${...}`) and allows programmatic checking of active profiles.
- Used internally by Spring for property resolution and profile-based bean registration.

### Bean post-processors
- `BeanPostProcessor` defines two callback methods: `postProcessBeforeInitialization` and `postProcessAfterInitialization`. It allows custom modification of new bean instances.
- They are applied to all beans in the container, enabling tasks like wrapping beans in proxies (AOP), checking marker interfaces, or modifying bean properties after instantiation.
- Commonly used internally by Spring for features like `@Autowired` injection, `@PostConstruct` invocation, and AOP proxy creation.
- `BeanPostProcessor` beans are themselves instantiated early in the container lifecycle, so they should not depend on other beans that might not be ready.
- Examples: `AutowiredAnnotationBeanPostProcessor`, `CommonAnnotationBeanPostProcessor`, `PersistenceExceptionTranslationPostProcessor`.

### BeanFactoryPostProcessor
- `BeanFactoryPostProcessor` operates on the bean configuration metadata (bean definitions) before any beans are instantiated. It allows modification of bean definitions.
- It is invoked after all bean definitions have been loaded but before any beans are created. Useful for customizing property placeholders or overriding bean properties.
- Common implementations: `PropertyPlaceholderConfigurer` (replaces placeholders), `PropertySourcesPlaceholderConfigurer` (modern replacement), `CustomScopeConfigurer`.
- `BeanFactoryPostProcessor` beans are instantiated early and should not depend on other beans that require post-processing.
- Ordering can be controlled via `PriorityOrdered` or `Ordered` interface.

### FactoryBean
- `FactoryBean` is an interface that allows a class to act as a factory for other beans. It is used when the creation logic for a bean is complex or cannot be expressed via simple constructor/property wiring.
- The `FactoryBean` implementation is registered as a bean, but the actual object returned by `getObject()` is the bean exposed to the container.
- It has methods: `getObject()`, `getObjectType()`, and `isSingleton()`. The container uses these to obtain the bean instance.
- The bean name prefixed with `&` (e.g., `&myFactoryBean`) retrieves the factory bean itself, not the produced object.
- Common use cases: integrating with third-party libraries that require custom instantiation, creating proxy-based beans, or building complex objects.

### Lazy initialization
- By default, Spring eagerly instantiates all singleton beans at startup (or context refresh). Lazy initialization defers bean creation until first request.
- `@Lazy` annotation can be used on beans or configuration classes to indicate lazy loading. It can also be applied to `@Autowired` dependencies to create a lazy-resolution proxy.
- Lazy beans are still singletons; they are created once upon first access, then cached.
- Benefits: faster startup time, especially if many beans are not needed initially. Drawbacks: may hide dependency issues until runtime, and first request might be slower.
- Lazy resolution can also be used for circular dependencies: one of the beans can be lazily injected to break the cycle.

### Circular dependencies
- Circular dependencies occur when two or more beans depend on each other (e.g., A depends on B, B depends on A). Spring tries to resolve them at runtime.
- Constructor injection circular dependencies are generally unresolvable and throw `BeanCurrentlyInCreationException`. Setter injection can often be resolved because Spring can instantiate beans without fully initializing them.
- Spring uses a three-level cache (singleton objects) to handle circular dependencies for setter injection: early reference exposure (via `ObjectFactory`) allows injecting a proxy before full initialization.
- Solutions: refactor code to avoid circular dependencies, use setter/field injection instead of constructor, use `@Lazy` on one of the dependencies, or use `@PostConstruct` to break the cycle.
- Circular dependencies often indicate design issues; consider applying Dependency Inversion or introducing a mediator.

### Conditional bean loading
- Conditional loading allows beans to be registered only if certain conditions are met. Spring provides `@Conditional` annotation and built-in condition annotations like `@ConditionalOnClass`, `@ConditionalOnProperty`, etc. (Spring Boot).
- `@Conditional` can be used with a custom `Condition` implementation that evaluates based on the `ConditionContext` and `AnnotatedTypeMetadata`.
- Spring Boot's auto-configuration heavily uses conditional annotations to apply configurations only when needed (e.g., if a class is on the classpath, or a property is set).
- Common condition annotations: `@ConditionalOnMissingBean`, `@ConditionalOnBean`, `@ConditionalOnClass`, `@ConditionalOnProperty`, `@ConditionalOnWebApplication`.
- This enables flexible, environment-specific bean registration and helps prevent conflicts in auto-configuration.

## Spring Bean Management

### Singleton vs Prototype
- **Singleton scope**: One instance per Spring IoC container. All requests for that bean return the same instance. Suitable for stateless beans, services, and DAOs.
- **Prototype scope**: A new instance is created each time the bean is requested (via `getBean()` or injected into another bean). Ideal for stateful beans.
- Singleton beans are created at container startup (unless lazy) and cached. Prototype beans are created on demand, and the container does not manage their full lifecycle (destroy callbacks are not called unless custom).
- Injection of prototype into singleton: if a singleton bean holds a reference to a prototype, that prototype is injected once (at singleton creation). To get a new prototype each time, use `@Lookup` method injection or `ObjectFactory` / `Provider`.
- Performance considerations: singletons reduce object creation overhead but may cause concurrency issues if stateful; prototypes increase object creation but are thread-safe by nature.

### Request, Session, Application scopes
- These scopes are available in web-aware Spring applications (e.g., Spring MVC). They are backed by a proxy mechanism.
- **Request scope**: A bean instance is created for each HTTP request and is destroyed when the request completes. Useful for request-specific data (e.g., form backing objects).
- **Session scope**: One bean per HTTP session. Lives as long as the session. Suitable for user preferences, shopping carts.
- **Application scope**: Bean is scoped to the `ServletContext` lifecycle, essentially a singleton per web application but visible only in web context.
- These scoped beans are typically proxied (using scoped proxy mode) because they need to be injected into longer-lived beans (e.g., singletons). The proxy delegates to the actual scoped instance at runtime.
- Configuration: `@Scope(value = "request", proxyMode = ScopedProxyMode.TARGET_CLASS)`.

### Custom scopes
- Spring allows defining custom scopes beyond the built-in ones by implementing the `Scope` interface.
- The `Scope` interface defines methods like `get`, `remove`, `registerDestructionCallback`, `resolveContextualObject`, and `getConversationId`.
- Custom scopes are useful for specialized contexts like thread-based, job-based, or conversation scopes (e.g., in JSF or custom workflows).
- To register a custom scope, use `ConfigurableBeanFactory.registerScope(String scopeName, Scope scope)`.
- Scoped beans may require proxy mode to be injected into longer-lived beans; this is handled by the scope implementation and proxy generation.

### Bean initialization callbacks
- Spring provides several ways to define custom initialization logic after bean properties are set:
  - Implementing `InitializingBean` interface and overriding `afterPropertiesSet()`.
  - Using `@PostConstruct` annotation on a method.
  - Specifying a custom init method via `@Bean(initMethod = "init")` or XML `init-method`.
- `@PostConstruct` is the preferred approach as it is not tied to Spring-specific interfaces and works with JSR-250.
- Order of execution: `@PostConstruct` → `afterPropertiesSet()` → custom init method.
- Initialization callbacks are invoked after dependency injection is complete, making them safe to use for setting up resources.

### @PostConstruct / @PreDestroy
- `@PostConstruct` and `@PreDestroy` are JSR-250 annotations for lifecycle callbacks.
- `@PostConstruct` marks a method to be executed after dependency injection is done to perform any initialization (e.g., opening connections, validating configuration).
- `@PreDestroy` marks a method to be executed just before the bean is removed from the container (e.g., closing resources, cleaning up).
- These methods must have no parameters and may return void. They can throw unchecked exceptions.
- They are supported by Spring's `CommonAnnotationBeanPostProcessor` (registered by default in `ApplicationContext`).

### InitializingBean / DisposableBean
- `InitializingBean` and `DisposableBean` are Spring-specific lifecycle interfaces.
- `InitializingBean` has a single method `afterPropertiesSet()` called after all properties are set.
- `DisposableBean` has a single method `destroy()` called when the container is shutting down.
- Using these interfaces ties your code to Spring, so they are less favored than `@PostConstruct`/`@PreDestroy` or custom init/destroy methods.
- However, they can be useful when you need to implement a bean that is not annotation-driven.

### @DependsOn
- `@DependsOn` annotation forces one or more beans to be initialized before the annotated bean. It can be used at class level or method level (for `@Bean`).
- It is useful when a bean depends on other beans not directly referenced as properties, such as database schema initialization or singleton startup tasks.
- The annotation accepts an array of bean names that must be created first.
- Without `@DependsOn`, Spring determines initialization order based on direct dependencies; `@DependsOn` adds explicit ordering.

### Bean definition overriding
- Bean definition overriding occurs when multiple bean definitions with the same name exist in the container. The behavior depends on the configuration and application context.
- By default, Spring allows overriding: the last definition wins. This can be controlled via `allowBeanDefinitionOverriding` property (e.g., in `DefaultListableBeanFactory`).
- In Spring Boot, overriding is allowed by default but can be disabled via `spring.main.allow-bean-definition-overriding=false`.
- Overriding can lead to unexpected behavior; it's generally better to use unique bean names or profile-specific definitions.
- Common scenario: overriding auto-configured beans with custom ones (e.g., custom `DataSource`).

### Proxy-based beans
- Proxy-based beans are used to implement cross-cutting concerns like AOP, transaction management, or scoped proxies. Spring creates a proxy object that wraps the target bean.
- Proxies intercept method calls and apply additional behavior before/after invoking the target.
- Two proxy mechanisms: JDK dynamic proxies (based on interfaces) and CGLIB proxies (subclassing).
- Proxies are often transparent but have implications: self-invocation problems, final methods cannot be proxied, and proxy type matters for injection.
- Examples: `@Transactional` beans, scoped proxies, `@Async` beans.

### CGLIB vs JDK proxy
- **JDK dynamic proxy**: Creates a proxy for a target object by implementing interfaces. The proxy implements all interfaces of the target. Requires at least one interface.
- **CGLIB proxy**: Creates a subclass of the target class, intercepting method calls via inheritance. Can proxy classes without interfaces. It uses bytecode generation.
- Spring chooses proxy type based on configuration: if target implements at least one interface, JDK proxy is used by default; otherwise, CGLIB. This can be forced to CGLIB via `proxyTargetClass=true`.
- CGLIB cannot proxy final classes or methods; JDK proxies require interfaces.
- Performance: CGLIB is generally faster for proxy creation but may have overhead in method invocation. Modern versions are comparable.

## Spring AOP

### Aspect-Oriented Programming basics
- AOP is a programming paradigm that aims to increase modularity by allowing separation of cross-cutting concerns (e.g., logging, security, transactions).
- It complements OOP by providing a way to apply behaviors (advice) across multiple points in an application without modifying the code directly.
- Core concepts: Aspect (module encapsulating concern), Join Point (point in execution), Advice (action taken), Pointcut (expression selecting join points), Introduction (adding new methods/fields), Target object (object being advised), Weaving (linking aspects with objects).
- Spring AOP is proxy-based, using either JDK dynamic proxies or CGLIB. It is limited to method interception on Spring beans.
- AspectJ is a more powerful, full-featured AOP framework that supports compile-time, post-compile, and load-time weaving; Spring can integrate with AspectJ.

### Cross-cutting concerns
- Cross-cutting concerns are functionalities that span multiple modules of an application and are not easily modularized using traditional OOP (e.g., logging, transaction management, security, caching).
- In AOP, these concerns are encapsulated in aspects, which can be applied declaratively across the application.
- This reduces code duplication, improves maintainability, and makes core business logic cleaner.
- Examples: logging method entry/exit, measuring method execution time, checking user permissions before method execution.
- By separating these concerns, you adhere to the Single Responsibility Principle and make the codebase more modular.

### JoinPoint
- A join point is a point during the execution of a program, such as method invocation, constructor call, field access, etc. In Spring AOP, join points always represent method execution.
- In advice methods, you can access join point information (method signature, arguments, target object) via `JoinPoint` parameter (for `@Before`, `@After`, etc.) or `ProceedingJoinPoint` (for `@Around`).
- `JoinPoint` provides methods: `getArgs()`, `getSignature()`, `getTarget()`, `getThis()`, `getKind()`, etc.
- `ProceedingJoinPoint` extends `JoinPoint` and adds `proceed()` to continue the method execution (used in around advice).
- Join point metadata is useful for logging, auditing, or conditional logic based on method parameters.

### Pointcut expressions
- Pointcut expressions define which join points should be matched by advice. They are written using AspectJ pointcut syntax.
- Common designators: `execution()` (method execution), `within()` (within certain types), `args()` (argument types), `@annotation()` (methods with certain annotation), `bean()` (Spring bean name), `this()`, `target()`.
- Pointcut expressions can be combined with logical operators `&&`, `||`, `!`.
- Example: `@Pointcut("execution(* com.example.service.*.*(..))")` matches all methods in classes under `service` package.
- Pointcuts can be defined separately and reused across advice methods.

### Advice types (Before, After, Around, AfterReturning, AfterThrowing)
- **@Before**: Executed before the join point. Cannot prevent method execution (unless exception thrown).
- **@After**: Executed after the join point, regardless of the outcome (success or exception). Like finally block.
- **@AfterReturning**: Executed after successful return from the join point. Can access return value via `returning` attribute.
- **@AfterThrowing**: Executed if the join point throws an exception. Can access exception via `throwing` attribute.
- **@Around**: Most powerful; wraps the join point, allowing custom behavior before, after, and control over whether the method proceeds. Must call `proceed()` on `ProceedingJoinPoint`.
- Advice execution order can be controlled via `@Order` or implementing `Ordered`.

### @Aspect
- `@Aspect` annotation marks a class as an aspect containing advice and pointcut definitions. It must be declared as a Spring bean (e.g., via `@Component` or `@Bean`).
- Spring AOP processes `@Aspect` classes and creates proxies for beans whose methods match pointcuts.
- Within an aspect class, you define pointcuts with `@Pointcut` methods and advice with annotations like `@Before`.
- Aspect classes can have regular methods, fields, and dependencies injected.
- Aspect instantiation models: singleton (default), per-target, per-control-flow (via `@Scope("prototype")`? Actually, AspectJ supports perthis, pertarget, but Spring AOP only supports singleton aspects per bean).

### Proxy mechanism
- Spring AOP creates proxies for target beans to apply advice. The proxy intercepts method calls and delegates to the appropriate advice chain.
- Two proxy factories: `ProxyFactoryBean` for declarative proxy creation, and `ProxyFactory` for programmatic use. Spring auto-proxies beans based on advisor definitions.
- The proxy mechanism determines which proxy type to use (JDK or CGLIB). By default, if target implements at least one interface, JDK proxy; otherwise CGLIB. Can be forced to CGLIB.
- Proxies are created during bean initialization phase, after `BeanPostProcessor`s like `AnnotationAwareAspectJAutoProxyCreator`.
- Self-invocation (calling a method within the same class) bypasses the proxy, so advice is not applied. This is a common pitfall.

### AOP vs AspectJ
- **Spring AOP**: Proxy-based, simpler, integrates with Spring IoC, only supports method execution join points, runs within the Spring container. Ideal for most enterprise scenarios.
- **AspectJ**: Full-fledged AOP framework, supports all join points (field access, constructors, etc.), uses compile-time or load-time weaving, can be used outside Spring.
- Spring can integrate with AspectJ via `@AspectJ` annotations (using AspectJ's pointcut language) while still using Spring AOP for proxying. For load-time weaving, Spring can also use AspectJ weaver.
- AspectJ offers better performance for some scenarios and more power, but adds complexity (compiler or agent).
- Decision: Use Spring AOP for most method-level cross-cutting concerns; use AspectJ if you need finer-grained join points or are outside Spring environment.

### Transactional proxies
- `@Transactional` annotation is implemented via Spring AOP. A proxy is created around the target bean, and transaction management advice is applied.
- The proxy begins, commits, or rolls back a transaction based on method execution.
- Transactional proxies are JDK or CGLIB proxies. The proxy must be public and not called from within the same class (self-invocation issue).
- Transaction attributes (propagation, isolation, timeout, read-only) are configured via `@Transactional` parameters.
- Proxies integrate with Spring's `PlatformTransactionManager` (or reactive equivalent) to manage transactions.

### Self-invocation problem
- When a method within a bean calls another method of the same bean, the call is a direct `this` reference, not through the proxy. Thus, any advice (transactional, security, caching) on the called method is not applied.
- Solutions:
  - Refactor to separate classes (move the method to another bean) so calls go through proxy.
  - Use self-injection: inject the bean into itself (using `@Autowired` or `ApplicationContext.getBean()`) and call via that proxy.
  - Use `AopContext.currentProxy()` (requires exposing proxy via `@EnableAspectJAutoProxy(exposeProxy=true)`).
  - Use AspectJ compile-time weaving (LTW) to avoid proxies.
- This issue is common with `@Transactional` methods calling each other.

### Performance implications of AOP
- Proxy-based AOP adds a small overhead due to method interception, reflection, and advice execution. However, this is usually negligible for most applications.
- The overhead increases with number of advice and complexity of pointcut matching. Complex pointcuts evaluated at runtime can degrade performance.
- Around advice that does heavy processing before/after method execution can introduce latency.
- For high-performance scenarios, consider using AspectJ compile-time weaving which may have less runtime overhead.
- Caching of pointcut evaluation and proxy creation helps; Spring caches proxies and pointcut matches.

## Spring Data Access

### JDBC Template
- `JdbcTemplate` is Spring's central class for JDBC operations, simplifying raw JDBC usage and eliminating boilerplate code (resource handling, exception translation).
- It executes SQL queries, updates, and batch operations. Handles connection acquisition and release automatically.
- Provides methods for querying: `query()`, `queryForObject()`, `queryForList()`, etc., with `RowMapper` to map result sets to objects.
- It also handles statement creation and prepared statement parameter setting via `PreparedStatementSetter` or varargs.
- `JdbcTemplate` works with a `DataSource` and translates `SQLException` into Spring's `DataAccessException` hierarchy.

### NamedParameterJdbcTemplate
- Extension of `JdbcTemplate` that supports named parameters in SQL (e.g., `:name` instead of `?`). This improves readability and maintainability.
- Uses `SqlParameterSource` (Map, BeanPropertySqlParameterSource, or MapSqlParameterSource) to bind parameters.
- Internally, it delegates to a wrapped `JdbcTemplate` after converting named parameters to traditional placeholders.
- Particularly useful when there are many parameters or when using Java beans directly.
- Supports batch updates with named parameters.

### Spring ORM
- Spring ORM module provides integration with object-relational mapping (ORM) frameworks like Hibernate, JPA, JDO, and iBatis.
- It offers consistent resource management, transaction abstraction, and exception translation specific to each ORM.
- Key classes: `HibernateTemplate` (for Hibernate 3/4, but now less used), `JpaTemplate` (deprecated in favor of native EntityManager usage with `@PersistenceContext`).
- Spring simplifies ORM by handling `SessionFactory` or `EntityManagerFactory` lifecycle, and providing `@Repository` with exception translation.
- It also integrates with declarative transaction management.

### Hibernate integration
- Spring can integrate with Hibernate by configuring a `LocalSessionFactoryBean` to create a Hibernate `SessionFactory` as a Spring bean.
- The `SessionFactory` can be injected into DAOs using `@Autowired` or `@Resource`. DAOs can then use `getCurrentSession()` to obtain a session bound to the transaction.
- Spring's `HibernateTransactionManager` manages transactions and binds Hibernate sessions to threads.
- With Hibernate 5+, Spring's `HibernateJpaVendorAdapter` is used for JPA integration; native Hibernate integration is still possible via `HibernateTemplate` (deprecated) or direct `SessionFactory` usage.
- Exception translation is provided by `@Repository` annotation (via `PersistenceExceptionTranslationPostProcessor`).

### JPA integration
- Spring integrates with JPA by providing `LocalEntityManagerFactoryBean` or `LocalContainerEntityManagerFactoryBean` for more control (e.g., specifying `PersistenceUnitManager`, `DataSource`, etc.).
- `EntityManagerFactory` is created as a Spring bean, and `@PersistenceContext` can inject a container-managed `EntityManager` into DAOs (with transaction-scoped persistence context).
- Spring's `JpaTransactionManager` manages JPA transactions, supporting both local and JTA transactions.
- `@Repository`-annotated DAOs automatically get JPA exception translation to Spring's `DataAccessException`.
- Spring Data JPA builds on this to provide repository abstractions with query methods.

### EntityManager
- `EntityManager` is the JPA interface for interacting with the persistence context: find, persist, merge, remove, and query entities.
- In Spring, `EntityManager` is typically injected via `@PersistenceContext` (not `@Autowired`). Spring manages the lifecycle and transaction binding.
- The injected `EntityManager` is a thread-safe proxy that delegates to the actual entity manager bound to the current transaction (transaction-scoped).
- Container-managed entity managers are preferred; you should not create or close them manually.
- For programmatic use, you can obtain an `EntityManager` via `EntityManagerFactory.createEntityManager()` but must manage its lifecycle and transactions.

### Transaction management
- Spring provides a consistent abstraction for transaction management across different APIs (JDBC, JPA, Hibernate, JTA).
- Two types: programmatic (via `TransactionTemplate` or `PlatformTransactionManager`) and declarative (via `@Transactional` annotation or XML configuration).
- The core interface is `PlatformTransactionManager` (or `ReactiveTransactionManager` for reactive). Implementations include `DataSourceTransactionManager`, `JpaTransactionManager`, `HibernateTransactionManager`, `JtaTransactionManager`.
- Transactions are associated with a thread (thread-bound) via `TransactionSynchronizationManager`. For reactive, they are tied to reactor context.
- Spring manages transaction propagation, isolation, read-only hints, and rollback rules.

### Declarative transactions
- Declarative transactions are enabled by `@EnableTransactionManagement` (or XML equivalent) and `@Transactional` annotations.
- `@Transactional` can be applied to interfaces, classes, or methods. It defines transaction boundaries and attributes.
- The annotation is processed by a `BeanPostProcessor` that creates AOP proxies around `@Transactional` beans.
- It works with any Spring-managed `PlatformTransactionManager` bean.
- Declarative transactions keep business code free of transaction boilerplate.

### @Transactional propagation
- Propagation defines how transactions relate to each other when a transactional method is called within an existing transaction.
- Common propagation levels:
  - `REQUIRED` (default): join existing transaction or create new one if none exists.
  - `REQUIRES_NEW`: suspend current transaction and create a new one.
  - `SUPPORTS`: execute within a transaction if exists, else non-transactionally.
  - `MANDATORY`: must run within a transaction, else throw exception.
  - `NOT_SUPPORTED`: suspend transaction if exists, execute non-transactionally.
  - `NEVER`: must not run within a transaction, else throw exception.
  - `NESTED`: execute within a nested transaction if supported (JDBC savepoints).
- Choosing correct propagation is critical for data integrity and performance.

### Isolation levels
- Isolation level defines how transaction changes are visible to other concurrent transactions, addressing phenomena like dirty reads, non-repeatable reads, and phantom reads.
- Spring supports the standard JDBC isolation levels:
  - `DEFAULT`: use underlying datasource default.
  - `READ_UNCOMMITTED`: lowest, allows dirty reads.
  - `READ_COMMITTED`: prevents dirty reads, non-repeatable reads possible.
  - `REPEATABLE_READ`: prevents dirty and non-repeatable reads, phantom reads possible.
  - `SERIALIZABLE`: highest, fully isolated.
- Higher isolation increases data consistency but reduces concurrency and may cause locks.
- Isolation level is set via `@Transactional(isolation = Isolation.READ_COMMITTED)`.

### Rollback rules
- By default, declarative transactions roll back on runtime exceptions (`RuntimeException` and `Error`) but not on checked exceptions.
- Rollback rules can be customized using `rollbackFor`, `rollbackForClassName`, `noRollbackFor`, `noRollbackForClassName` attributes.
- Example: `@Transactional(rollbackFor = Exception.class)` to roll back on any exception.
- Rollback behavior is essential for exception handling and ensuring data consistency.
- Programmatic rollback can also be done via `TransactionStatus.setRollbackOnly()`.

### Programmatic transactions
- Programmatic transaction management gives fine-grained control over transactions directly in code.
- Two approaches: using `TransactionTemplate` (with callback) or directly using `PlatformTransactionManager`.
- `TransactionTemplate` is similar to `JdbcTemplate`; you provide a `TransactionCallback` (or `TransactionCallbackWithoutResult`) and it handles transaction lifecycle.
- Direct use of `PlatformTransactionManager` involves `getTransaction()`, `commit()`, `rollback()`.
- Programmatic transactions are less common but useful when declarative approach is insufficient (e.g., transaction logic depends on runtime conditions).

### Lazy loading issues
- Lazy loading in ORM (e.g., Hibernate) means associated data is loaded only when accessed. In web applications, this can cause `LazyInitializationException` if accessed after the session/transaction is closed.
- Common solutions: ensure session is open during view rendering (Open Session in View pattern), eagerly fetch needed associations (JOIN FETCH), use DTOs to load required data, or use `@Transactional` in service layer to keep session open.
- Spring can handle `LazyInitializationException` by configuring an OpenEntityManagerInViewFilter (web.xml) or interceptor (Spring Boot includes it by default, though it's debated).
- Lazy loading can impact performance if not managed; profiling is needed to detect unnecessary lazy loads.

### N+1 problem
- The N+1 problem occurs when an ORM executes one query to fetch parent entities and then N additional queries to fetch related collections for each parent.
- Example: Fetching authors and their books with lazy loading leads to 1 query for authors + N queries for each author's books.
- Solutions: use eager fetching (JOIN FETCH), batch fetching (`@BatchSize`), entity graphs, or write custom JPQL with fetch joins.
- Spring Data JPA's `@EntityGraph` allows defining fetch strategies dynamically.
- N+1 can severely degrade performance; it's crucial to detect via SQL logging or tools like `hibernate.stat`.

### Open Session in View
- Open Session in View (OSIV) is a pattern where the Hibernate session (or JPA EntityManager) is kept open during the entire web request, allowing lazy loading in the view layer.
- Implemented via a servlet filter (e.g., `OpenEntityManagerInViewFilter`) or interceptor. Spring Boot enables it by default (`spring.jpa.open-in-view=true`).
- Pros: simplifies lazy loading in views, reduces LazyInitializationException.
- Cons: holds database connection longer, can cause transaction issues (e.g., writes in view), may lead to performance problems under load.
- Many architects advise disabling OSIV and using DTOs or explicit fetching to avoid pitfalls.

## Spring MVC

### DispatcherServlet
- `DispatcherServlet` is the front controller in Spring MVC. It intercepts all HTTP requests (based on url-pattern) and delegates to various components to handle the request.
- It uses `HandlerMapping` to determine the appropriate controller, `HandlerAdapter` to invoke the controller, and `ViewResolver` to resolve views.
- The servlet is configured in `web.xml` or via Java configuration (e.g., `AbstractAnnotationConfigDispatcherServletInitializer`). In Spring Boot, it's auto-configured.
- It also handles multipart requests, locale resolution, theme resolution, and exception handling.
- `DispatcherServlet` initializes its context (WebApplicationContext) which may be a child of the root application context.

### HandlerMapping
- `HandlerMapping` maps incoming requests to handlers (controller methods) based on request metadata (URL, HTTP method, parameters, headers).
- Implementations: `RequestMappingHandlerMapping` (for annotated controllers with `@RequestMapping`), `BeanNameUrlHandlerMapping` (maps beans to URLs by name), `SimpleUrlHandlerMapping` (explicit mapping).
- `HandlerMapping` returns a `HandlerExecutionChain` containing the handler and any interceptors.
- In Spring MVC, `RequestMappingHandlerMapping` is the default for `@Controller` classes.
- Ordering of multiple `HandlerMapping` beans can be defined; the first one returning a handler is used.

### HandlerAdapter
- `HandlerAdapter` is responsible for actually invoking the handler method. It adapts the handler to the servlet API.
- For annotated controllers, `RequestMappingHandlerAdapter` is used. It resolves method arguments (e.g., `@RequestParam`, `@PathVariable`) and handles return values (ModelAndView, ResponseEntity, etc.).
- Other adapters: `HttpRequestHandlerAdapter` (for `HttpRequestHandler`), `SimpleControllerHandlerAdapter` (for legacy `Controller` interface).
- `HandlerAdapter` also participates in exception handling via `@ExceptionHandler` methods.
- It uses `HandlerMethodArgumentResolver` and `HandlerMethodReturnValueHandler` internally.

### ViewResolver
- `ViewResolver` resolves logical view names returned by controllers to actual `View` implementations (e.g., JSP, Thymeleaf, JSON).
- Common implementations: `InternalResourceViewResolver` (for JSP), `ThymeleafViewResolver`, `BeanNameViewResolver`, `ContentNegotiatingViewResolver`.
- `View` objects render the model data to the response (HTML, JSON, PDF, etc.).
- View resolution can be chained; multiple resolvers can be tried in order.
- In REST APIs, controllers often return `@ResponseBody` or `ResponseEntity`, bypassing view resolution.

### @Controller
- `@Controller` is a stereotype annotation indicating that a class serves as a web controller, capable of handling HTTP requests.
- It is typically used with `@RequestMapping` at class and method levels to map URLs.
- Controller methods can accept various arguments (request params, path variables, model attributes) and return model and view names, `ModelAndView`, or `@ResponseBody` for REST.
- `@Controller` classes are auto-detected via classpath scanning when `@ComponentScan` is used.
- They can also be annotated with `@SessionAttributes` to store model attributes in HTTP session.

### @RestController
- `@RestController` is a convenience annotation combining `@Controller` and `@ResponseBody`. It implies that every handler method in the class writes directly to the HTTP response body (like REST API).
- It eliminates the need to annotate each method with `@ResponseBody`.
- Commonly used for RESTful web services returning JSON/XML.
- `@RestControllerAdvice` can be used for global exception handling in REST controllers.
- It integrates with Spring's `HttpMessageConverter` for automatic marshalling.

### @RequestMapping
- `@RequestMapping` maps HTTP requests to handler methods of MVC and REST controllers. Can be applied at class level (base URL) and method level.
- Attributes: `value` / `path` (URL), `method` (HTTP method), `params` (filter by parameters), `headers` (filter by headers), `consumes` (media types request can consume), `produces` (media types response can produce).
- Shortcut annotations: `@GetMapping`, `@PostMapping`, `@PutMapping`, `@DeleteMapping`, `@PatchMapping` – these are composed annotations with predefined HTTP method.
- It supports URI templates with path variables (e.g., `/users/{id}`).
- In Spring 4.3+, the shortcut annotations are preferred for clarity.

### @GetMapping / @PostMapping
- These are specialized versions of `@RequestMapping` for specific HTTP methods.
- `@GetMapping`: equivalent to `@RequestMapping(method = RequestMethod.GET)`. Used for read operations.
- `@PostMapping`: for POST, typically used for creating resources.
- `@PutMapping`: for PUT (update/replace).
- `@DeleteMapping`: for DELETE.
- `@PatchMapping`: for partial updates.
- They support the same attributes as `@RequestMapping` (value, produces, consumes, etc.) and improve readability.

### Path variables
- Path variables are extracted from the URI template using `@PathVariable` annotation in handler method parameters.
- Example: `@GetMapping("/users/{id}") public String getUser(@PathVariable("id") Long id)`.
- Path variables can be optional (if not required) and can be renamed via annotation value.
- They are automatically converted to the method parameter type (e.g., String to Long).
- Multiple path variables can be used in a single mapping.

### Request parameters
- Request parameters (query string) are extracted with `@RequestParam` annotation.
- Example: `@GetMapping("/users") public String getUsers(@RequestParam("page") int page)`.
- `@RequestParam` attributes: `value` (parameter name), `required` (default true), `defaultValue` (if parameter not present).
- For optional parameters, set `required=false` and use wrapper types or default value.
- When a Map is used without specifying name, all request parameters are injected.

### @RequestBody
- `@RequestBody` binds the HTTP request body to a Java object using registered `HttpMessageConverter` (e.g., JSON to object via Jackson).
- Typically used in REST APIs with POST/PUT to accept JSON/XML payloads.
- Validation can be applied with `@Valid` on the parameter, triggering method-level validation.
- The request body is read and converted once; cannot be read again in the same request.
- For file uploads, `MultipartFile` is used with `@RequestParam`, not `@RequestBody`.

### @ResponseBody
- `@ResponseBody` indicates that the return value of a handler method should be written directly to the HTTP response body (not as a view name). The value is converted using `HttpMessageConverter`.
- In `@RestController`, it's implied on all methods.
- It can be used in `@Controller` methods to return JSON/XML data.
- Examples: returning a `ResponseEntity` (with status and headers) also implies `@ResponseBody`.
- When used with `@ResponseStatus`, you can set the HTTP status.

### HttpMessageConverters
- `HttpMessageConverter` is a strategy interface for converting HTTP requests and responses between Java objects and various media types (JSON, XML, etc.).
- Spring Boot auto-configures converters based on classpath dependencies (e.g., Jackson for JSON, JAXB for XML).
- Common converters: `MappingJackson2HttpMessageConverter` (JSON), `Jaxb2RootElementHttpMessageConverter` (XML), `FormHttpMessageConverter` (form data).
- Custom converters can be added by implementing `HttpMessageConverter` and registering in Spring MVC configuration.
- Used by `@RequestBody`, `@ResponseBody`, and `ResponseEntity` handling.

### Exception handling (@ExceptionHandler, @ControllerAdvice)
- `@ExceptionHandler` within a controller handles exceptions thrown by that controller's methods. It can return a `ModelAndView` or `ResponseEntity`.
- `@ControllerAdvice` (or `@RestControllerAdvice`) allows global exception handling across multiple controllers. It can contain `@ExceptionHandler`, `@InitBinder`, `@ModelAttribute` methods.
- `@ExceptionHandler` methods can access the exception via parameter and have flexible return types (String, ModelAndView, ResponseEntity).
- For REST APIs, using `@RestControllerAdvice` with `ResponseEntity` provides consistent error responses.
- It's possible to handle specific exceptions or use a catch-all with `Exception.class`.

### Validation (@Valid, @Validated)
- Bean Validation (JSR-380) is supported in Spring MVC via `@Valid` annotation on `@RequestBody`, `@RequestPart`, or `@ModelAttribute` parameters.
- `@Validated` is a Spring variant that supports validation groups (useful for partial validation in different scenarios).
- Validation errors are captured in a `BindingResult` parameter that must immediately follow the validated object.
- If `BindingResult` is not present, a `MethodArgumentNotValidException` is thrown, which can be handled globally.
- For path variables and request parameters, validation can be done by annotating the controller with `@Validated` and using validation annotations on parameters (e.g., `@Min`).

### BindingResult
- `BindingResult` holds the result of data binding and validation for a command object (e.g., `@ModelAttribute` or `@RequestBody` when used with form handling).
- It must appear immediately after the validated object in the method signature, else validation errors throw an exception.
- `BindingResult` allows checking for errors (`hasErrors()`) and accessing field errors.
- Typically used in traditional MVC form handling to return to the form view with error messages.
- In REST, `BindingResult` is less common; instead, exceptions are used.

### CORS configuration
- Cross-Origin Resource Sharing (CORS) allows web applications from one origin to access resources from another origin.
- Spring MVC provides `@CrossOrigin` annotation at controller or method level to enable CORS.
- Global CORS configuration can be done by overriding `addCorsMappings` in `WebMvcConfigurer`.
- CORS configuration includes allowed origins, methods, headers, exposed headers, allow credentials, and max age.
- Spring Boot also allows configuration via properties (`spring.web.cors.*`).

### File upload
- File upload in Spring MVC is handled by configuring a `MultipartResolver` (e.g., `CommonsMultipartResolver` or `StandardServletMultipartResolver`).
- In controller methods, `@RequestParam("file") MultipartFile file` binds the uploaded file.
- `MultipartFile` provides methods to get original filename, content type, bytes, and transfer to file.
- Multiple files can be handled via `MultipartFile[]` or `List<MultipartFile>`.
- Configuration includes max file size, max request size, and temporary location.

### Content negotiation
- Content negotiation determines the media type to return based on request headers (Accept), URL suffix (.json, .xml), or URL parameter.
- Spring MVC supports content negotiation via `ContentNegotiationManager`. Configuration can be done in `WebMvcConfigurer`.
- `ContentNegotiatingViewResolver` can choose the appropriate view based on requested media type.
- For REST controllers returning `ResponseEntity`, `HttpMessageConverter` selection is based on `Accept` header and `produces` attribute.
- Strategies: path extension (deprecated in favor of Accept header), parameter (e.g., `?format=json`), and header.

### REST principles
- Representational State Transfer (REST) is an architectural style for designing networked applications.
- Key principles: client-server, stateless, cacheable, layered system, uniform interface, and optionally code on demand.
- Uniform interface includes resource identification (URIs), resource manipulation through representations, self-descriptive messages, and hypermedia as the engine of application state (HATEOAS).
- In Spring, RESTful services are built using `@RestController`, HTTP methods as CRUD operations, and appropriate status codes.
- REST APIs should be stateless; each request contains all necessary information.

## Spring Boot

### Auto-configuration
- Spring Boot auto-configuration automatically configures Spring beans based on classpath dependencies, property settings, and other conditions.
- It is implemented via `@EnableAutoConfiguration` (included in `@SpringBootApplication`), which triggers `AutoConfigurationImportSelector`.
- Auto-configuration classes are listed in `META-INF/spring.factories` (Spring Boot 2.7+) or `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (Spring Boot 3+).
- Each auto-configuration is conditional (using `@ConditionalOnClass`, `@ConditionalOnMissingBean`, etc.) to avoid conflicting with user-defined beans.
- Common auto-configurations: `DataSourceAutoConfiguration`, `JpaRepositoriesAutoConfiguration`, `WebMvcAutoConfiguration`.

### Spring Boot starter dependencies
- Starters are convenient dependency descriptors that bundle a set of libraries for a particular feature.
- Example: `spring-boot-starter-web` includes dependencies for building web applications (Spring MVC, Tomcat, Jackson).
- Starters simplify Maven/Gradle configuration by providing curated sets of dependencies with compatible versions.
- They also often include auto-configuration for the feature.
- Custom starters can be created for internal reusable modules.

### @SpringBootApplication
- `@SpringBootApplication` is a composite annotation combining `@Configuration`, `@EnableAutoConfiguration`, and `@ComponentScan` (with default attributes).
- It marks the main class of a Spring Boot application. The `main` method typically runs `SpringApplication.run(Class, args)`.
- The annotation enables auto-configuration and component scanning from the main class's package and sub-packages.
- It can be customized to exclude auto-configuration classes or scan specific packages.
- In Spring Boot 2.7+, `@SpringBootApplication` also includes `@PropertySource` scanning.

### Embedded servers
- Spring Boot supports embedded servlet containers (Tomcat, Jetty, Undertow) and reactive servers (Netty).
- They are included via starter dependencies (e.g., `spring-boot-starter-tomcat`). The application runs as a standalone JAR with embedded server.
- Configuration of embedded server (port, context path, SSL) is done via properties (e.g., `server.port=8081`).
- Programmatic customization can be done via `WebServerFactoryCustomizer`.
- Embedded servers simplify deployment and development; no need to deploy WAR files to external containers.

### External configuration
- Spring Boot allows externalizing configuration via properties files, YAML files, environment variables, command-line arguments, etc.
- The `@ConfigurationProperties` annotation binds external properties to structured Java objects (type-safe).
- Property sources are loaded in a defined order: command-line arguments, JNDI attributes, system properties, environment variables, `application.properties`/YAML, etc.
- `@Value` can also inject individual properties.
- Profiles allow different configurations per environment (e.g., `application-dev.properties`).

### application.yml vs properties
- Both formats are supported for configuration. `.properties` is key=value, while `.yml` (YAML) is hierarchical and more readable for complex structures.
- YAML supports lists, maps, and nested properties more concisely. For example:
  ```yaml
  spring:
    datasource:
      url: jdbc:mysql://localhost/test
  ```
- YAML is parsed via SnakeYAML; properties via `java.util.Properties`.
- In a single project, you can use either; Spring Boot loads both if present, with properties having higher precedence if both define same key (order depends on file name).
- YAML cannot be loaded via `@PropertySource` (only properties). For YAML, use `@ConfigurationProperties` or bootstrap.

### Profiles
- Profiles in Spring Boot allow environment-specific configuration. They are defined using `spring.profiles.active` or `spring.profiles.include`.
- Profile-specific property files: `application-{profile}.properties` (or YAML) override the default ones.
- In YAML, profiles can be represented using document separators and `spring.config.activate.on-profile`.
- Beans can be conditionally registered based on profile using `@Profile`.
- Profiles are also used for test configuration via `@ActiveProfiles`.

### Actuator
- Spring Boot Actuator provides production-ready features for monitoring and managing applications via HTTP endpoints or JMX.
- Common endpoints: `/health`, `/info`, `/metrics`, `/env`, `/beans`, `/mappings`, `/loggers`.
- Endpoints can be exposed and secured. Custom endpoints can be added.
- Actuator integrates with Micrometer for metrics export to monitoring systems (Prometheus, Graphite, etc.).
- Configuration includes enabling/disabling endpoints, exposure (HTTP or JMX), and sensitivity.

### Custom auto-configuration
- Creating custom auto-configuration involves:
  - Annotating a configuration class with `@Configuration` and `@AutoConfigureAfter/Before` for ordering.
  - Using condition annotations (`@ConditionalOnClass`, `@ConditionalOnMissingBean`, etc.) to apply conditionally.
  - Registering the auto-configuration in `META-INF/spring.factories` (or new imports file).
- Typically packaged as a separate starter project.
- Should consider providing configuration properties via `@ConfigurationProperties` for user customization.
- Ensure that auto-configuration does not interfere with user-defined beans.

### Condition annotations
- Spring Boot provides numerous condition annotations for auto-configuration:
  - `@ConditionalOnClass` / `@ConditionalOnMissingClass`: based on classpath presence.
  - `@ConditionalOnBean` / `@ConditionalOnMissingBean`: based on bean existence.
  - `@ConditionalOnProperty`: based on property value (e.g., `my.feature.enabled=true`).
  - `@ConditionalOnResource`: based on resource presence.
  - `@ConditionalOnWebApplication` / `@ConditionalOnNotWebApplication`.
  - `@ConditionalOnExpression`: SpEL expression.
- These annotations enable fine-grained conditional configuration.
- They are processed by `ConditionEvaluator` during context refresh.

### Boot lifecycle
- Spring Boot application lifecycle includes:
  - Initialization: `SpringApplication` run method, prepare environment, create `ApplicationContext`.
  - Starting: publish `ApplicationStartingEvent`, create and prepare context, refresh context (bean initialization).
  - Running: after refresh, publish `ApplicationReadyEvent` (if web server started), application is ready to serve requests.
  - Shutdown: on context close, publish `ContextClosedEvent`, destroy beans, close embedded server.
- `ApplicationRunner` and `CommandLineRunner` beans execute after context refresh but before the application starts serving.
- `@EventListener` can be used to listen to application events.

### DevTools
- Spring Boot DevTools provides development-time features: automatic restart, live reload, property defaults, and remote debugging support.
- Automatic restart triggers when classpath files change; uses two classloaders to speed up restart (base classloader for dependencies, restart classloader for application code).
- LiveReload server can trigger browser refresh when resources change.
- DevTools also caches configurations (like template cache) disabled by default for development.
- It is recommended to exclude DevTools in production builds.

### Boot logging configuration
- Spring Boot uses Commons Logging but provides default configuration for Logback, Log4j2, and Java Util Logging via starters.
- Default logging level is INFO, output includes date, thread, level, logger, message.
- Configuration can be done via properties: `logging.level.com.example=DEBUG`, `logging.file.name=app.log`, `logging.pattern.console`.
- For advanced customization, you can provide your own log configuration file (logback-spring.xml, log4j2-spring.xml) leveraging Spring Boot's placeholders and profile support.
- Logging groups can be defined to set levels for multiple loggers at once.

## Spring Security

### Authentication vs Authorization
- **Authentication** is the process of verifying identity (who the user is). In Spring Security, it typically involves username/password, tokens, etc.
- **Authorization** determines what an authenticated user is allowed to do (access resources, perform actions). Based on roles/permissions.
- Spring Security manages authentication via `AuthenticationManager` and authorization via access control rules (e.g., `hasRole()`).
- The result of authentication is an `Authentication` object stored in the `SecurityContext`.
- Authorization can be at URL level (HttpSecurity) or method level (`@PreAuthorize`, `@Secured`).

### Security filter chain
- Spring Security's core is a chain of servlet filters (`FilterChainProxy`). Each filter handles specific security concerns (authentication, authorization, CSRF, etc.).
- Common filters: `UsernamePasswordAuthenticationFilter` (form login), `BasicAuthenticationFilter` (HTTP Basic), `ExceptionTranslationFilter` (handles access denied), `FilterSecurityInterceptor` (last, performs authorization).
- Filters are ordered; you can add custom filters at specific positions.
- The filter chain is configured in `SecurityFilterChain` bean (or legacy `WebSecurityConfigurerAdapter`).
- Each request passes through the chain; if authentication or authorization fails, appropriate responses (login page, 403) are returned.

### WebSecurityConfigurerAdapter (legacy)
- `WebSecurityConfigurerAdapter` was the traditional way to configure web security in Spring Security up to version 5.7.
- By extending it and overriding `configure(HttpSecurity http)` and `configure(AuthenticationManagerBuilder auth)`, you could define security rules.
- It is now deprecated in favor of component-based configuration using `SecurityFilterChain` and `UserDetailsService` beans.
- The new approach uses `@Bean` definitions for `SecurityFilterChain`, `AuthenticationManager`, `UserDetailsService`, etc., giving more flexibility.

### SecurityFilterChain (modern)
- From Spring Security 5.8+, the recommended approach is to define `SecurityFilterChain` beans.
- A `SecurityFilterChain` bean is a `@Bean` of type `SecurityFilterChain` that configures `HttpSecurity` and returns the built chain.
- Multiple `SecurityFilterChain` beans can be defined, ordered by `@Order`, to handle different URL patterns (e.g., public vs private).
- This component-based style is more flexible and easier to test.
- Example: `@Bean public SecurityFilterChain filterChain(HttpSecurity http) throws Exception { ... return http.build(); }`

### OAuth2
- OAuth2 is an authorization framework that allows third-party applications to obtain limited access to user accounts on an HTTP service.
- Spring Security provides OAuth2 client and resource server support.
- OAuth2 client can be used for login with providers (Google, GitHub) via `OAuth2LoginAuthenticationFilter`.
- Resource server validates JWT or opaque tokens, extracting authentication from the `Authorization` header.
- Spring Boot simplifies configuration with properties like `spring.security.oauth2.client.registration.*`.
- OAuth2 integration includes support for PKCE, JWT, and introspection.

### JWT
- JSON Web Tokens (JWT) are a compact, URL-safe means of representing claims between parties. Often used for stateless authentication.
- In Spring Security, JWT can be used with OAuth2 resource server or custom authentication filters.
- A JWT typically contains header, payload, signature. It can be self-contained (no server-side session).
- Spring Security provides `JwtDecoder` to validate and parse JWT tokens (using Nimbus JOSE).
- Common pattern: issue JWT upon login, then include it in `Authorization: Bearer` header for subsequent requests.

### CSRF
- Cross-Site Request Forgery (CSRF) is an attack that forces authenticated users to execute unwanted actions.
- Spring Security provides CSRF protection enabled by default for state-changing operations (POST, PUT, DELETE) in traditional web applications.
- CSRF protection works by synchronizing a token (generated server-side) with the client; the token must be included in requests (e.g., as a header or form parameter).
- For stateless REST APIs (using token-based auth), CSRF protection is usually disabled because there's no session to steal.
- CSRF configuration can be customized or disabled via `http.csrf().disable()` (not recommended for browser-based apps).

### CORS
- CORS (Cross-Origin Resource Sharing) is a security mechanism that allows web pages from one domain to access resources on another.
- Spring Security integrates with Spring MVC's CORS support. You can configure CORS globally via `CorsConfigurationSource` bean or in `HttpSecurity` with `http.cors()`.
- `CorsConfigurationSource` defines allowed origins, methods, headers, etc.
- CORS must be configured to allow cross-origin requests from legitimate frontend applications.
- It is separate from CSRF but often used together.

### Method-level security
- Method-level security enables authorization at the method invocation level using annotations.
- Enable it with `@EnableGlobalMethodSecurity(prePostEnabled = true, securedEnabled = true, jsr250Enabled = true)`.
- Common annotations: `@PreAuthorize` (SpEL expression), `@PostAuthorize`, `@Secured` (roles), `@RolesAllowed` (JSR-250).
- These annotations work via AOP proxies, so self-invocation issues apply.
- Method security can be applied to any Spring bean, not just web controllers.

### Role-based access control
- Roles are granted to users (e.g., ROLE_ADMIN, ROLE_USER). Spring Security uses roles in authorization expressions.
- Typically, roles are prefixed with `ROLE_` (configurable). In expressions, `hasRole('ADMIN')` checks for `ROLE_ADMIN`.
- Authorities are more granular permissions (e.g., `READ_PRIVILEGES`). `hasAuthority('READ_PRIVILEGES')` checks exact string.
- Role hierarchy can be defined to imply roles (e.g., ADMIN includes USER). Configure with `RoleHierarchy`.
- Roles are assigned via `UserDetailsService` (e.g., from database).

### Custom authentication provider
- `AuthenticationProvider` is an interface for performing authentication. Spring Security has several built-in providers (DaoAuthenticationProvider, LdapAuthenticationProvider).
- To implement custom authentication (e.g., against legacy system), you can create a class implementing `AuthenticationProvider` and override `authenticate()` and `supports()`.
- Register the custom provider with `AuthenticationManagerBuilder` (or as a bean in modern config).
- The provider can throw `AuthenticationException` on failure.
- Useful for two-factor authentication, external authentication services, etc.

### Password encoders
- Password encoding is critical for securely storing passwords. Spring Security provides `PasswordEncoder` interface.
- Common implementations: `BCryptPasswordEncoder` (bcrypt hashing, strong), `Argon2PasswordEncoder`, `Pbkdf2PasswordEncoder`, and `SCryptPasswordEncoder`.
- `NoOpPasswordEncoder` is plain text (for legacy only, never use in production).
- The `PasswordEncoder` should be registered as a bean; `DaoAuthenticationProvider` uses it to verify passwords.
- Modern practice uses `BCryptPasswordEncoder` with a strength (work factor) appropriate for your environment.

### Stateless security
- Stateless security means the server does not maintain session state. Each request must contain authentication information (e.g., JWT token).
- In Spring Security, you configure `sessionManagement().sessionCreationPolicy(SessionCreationPolicy.STATELESS)`.
- With stateless, there is no `HttpSession`; CSRF protection is typically disabled.
- Authentication is re-established on each request (e.g., via JWT filter that sets `SecurityContext`).
- Common for REST APIs and microservices.

### SSO integration
- Single Sign-On (SSO) allows users to log in once and access multiple applications.
- Spring Security can integrate with SSO providers using OAuth2, SAML, or CAS.
- For OAuth2 SSO, use `spring-security-oauth2-client` with login configuration. The provider handles authentication and returns tokens.
- For SAML, Spring Security SAML extension can be used.
- With Spring Boot, OAuth2 login can be configured easily with properties and `@EnableOAuth2Sso` (deprecated in favor of OAuth2 client).
- SSO typically involves redirects to identity provider (IdP) and callback handling.

## Spring REST & Web

### RESTful API design
- RESTful APIs are designed around resources identified by URIs, using HTTP methods (GET, POST, PUT, DELETE) to perform operations.
- Best practices: use nouns for resource names (e.g., `/users`), avoid verbs in URIs, use plural for collections.
- HTTP status codes indicate result (200 OK, 201 Created, 400 Bad Request, 404 Not Found, 500 Internal Server Error).
- Use consistent error response format. Version APIs (via URI or Accept header).
- Support filtering, sorting, pagination via query parameters.

### HATEOAS
- HATEOAS (Hypermedia as the Engine of Application State) is a constraint of REST where responses include links to related resources, guiding client interactions.
- Spring HATEOAS provides `EntityModel` and `CollectionModel` to wrap resources with links.
- `Link` objects can be created using `WebMvcLinkBuilder` to generate URIs from controller methods.
- Clients can navigate the API dynamically via provided links, reducing coupling.
- Useful for building discoverable APIs.

### Error handling strategies
- Consistent error handling is crucial for APIs. Common strategy: return a structured error response with fields like timestamp, status, error, message, path.
- Use `@ControllerAdvice` or `@RestControllerAdvice` with `@ExceptionHandler` to catch exceptions and return appropriate `ResponseEntity`.
- For validation errors, return details about which fields failed.
- Avoid exposing stack traces; log them server-side.
- Define custom exceptions for business logic errors.

### API versioning
- API versioning allows multiple versions to coexist, enabling smooth upgrades for clients.
- Strategies: URI path versioning (`/v1/users`), query parameter (`?version=1`), header (`Accept: application/vnd.company.v1+json`), or content negotiation.
- Each has pros/cons: URI is simple and cacheable; header is cleaner but less discoverable.
- In Spring, you can use `@RequestMapping` with path variable or custom version mapping.
- Consider using URL versioning for public APIs for simplicity.

### Rate limiting
- Rate limiting restricts the number of requests a client can make in a given time window to prevent abuse and ensure fair usage.
- Not built into Spring by default, but can be implemented with filters/interceptors, or use libraries like Bucket4j, Resilience4j (RateLimiter), or Spring Cloud Gateway.
- Rate limiting can be based on IP, API key, user, etc.
- It often returns HTTP status 429 (Too Many Requests) with `Retry-After` header.
- Can be integrated with distributed caching (Redis) for cluster-wide limits.

### Global exception handling
- Global exception handling ensures consistent error responses across all controllers.
- In Spring MVC/REST, use `@ControllerAdvice` (or `@RestControllerAdvice`) with `@ExceptionHandler` methods.
- You can handle specific exceptions (e.g., `EntityNotFoundException`) and also fallback for `Exception`.
- Use `ResponseEntityExceptionHandler` as base class to override Spring MVC's default error handling.
- For validation errors, `MethodArgumentNotValidException` and `ConstraintViolationException` can be handled.

### API documentation (OpenAPI)
- OpenAPI (formerly Swagger) is a specification for documenting REST APIs. Springdoc or Springfox can generate OpenAPI docs from code.
- Annotations like `@Operation`, `@ApiResponse` describe endpoints, parameters, and responses.
- The generated JSON/YAML can be used by Swagger UI for interactive documentation.
- Spring Boot integration with springdoc-openapi is common: add dependency, then access `/swagger-ui.html`.
- Helps keep documentation in sync with code.

### Request interceptors
- `HandlerInterceptor` in Spring MVC allows pre-processing and post-processing of requests. Can be used for logging, modifying model, or checking custom headers.
- Pre-handle (before controller), post-handle (after controller but before view), after-completion (after view rendering).
- Configure interceptors by overriding `addInterceptors` in `WebMvcConfigurer`.
- For REST APIs, interceptors can log request/response or validate tokens.
- Unlike filters, interceptors have access to Spring MVC components (e.g., `ModelAndView`).

### Response filters
- Filters (javax.servlet.Filter) operate before request reaches DispatcherServlet and after response is generated. Can be used for cross-cutting concerns like compression, authentication, CORS.
- In Spring Boot, you can register filters as `@Bean` (implementing `Filter`) and they are auto-added to the filter chain.
- For finer control, use `FilterRegistrationBean`.
- Response can be wrapped using `HttpServletResponseWrapper` to modify output.
- Used for adding headers, gzip compression, etc.

### Idempotency
- Idempotent operations can be called multiple times without different outcomes. GET, PUT, DELETE are idempotent; POST is not.
- For APIs, ensuring idempotency for non-idempotent methods (e.g., POST) can be achieved by client sending an idempotency key header. Server uses this key to detect duplicate requests.
- This prevents duplicate charges or resource creation.
- Implementation: store idempotency key and result in a cache/database for a certain time.
- In Spring, you can create an interceptor or use AOP to handle idempotency checks.

### HTTP status codes best practices
- Use appropriate status codes to convey result clearly:
  - 200 OK: successful GET, PUT, PATCH, DELETE.
  - 201 Created: successful POST creating resource. Include Location header.
  - 204 No Content: successful request with no response body (e.g., DELETE).
  - 400 Bad Request: malformed syntax or validation error.
  - 401 Unauthorized: missing or invalid authentication.
  - 403 Forbidden: authenticated but insufficient permissions.
  - 404 Not Found: resource doesn't exist.
  - 409 Conflict: state conflict (e.g., duplicate entry).
  - 422 Unprocessable Entity: validation failed (often used for semantic errors).
  - 429 Too Many Requests: rate limit exceeded.
  - 500 Internal Server Error: unexpected server error.
- Avoid overusing 200 for errors; use appropriate 4xx/5xx.

## Spring Reactive (WebFlux)

### Reactive programming basics
- Reactive programming is a declarative programming paradigm concerned with data streams and propagation of change. It enables building asynchronous, non-blocking, event-driven applications.
- Core principles: responsive, resilient, elastic, and message-driven (Reactive Manifesto).
- In Java, Project Reactor provides `Flux` (0..N elements) and `Mono` (0..1 element) types, implementing Reactive Streams specification.
- Reactive programming allows efficient resource utilization by avoiding thread blocking; a small number of threads can handle many concurrent requests.
- Backpressure ensures that consumers can control the rate of data emission to avoid being overwhelmed.

### Mono vs Flux
- `Mono<T>` represents a reactive stream that emits at most one element (or empty/error). Useful for single-value async operations (e.g., HTTP request, database fetch).
- `Flux<T>` represents a stream that emits 0 to N elements (potentially infinite). Used for collections, streams of events.
- Both are publishers in Reactor; they can be transformed using operators (map, flatMap, filter, etc.).
- They are lazy: nothing happens until you subscribe.
- Subscribing triggers the actual execution; you can consume data via lambda callbacks.

### WebFlux vs MVC
- Spring MVC is built on Servlet API, blocking I/O (thread-per-request model). Suitable for traditional applications.
- Spring WebFlux is fully non-blocking, supports reactive backpressure, and runs on Netty (default), Undertow, or Servlet 3.1+ containers. It can be used with functional endpoints or annotated controllers.
- WebFlux can coexist with MVC in same application but not recommended due to different threading models.
- WebFlux is a good fit for applications with high concurrency, streaming data, or microservices using reactive clients.
- MVC may be simpler for CRUD applications with low to moderate traffic.

### Netty server
- Netty is an asynchronous event-driven network framework. Spring WebFlux uses Netty as its default embedded server.
- Netty provides high performance and scalability for non-blocking I/O.
- In Spring Boot, `spring-boot-starter-webflux` includes Netty and Reactor Netty.
- Netty can be configured via properties (`server.netty.*`) or programmatically.
- It also supports HTTP/2, WebSockets, and other protocols.

### Backpressure
- Backpressure is a mechanism to handle flow control in reactive streams, ensuring that a fast producer doesn't overwhelm a slow consumer.
- In Reactor, backpressure is built-in: subscribers request a certain amount of data, and publishers respect that demand.
- Operators like `onBackpressureBuffer`, `onBackpressureDrop`, `onBackpressureLatest` allow tuning behavior when demand is insufficient.
- Backpressure is essential for building resilient systems that can handle load spikes gracefully.
- It propagates upstream, allowing the source to slow down or buffer.

### Reactive streams
- Reactive Streams is a specification for asynchronous stream processing with non-blocking backpressure. It defines four interfaces: `Publisher`, `Subscriber`, `Subscription`, and `Processor`.
- Project Reactor implements these interfaces; other implementations include RxJava, Akka Streams.
- The spec ensures interoperability between different reactive libraries.
- Spring WebFlux uses Reactor internally but can work with other reactive libraries via adapters.
- Reactive Streams enables building end-to-end reactive systems (from database to client).

### Functional endpoints
- In addition to annotated controllers, WebFlux supports functional programming model using `RouterFunction` and `HandlerFunction`.
- `RouterFunction` maps requests to handler functions (similar to routing in Express.js).
- Handler functions take `ServerRequest` and return `Mono<ServerResponse>`.
- This approach is more composable and often used for simpler, more testable routes.
- Example: `RouterFunctions.route().GET("/hello", request -> ok().bodyValue("Hello"))`.
- Functional endpoints can be combined and nested.

### Reactive data access
- Spring Data provides reactive repositories for various databases (MongoDB, Cassandra, Redis, Couchbase) via reactive drivers.
- Reactive repositories return `Mono` or `Flux` and are non-blocking.
- R2DBC (Reactive Relational Database Connectivity) allows reactive access to SQL databases (PostgreSQL, MySQL, H2).
- Spring Data R2DBC offers `ReactiveCrudRepository`.
- For JPA, there is no reactive standard; alternatives are Hibernate Reactive (experimental) or using R2DBC directly.

### Thread model differences
- MVC uses a thread-per-request model: each request occupies a thread from the container pool until response is generated (blocking I/O).
- WebFlux uses an event-loop model with a small number of threads (typically number of CPU cores). Requests are handled asynchronously; threads are never blocked.
- In WebFlux, long-running operations must be non-blocking; any blocking operation (e.g., JDBC call) will hurt performance and should be offloaded to a separate scheduler (e.g., `publishOn` with bounded elastic scheduler).
- This model allows higher concurrency with fewer resources.
- Developers must be mindful to avoid blocking in reactive code.

### Blocking vs non-blocking
- Blocking code causes the current thread to wait (sleep) until the operation completes, wasting resources.
- Non-blocking code initiates an operation and immediately returns, allowing the thread to do other work. Completion is signaled via callbacks or reactive types.
- In WebFlux, all components (web layer, data access) should be non-blocking to realize benefits.
- Mixing blocking code in a reactive pipeline (e.g., calling a blocking API) can be isolated using `Schedulers.boundedElastic()` to avoid starving event loop.
- True end-to-end non-blocking requires reactive drivers for databases, HTTP clients (WebClient), etc.

## Spring Cloud

### Microservices fundamentals
- Microservices architecture structures an application as a collection of loosely coupled, independently deployable services.
- Each service is focused on a business capability, owns its data, and communicates via lightweight protocols (HTTP/REST, messaging).
- Benefits: scalability, resilience, technology heterogeneity, and faster deployments.
- Challenges: distributed systems complexity, network latency, service discovery, configuration management, observability.
- Spring Cloud provides tools to address these challenges.

### Service discovery (Eureka)
- Service discovery allows services to find each other without hardcoding host/port. Eureka is Netflix's service discovery server.
- Services register themselves with Eureka server at startup and periodically send heartbeats. Clients query Eureka to locate service instances.
- Spring Cloud Netflix Eureka provides `@EnableEurekaServer` for server and `@EnableDiscoveryClient` for clients (auto-configured).
- Eureka client caches registry information locally for resilience.
- Alternatives: Consul, Zookeeper, Kubernetes native discovery.

### Config server
- Spring Cloud Config Server provides externalized configuration for distributed systems. It serves configuration from Git, file system, or Vault.
- Config clients fetch configuration at startup and can optionally refresh without restart (using `/refresh` endpoint or Spring Cloud Bus).
- Configuration files are organized by application name and profile (e.g., `myapp-dev.properties`).
- Config server can be made highly available using backend storage (Git) and multiple instances.
- It integrates with Spring Boot's `@RefreshScope` for dynamic bean updates.

### API Gateway
- API Gateway is a single entry point for all clients, routing requests to appropriate microservices. It can handle cross-cutting concerns: authentication, logging, rate limiting, load balancing.
- Spring Cloud Gateway (built on WebFlux) is the recommended gateway (replaced Netflix Zuul 1.x).
- It uses route definitions with predicates and filters. Can be configured via Java DSL or properties.
- Gateway integrates with service discovery to dynamically route to service instances.
- It also supports WebSocket, resilience patterns, and request/response transformation.

### Load balancing
- Client-side load balancing: Spring Cloud LoadBalancer (replacing Netflix Ribbon) distributes requests among service instances obtained from discovery.
- `@LoadBalanced` on `RestTemplate` or `WebClient.Builder` enables load-balanced HTTP calls.
- Load balancer can use algorithms like round-robin, random, or custom.
- Server-side load balancing: API Gateway can also load balance across instances.
- Load balancing improves resilience and scalability.

### Circuit breaker
- Circuit breaker pattern prevents cascading failures by stopping calls to a failing service. After a threshold of failures, the circuit opens and subsequent calls fail fast (or fallback).
- Spring Cloud Circuit Breaker provides an abstraction over Resilience4j, Hystrix (deprecated), and others.
- With Resilience4j, you annotate methods with `@CircuitBreaker`, `@Retry`, `@Bulkhead`, `@RateLimiter`, `@TimeLimiter`.
- Circuit breaker monitors failures and transitions between closed, open, half-open states.
- Fallback methods can be specified to provide degraded functionality.

### Resilience4j
- Resilience4j is a lightweight fault tolerance library inspired by Hystrix but designed for Java 8 and functional programming.
- It provides modules: circuit breaker, rate limiter, retry, bulkhead, time limiter, and cache.
- Configuration can be done via properties or programmatically.
- In Spring Boot, `spring-cloud-starter-circuitbreaker-resilience4j` integrates it with `@CircuitBreaker` annotation and `CircuitBreakerFactory`.
- Resilience4j promotes functional composition and can be used with reactive types (Mono/Flux).

### Distributed tracing
- Distributed tracing tracks requests across multiple services, helping to diagnose latency issues and understand system behavior.
- Spring Cloud Sleuth (deprecated in favor of Micrometer Tracing) adds trace and span IDs to logs and propagates them across service calls.
- It integrates with Zipkin or Jaeger for visualization.
- Trace data includes timings, service names, and custom tags.
- Micrometer Tracing is the new standard, integrated with Brave or OpenTelemetry.

### Spring Cloud Stream
- Spring Cloud Stream is a framework for building message-driven microservices. It abstracts away the specifics of messaging middleware (RabbitMQ, Kafka, etc.).
- It provides `@EnableBinding` (legacy) and functional programming model (`java.util.function`). With functional model, you define `Supplier`, `Function`, `Consumer` beans.
- Binders connect to message brokers and handle serialization/deserialization.
- It supports partitioning, consumer groups, and error handling.
- Useful for event-driven architectures and integration.

### Messaging integration
- Spring Cloud integrates with various messaging systems:
  - RabbitMQ: via AMQP binder.
  - Apache Kafka: via Kafka binder.
  - Amazon SQS, Google PubSub, etc., via partner binders.
- Messaging enables asynchronous communication between microservices.
- Patterns: publish-subscribe, message routing, event sourcing, CQRS.
- Spring Integration can also be used for more complex integration flows.

### Sleuth
- Spring Cloud Sleuth (now deprecated in favor of Micrometer Tracing) provided distributed tracing capabilities.
- It automatically added trace and span IDs to logs and MDC.
- Supported HTTP and messaging headers for propagation.
- Integrated with Zipkin for reporting.
- With Micrometer Tracing, you use `Observation` API and configure reporters.

### Feign client
- Feign is a declarative HTTP client developed by Netflix. Spring Cloud OpenFeign integrates it with Spring Boot.
- With `@FeignClient`, you define a Java interface with `@RequestMapping` methods, and Feign implements the HTTP calls.
- It integrates with service discovery (Ribbon/LoadBalancer) and can include fallbacks (with Hystrix or Resilience4j).
- Feign simplifies writing REST clients and reduces boilerplate.
- It supports request/response compression, logging, and custom configuration.

### Cloud-native patterns
- Cloud-native patterns are architectural designs that leverage cloud platforms to build resilient, scalable, and manageable applications.
- Examples: externalized configuration, service discovery, circuit breaker, API gateway, distributed tracing, health checks, graceful shutdown, and blue-green deployments.
- Spring Cloud provides implementations for many of these patterns.
- These patterns enable applications to be elastic and fault-tolerant.
- They are often used with container orchestration (Kubernetes) for deployment and management.

## Spring Integration & Messaging

### Message channels
- Message channels are pipes that connect message endpoints in Spring Integration. They carry messages from one component to another.
- Types: `DirectChannel` (point-to-point, synchronous), `PublishSubscribeChannel` (broadcast), `QueueChannel` (buffered), `PriorityChannel`, `RendezvousChannel`.
- Channels can be pollable (for queue) or subscribable (for direct).
- Message consists of payload and headers.
- Channel adapters and gateways connect to external systems.

### Integration flows
- Integration flows define the path of messages through various endpoints (transformers, filters, routers, splitters, aggregators, service activators).
- In Spring Integration, you can define flows using Java DSL (IntegrationFlows), XML, or annotations.
- Example: `IntegrationFlows.from("inputChannel").handle("service", "method").channel("outputChannel").get()`
- Flows can include error handling, transformation, and routing.
- They enable building EIP (Enterprise Integration Patterns) applications.

### Message brokers
- Message brokers (RabbitMQ, Apache Kafka, ActiveMQ) act as intermediaries for message exchange between applications.
- They support different messaging models: point-to-point (queues) and publish-subscribe (topics).
- Spring Integration provides channel adapters for popular brokers, enabling seamless integration.
- They offer durability, persistence, and reliability.
- In microservices, brokers are often used for event-driven communication.

### Kafka integration
- Apache Kafka is a distributed streaming platform. Spring provides Spring for Apache Kafka and Spring Integration Kafka adapters.
- `KafkaTemplate` for sending messages, `@KafkaListener` for consuming.
- Spring Cloud Stream can also bind to Kafka.
- Integration supports exactly-once semantics, transactions, and error handling.
- Kafka is used for event sourcing, stream processing, and log aggregation.

### AMQP
- Advanced Message Queuing Protocol (AMQP) is an open standard for messaging. RabbitMQ is a popular AMQP broker.
- Spring AMQP provides `RabbitTemplate` for sending, `@RabbitListener` for receiving.
- Spring Integration has AMQP channel adapters.
- AMQP features: exchanges, queues, bindings, routing keys.
- It's ideal for reliable, transactional messaging.

### Asynchronous processing
- Asynchronous processing allows tasks to run in the background without blocking the main thread, improving responsiveness.
- In Spring, `@Async` annotation on methods enables asynchronous execution using a `TaskExecutor`.
- Return values can be `Future`, `CompletableFuture`, or `ListenableFuture`.
- For reactive applications, asynchronous is built-in via reactive types.
- Asynchronous messaging (via JMS, Kafka) also decouples components.

### Event-driven systems
- Event-driven architecture (EDA) is based on the production, detection, consumption of events. Services communicate via events rather than direct calls.
- Events represent state changes (e.g., "OrderCreated"). They are published to a broker, and interested services react.
- Spring provides `ApplicationEvent` and `@EventListener` for internal events within the same JVM.
- For distributed events, use Spring Cloud Stream or messaging integration.
- EDA enables loose coupling, scalability, and real-time reactions.

### SAGA pattern
- SAGA is a pattern for managing distributed transactions across multiple microservices. Instead of a single ACID transaction, a SAGA consists of a series of local transactions, each with a compensating action for rollback.
- Two main approaches: choreography (each service listens for events and publishes next) and orchestration (central coordinator).
- Spring can implement SAGAs using event-driven communication (e.g., with Kafka) and compensating transactions.
- Libraries like Axon Framework or Eventuate Tram provide support.
- SAGA ensures data consistency in distributed systems without distributed transactions (which are hard).

### Retry mechanisms
- Retry mechanisms improve resilience by automatically retrying failed operations (e.g., network glitches).
- Spring Retry provides `@Retryable` annotation and `RetryTemplate` for declarative retries.
- Features: backoff policies (fixed, exponential), retry listeners, and stateful retry.
- Integration with Spring's `@Transactional` can be tricky (requires careful configuration).
- In messaging, retries can be built into listeners with error handlers and DLQs.

## Testing in Spring

### @SpringBootTest
- `@SpringBootTest` loads the full application context for integration tests. It starts the embedded server (if web environment is enabled) and autoconfigures.
- Can specify web environment: `WebEnvironment.MOCK` (use mock servlet), `RANDOM_PORT` (real server with random port), `DEFINED_PORT`.
- It searches for `@SpringBootConfiguration` (main class) to bootstrap context.
- Use `@TestConfiguration` to override or add beans for testing.
- It's slower than sliced tests, so should be used for full integration scenarios.

### @WebMvcTest
- `@WebMvcTest` is a sliced test that focuses only on web layer (controllers, filters, interceptors). It does not load service/repository beans.
- It auto-configures Spring MVC infrastructure and can be used with `@MockBean` to mock dependencies.
- Typically used for unit testing controllers in isolation.
- It also configures `MockMvc` for performing requests and assertions.
- Useful for testing controller logic, validation, and serialization.

### @DataJpaTest
- `@DataJpaTest` slices the context for JPA repositories. It configures an in-memory database (by default), EntityManager, and Spring Data JPA.
- It loads only repository-related beans; services are not loaded.
- By default, tests are transactional and roll back at the end.
- You can use `@AutoConfigureTestDatabase` to replace the datasource with an embedded one.
- Useful for testing repository queries and JPA mappings.

### Test slicing
- Test slicing annotations load a subset of the application context to speed up tests and focus on a specific layer.
- Provided slices: `@WebMvcTest`, `@DataJpaTest`, `@JsonTest`, `@RestClientTest`, `@DataRedisTest`, `@DataMongoTest`, etc.
- Each slice auto-configures relevant components and excludes others.
- They can be combined with `@MockBean` for dependencies.
- Slices are recommended for unit/integration tests of specific layers.

### MockMvc
- `MockMvc` is a Spring MVC test framework that allows performing HTTP requests against a controller without starting a server.
- It is configured via `@AutoConfigureMockMvc` or manually in tests.
- You can perform requests (get, post) with parameters, headers, and body, and assert responses (status, content, JSON path).
- Works with `@WebMvcTest` or `@SpringBootTest` with web environment MOCK.
- It is fast and suitable for testing controller logic, security, and serialization.

### TestRestTemplate
- `TestRestTemplate` is a convenience subclass of `RestTemplate` for integration tests when using a running server.
- It is auto-configured in `@SpringBootTest` with web environment RANDOM_PORT or DEFINED_PORT.
- It handles authentication, error responses, and has methods for common HTTP operations.
- Can be used to test full REST endpoints with actual HTTP.
- For reactive, there is `WebTestClient`.

### Mockito integration
- Mockito is a mocking framework often used in Spring tests to mock dependencies.
- `@MockBean` adds Mockito mocks to the Spring application context, replacing any existing beans of the same type.
- `@SpyBean` creates a spy of an existing bean.
- Mockito can be used with `@WebMvcTest`, `@DataJpaTest`, or `@SpringBootTest`.
- Useful for isolating the class under test and verifying interactions.
- Avoid over-mocking; prefer slices with real dependencies where possible.

### Transactional tests
- `@Transactional` on test methods makes them run within a transaction that rolls back by default after the test.
- This ensures tests are isolated and database is not polluted.
- For `@DataJpaTest`, tests are already `@Transactional`.
- For `@SpringBootTest`, you can annotate with `@Transactional` to roll back changes.
- However, if the test starts a real server (web environment), transactions may not be tied to the HTTP request thread, leading to unexpected behavior. Use with caution.

### Embedded database
- Spring Boot can auto-configure an embedded database (H2, HSQL, Derby) for testing when an in-memory database is detected on classpath.
- `@DataJpaTest` uses an embedded database by default.
- Embedded databases are fast and isolated, ideal for repository tests.
- You can also configure a specific embedded database via properties.
- For production-like testing, consider Testcontainers.

### Testcontainers
- Testcontainers is a Java library that provides lightweight, disposable containers (Docker) for integration testing.
- It allows you to run real databases, message brokers, or any service in a container during tests.
- Spring Boot integrates with Testcontainers via `@Testcontainers` and `@Container` annotations.
- You can dynamically set properties (e.g., JDBC URL) from the container.
- Useful for testing with the same database as production (e.g., PostgreSQL) instead of H2.

### Integration testing strategy
- Integration testing ensures that different parts of the application work together correctly.
- A good strategy includes:
  - Unit tests for individual classes (with mocks).
  - Sliced tests for web, data, etc.
  - Integration tests for specific modules (e.g., service with real DB using Testcontainers).
  - End-to-end tests for critical user journeys (maybe using `@SpringBootTest` with random port and TestRestTemplate).
- Use profiles to isolate test configuration.
- Aim for fast feedback; use in-memory DB for quick tests, but also include tests against production-like DB.

## Performance & Optimization

### Thread pools
- Thread pools manage a pool of worker threads to execute tasks asynchronously, reducing overhead of thread creation.
- Spring's `TaskExecutor` abstraction: `SimpleAsyncTaskExecutor` (creates new thread each time), `ThreadPoolTaskExecutor` (configurable thread pool).
- Configure via properties: `spring.task.execution.pool.core-size`, `max-size`, `queue-capacity`.
- `@Async` uses a `TaskExecutor` bean (default `SimpleAsyncTaskExecutor` if none defined; better to define a pool).
- Proper sizing depends on task type (CPU-bound vs I/O-bound). Use monitoring to tune.

### Async processing (@Async)
- `@Async` enables asynchronous method execution. The method returns `void` or `Future`/`CompletableFuture`.
- Must be enabled with `@EnableAsync`.
- By default, uses a `SimpleAsyncTaskExecutor` (creates a new thread per task). Better to provide a `TaskExecutor` bean.
- Exceptions in async methods are not propagated to caller; they must be handled by `AsyncUncaughtExceptionHandler`.
- Proxy-based, so self-invocation bypasses async behavior.

### Connection pooling
- Connection pooling reuses database connections to avoid overhead of creating new connections per request.
- Popular pools: HikariCP (default in Spring Boot), Tomcat JDBC, Commons DBCP2.
- HikariCP is lightweight and high-performance. Configuration: `maximumPoolSize`, `minimumIdle`, `connectionTimeout`, `idleTimeout`, `maxLifetime`.
- Proper pool size depends on database capacity, concurrent requests, and workload type.
- Monitor connection pool metrics via Actuator to detect leaks or under-sizing.

### Caching (@Cacheable)
- Caching stores method results to avoid repeated expensive computations.
- Spring provides caching abstraction: `@Cacheable`, `@CacheEvict`, `@CachePut`, `@Caching`, `@CacheConfig`.
- Cache managers: ConcurrentMap (simple), Ehcache, Caffeine, Redis, etc.
- `@Cacheable` checks cache before method execution; if found, returns cached value.
- `@CacheEvict` removes entries.
- Caching is proxy-based, so internal calls won't trigger caching.

### Cache managers
- Cache manager configures how caches are stored and evicted. Spring provides implementations:
  - `ConcurrentMapCacheManager`: in-memory, simple, for development.
  - `CaffeineCacheManager`: high-performance, uses Caffeine library.
  - `EhCacheCacheManager`: for Ehcache 2.x.
  - `JCacheCacheManager`: for JSR-107 compliant caches.
  - `RedisCacheManager`: for Redis-backed distributed cache.
- Cache manager bean defines default expiration, etc.
- Multiple cache managers can co-exist with qualifiers.

### Redis integration
- Redis is an in-memory data store used for caching, session management, pub/sub, etc.
- Spring Data Redis provides `RedisTemplate` and repository support.
- For caching, use `RedisCacheManager` and configure serialization (Jackson2JsonRedisSerializer, etc.).
- Redis can be used as a distributed cache across microservices.
- Session storage with Redis enables stateless apps and session sharing.

### Memory tuning
- JVM memory tuning: heap size (`-Xms`, `-Xmx`), GC tuning, metaspace.
- Spring Boot applications run in a JVM; memory usage includes heap, stack, metaspace, and off-heap (direct buffers).
- Use tools like VisualVM, JConsole, or Actuator `/metrics` to monitor.
- Common issues: memory leaks (unclosed resources, cached data), excessive object creation.
- Tune based on application load and performance testing.

### Profiling
- Profiling helps identify performance bottlenecks (CPU, memory, I/O).
- Tools: JProfiler, YourKit, VisualVM, Java Flight Recorder (JFR).
- Profiling in production may have overhead; use sampling and monitor.
- Look for high CPU methods, excessive GC, thread contention, slow database queries.
- Spring Boot Actuator with Micrometer can expose metrics to monitoring systems (Prometheus, Grafana).

### Monitoring with Actuator
- Spring Boot Actuator provides production-ready endpoints for monitoring: `/health`, `/metrics`, `/env`, `/threaddump`, `/heapdump`, `/loggers`.
- Metrics can be exported to systems like Prometheus, Datadog, etc.
- Custom metrics can be added using `MeterRegistry` (Micrometer).
- Health indicators for database, disk space, etc., can be extended.
- Enable security to protect sensitive endpoints.

### Observability
- Observability is the ability to infer internal state of a system from external outputs: logs, metrics, traces.
- Spring Boot 3+ integrates with Micrometer Tracing and Observation.
- `@Observed` annotation can instrument methods automatically.
- Combine logs (with trace IDs), metrics (request rates, latencies), and traces (distributed) for full observability.
- Essential for debugging and understanding system behavior in production.

## Deployment & DevOps

### Dockerizing Spring Boot
- Containerization with Docker packages application and its dependencies into a container image.
- Steps: create Dockerfile with base image (e.g., openjdk:17-jre-slim), copy JAR, expose port, define entrypoint.
- Use multi-stage builds to keep image small.
- Build with `docker build -t myapp .` and run with `docker run`.
- Use `.dockerignore` to exclude unnecessary files.
- Spring Boot 2.3+ supports layered JARs for efficient Docker image creation (using `spring-boot-jarmode-layertools`).

### Kubernetes deployment
- Kubernetes orchestrates containerized applications, providing scaling, self-healing, rolling updates.
- Deploy Spring Boot apps using Kubernetes manifests (Deployment, Service, ConfigMap, Secret).
- Deployment defines replicas, container image, ports, environment variables.
- Service exposes the app internally or via LoadBalancer/Ingress.
- Use liveness and readiness probes (Spring Boot Actuator endpoints).
- Helm charts can package and manage Kubernetes resources.

### Externalized config
- In cloud-native environments, configuration is externalized to manage different environments without rebuilding.
- Options: environment variables, Kubernetes ConfigMaps/Secrets, Spring Cloud Config Server, or external stores (Vault).
- Spring Boot's `@ConfigurationProperties` binds to external config.
- In Kubernetes, use ConfigMap for non-sensitive config, Secrets for sensitive data.
- Configuration can be reloaded at runtime via `@RefreshScope` and Spring Cloud Bus.

### Health checks
- Health checks allow orchestration systems (Kubernetes) to determine if an application is alive and ready to serve traffic.
- Spring Boot Actuator provides `/actuator/health` endpoint with details (status, components).
- Liveness probe indicates if app is running (restart if dead). Readiness probe indicates if app is ready to accept traffic (stop sending traffic during startup or heavy load).
- Custom health indicators can be added.
- Configure probes in Kubernetes deployment: `livenessProbe` and `readinessProbe`.

### Graceful shutdown
- Graceful shutdown allows in-flight requests to complete before the application shuts down, preventing client errors.
- Spring Boot supports graceful shutdown with `server.shutdown=graceful` and a `grace-period`.
- For embedded servers, it waits for active requests to finish (up to timeout).
- In Kubernetes, when a pod is terminated, it receives SIGTERM, then waits for termination grace period before SIGKILL.
- Combine with health checks: readiness should return failure after SIGTERM to stop new traffic.

### Logging strategy
- Logging strategy for production: use structured logging (JSON) for easier ingestion into log aggregation systems (ELK, Splunk).
- Configure log levels per environment; avoid excessive logging in production.
- Use MDC to include trace IDs, user IDs for correlation.
- Spring Boot supports Logback, Log4j2 with profile-specific configs.
- Consider asynchronous logging to reduce I/O overhead (Logback's AsyncAppender).

### CI/CD integration
- CI/CD pipelines automate build, test, and deployment.
- Typical stages: compile, test (unit, integration), package (JAR), build Docker image, push to registry, deploy to environment.
- Tools: Jenkins, GitLab CI, GitHub Actions, Azure DevOps.
- Use environment-specific configuration (profiles, ConfigMaps) during deployment.
- Implement canary or blue-green deployments for safer releases.

### Rolling deployment
- Rolling deployment updates instances gradually, ensuring no downtime. Kubernetes supports rolling updates by default.
- New pods are created while old ones are terminated, maintaining desired replica count.
- Can control max surge and max unavailable.
- Rollback is automatic if health checks fail during update.
- Ensures application remains available during upgrades.

### Blue-Green deployment
- Blue-green deployment reduces downtime by running two identical environments: blue (current) and green (new).
- After testing green, traffic is switched from blue to green (via load balancer or routing). Rollback is immediate by switching back.
- Requires extra resources (two full environments).
- In Kubernetes, can be implemented using labels and service selectors, or with ingress controllers.
- Ideal for critical applications where zero-downtime is mandatory.

## Advanced / Principal-Level Topics

### Spring internals
- Deep understanding of Spring's core: `ApplicationContext` implementations (ClassPathXml, AnnotationConfig, etc.), bean definition loading, `BeanFactory` hierarchy, `DefaultListableBeanFactory` as the backbone.
- The `BeanDefinition` interface holds metadata; `BeanDefinitionReader` parses config.
- `BeanPostProcessor` and `BeanFactoryPostProcessor` extension points.
- Event handling with `ApplicationEvent` and `ApplicationListener`.
- Resource abstraction (`ResourceLoader`) for accessing resources.

### Bean creation process
- The bean creation process involves:
  1. Instantiation (constructor or factory method)
  2. Populate properties (dependency injection)
  3. Aware interfaces (BeanNameAware, BeanFactoryAware, etc.)
  4. BeanPostProcessor pre-initialization
  5. InitializingBean / init-method / @PostConstruct
  6. BeanPostProcessor post-initialization
- For prototype beans, the process stops after creation; no destroy callbacks unless custom.
- The container caches singleton beans after full initialization.
- Circular dependencies resolved via early exposure (singleton factory) for setter injection.

### Proxy mechanism deep dive
- Spring uses proxies for AOP, transactions, caching, async, etc. Proxies intercept method calls.
- JDK dynamic proxies: require target to implement at least one interface. Proxy implements same interfaces, delegates to `InvocationHandler`.
- CGLIB: generates a subclass of the target class. Overrides methods, intercepts calls via `MethodInterceptor`.
- Proxy creation is handled by `ProxyFactory` (programmatic) or auto-proxy creators (e.g., `AnnotationAwareAspectJAutoProxyCreator`).
- Understanding proxy mode helps with self-invocation and performance considerations.
- Proxies can be detected via `AopUtils.isAopProxy()`.

### Classpath scanning
- Spring scans classpath for annotated components (`@Component`, `@Service`, etc.) using `ClassPathBeanDefinitionScanner`.
- Scanning is triggered by `@ComponentScan` or `<context:component-scan>`.
- It uses a `ClassLoader` to find classes in specified packages, then filters by annotations.
- Includes/excludes can be customized via filters.
- Performance can be impacted by scanning many packages; consider specifying base packages.

### Reflection usage
- Spring heavily uses reflection for dependency injection, reading annotations, invoking methods (e.g., in AOP), and instantiating classes.
- `ReflectionUtils` provides helper methods for reflection operations.
- Reflection can be slower than direct calls, but Spring caches reflection data (e.g., `Method` objects) to improve performance.
- Understanding reflection helps in debugging and extending Spring (e.g., custom annotations).

### Custom annotation processing
- Spring processes custom annotations via `BeanPostProcessor`s or `BeanFactoryPostProcessor`s.
- Example: `@Autowired` is processed by `AutowiredAnnotationBeanPostProcessor`.
- To create custom annotations, you typically implement a `BeanPostProcessor` that scans beans for your annotation and modifies them.
- For method-level annotations (like `@Cacheable`), you might use AOP.
- Use `AnnotationUtils` to find annotations on classes/methods (including meta-annotations).

### Custom starter creation
- Creating a custom Spring Boot starter involves:
  1. Creating an auto-configuration module with `@Configuration` and conditionals.
  2. Defining `@ConfigurationProperties` for configuration.
  3. Registering auto-configuration in `META-INF/spring.factories` (or new imports file).
  4. Providing a starter POM that includes the auto-configuration and required dependencies.
- Best practice: separate the auto-configuration code from any library code.
- Include a `spring-autoconfigure-metadata.properties` for performance.
- Document configuration properties.

### Spring modularization strategy
- In large applications, modularization with Spring involves dividing the codebase into modules (Maven/Gradle subprojects).
- Each module can have its own Spring configuration and can be a separate JAR.
- Use `@Import` or `@ComponentScan` to integrate modules.
- Consider using `spring-modules` pattern: core, domain, infrastructure, etc.
- Ensure modules have clear boundaries and avoid circular dependencies.

### Monolith to microservices migration
- Strategies: Strangler Fig pattern (incrementally replace pieces), split by bounded contexts, extract services one by one.
- Start with identifying seams: areas with low coupling.
- Use anti-corruption layer to isolate old and new.
- Refactor database: separate data per service, handle shared data.
- Implement inter-service communication (REST, messaging) and eventual consistency.
- Spring Cloud tools assist in discovery, config, and resilience.

### Transaction boundaries in distributed systems
- Distributed transactions (XA) are complex and not recommended in microservices.
- Instead, use SAGA pattern: choreography or orchestration with compensating transactions.
- Ensure idempotency and retry mechanisms.
- Eventual consistency is often acceptable; design accordingly.
- Use event-driven communication to maintain data consistency across services.

### Event-driven architecture in Spring
- Spring provides support for event-driven architecture via:
  - `ApplicationEvent` and `@EventListener` for intra-app events.
  - Spring Cloud Stream for messaging between services.
  - Spring Integration for EIP.
- Events represent state changes and can be persisted for replay.
- Patterns: event sourcing, CQRS, outbox pattern for reliable event publishing.
- Event-driven systems are scalable and decoupled.

### DDD + Spring alignment
- Domain-Driven Design (DDD) emphasizes domain model, aggregates, value objects, repositories, and domain events.
- Spring can implement DDD building blocks:
  - `@Entity` for JPA entities (may represent aggregates).
  - `@Repository` for repositories.
  - `@Service` for application services.
  - Domain events via `ApplicationEventPublisher`.
- Spring Data REST can expose aggregates directly.
- Use `@DomainEvents` and `@AfterDomainEventPublication` in aggregate roots.

### Clean architecture with Spring
- Clean architecture (by Robert Martin) layers: entities, use cases, interface adapters, frameworks & drivers.
- Spring fits in the outer layer (frameworks). Domain should have no Spring dependencies.
- Use dependency inversion: interfaces defined in inner layers implemented by outer layers.
- Spring can wire dependencies following this structure via constructor injection.
- This approach keeps business logic independent of infrastructure.

### Hexagonal architecture with Spring
- Hexagonal (ports and adapters) isolates core from external concerns.
- Ports are interfaces (e.g., repository, service). Adapters implement them (e.g., JPA repository, REST controller).
- Spring beans can be adapters; core can be POJOs with injected ports.
- Allows easy testing and swapping of adapters.
- Use `@Component` for adapters, with interfaces in core.

### Multi-tenant architecture
- Multi-tenancy supports multiple customers (tenants) with data isolation.
- Approaches: separate database, shared database separate schema, or shared schema with tenant discriminator.
- In Spring, you can implement:
  - Multi-tenant data source or connection provider.
  - Using `@TenantId` in entities and Hibernate filters.
  - Tenant resolution via subdomain, header, or JWT.
- For JPA, use `AbstractDataSourceBasedMultiTenantConnectionProviderImpl` and `CurrentTenantIdentifierResolver`.
- Spring Security can include tenant in authentication.

### Security hardening
- Hardening Spring applications:
  - Use strong password encoding (BCrypt).
  - Enable CSRF for state-changing operations in web apps.
  - Set secure HTTP headers via `http.headers()` (CSP, XSS protection, HSTS).
  - Validate all inputs (including path variables and headers).
  - Use `@PreAuthorize` for method security.
  - Regularly update dependencies for CVEs.
  - Avoid exposing actuator endpoints publicly, or secure them.

### Enterprise governance with Spring
- Governance includes standards, policies, and oversight for application development.
- Spring can support governance through:
  - Centralized configuration (Spring Cloud Config) enforcing consistency.
  - Shared libraries and custom starters enforcing coding standards.
  - Security policies via centralized authorization server (OAuth2).
  - Auditing with Spring Data Envers or custom event listeners.
  - Monitoring and logging aggregation for compliance.
- Use Spring Boot's production-ready features for observability and control.
